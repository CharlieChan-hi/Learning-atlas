# HT01 APP 上线资料学习说明：小白版

> 项目代号：HT01  
> 应用名：Bettermy-Heart rate&Health  
> 副标题：Blood Pressure&Blood Sugar  
> 产品方向：心率、血压、血糖、健康记录类 App  
> 文档用途：把原始表格里的每一项资料讲成“小白也能理解”的上线协作说明。

重要提醒：本文只放示例和占位符，不记录真实密码、密钥、Token、后台登录信息或共享密钥。

## 1. 这张表到底是什么

这张表是一份 **App 上线前资料清单**。

一个 App 不是写完代码就能上线。它还需要：

| 需要准备的东西 | 类比成现实世界 | 作用 |
| --- | --- | --- |
| 包名、APPLEID | 身份证、营业执照编号 | 证明“这到底是哪一个 App” |
| Firebase、Meta、Singular、AdMob | 监控系统、广告系统、数据统计系统 | 看 App 是否稳定、用户从哪里来、广告有没有赚钱 |
| 隐私协议、服务条款 | 用户须知、免责声明 | 告诉用户和平台：我们收集什么数据，怎么用，责任边界是什么 |
| 上线图、商店信息、关键词 | 门店招牌、海报、宣传语 | 让用户在 App Store 看懂并愿意下载 |
| 反馈邮箱、官网 | 客服邮箱、官方网站 | 用户、苹果审核、广告平台能找到你 |

所以这张表的核心不是“填资料”，而是：**让开发、广告、设计、法务、测试、运营这些人知道自己需要什么资料、从哪里拿、拿到后给谁。**

## 2. 先理解几个基础角色

### 2.1 客户端开发

客户端就是用户手机里安装的 App。

所以“客户端开发”一般就是写 iOS App 的开发同学。他们会把包名、Firebase、Meta、Singular、AdMob 等配置接进 App 代码里。

例子：

```text
用户打开 Bettermy-Heart rate&Health
手机里运行的这个 App = 客户端
负责写这个 App 的人 = 客户端开发
```

### 2.2 后端

后端是 App 背后的服务器。

比如用户购买订阅后，App 可能要把购买凭证发给后端，后端再去苹果那里校验这个订阅是不是真的。

例子：

```text
用户买了月订阅
App 把购买信息发给后端
后端拿共享密钥去苹果校验
校验通过后，给用户开通会员权益
```

### 2.3 后台

表格里的“后台”通常指某个平台的管理网站。

| 后台 | 是什么 |
| --- | --- |
| 苹果后台 | App Store Connect、Apple Developer，用来创建 App、配置内购、查看审核 |
| Meta 后台 | Facebook/Instagram 广告和应用配置后台 |
| Firebase 后台 | Google Firebase 控制台 |
| AdMob 后台 | Google AdMob 广告变现后台 |
| 公司项目后台 | 公司内部管理项目资料的系统 |

### 2.4 SDK

SDK 可以理解成“第三方平台提供的一包工具代码”。

如果 App 想接入某个平台能力，一般不是从零写，而是接入它的 SDK。

例子：

```text
想记录崩溃 → 接 Firebase SDK
想投 Facebook 广告并追踪效果 → 接 Meta SDK
想知道用户从哪个广告渠道来 → 接 Singular SDK
想在 App 里展示广告赚钱 → 接 AdMob SDK
```

SDK 的意义：**让你的 App 能和第三方平台连接起来。**

## 3. 最重要的小白概念：归因

### 3.1 归因是什么意思

归因就是：**判断一个用户是从哪里来的，并把他的安装、注册、订阅等行为算到对应渠道上。**

举个例子：

```text
用户 A 看到 Facebook 广告
↓
点击广告
↓
跳到 App Store
↓
下载 HT01
↓
打开 App
↓
购买订阅
```

广告团队会问：

```text
这个用户是谁带来的？
是 Facebook 广告？
是 Google 广告？
是 App Store 自然搜索？
是哪条广告素材带来的？
他最后有没有付费？
这笔广告费到底赚不赚钱？
```

能回答这些问题的过程，就叫 **归因**。

### 3.2 归因有什么意义

归因的意义是：**帮公司判断广告钱花得值不值。**

没有归因，就会变成：

```text
今天花了 1000 美元广告费
App 多了 500 个用户
但不知道这些用户来自哪里
也不知道哪个广告带来了付费用户
```

有归因之后，就可以变成：

```text
Facebook 广告 A 花了 300 美元，带来 100 个安装，10 个订阅
Google 广告 B 花了 300 美元，带来 80 个安装，1 个订阅
TikTok 广告 C 花了 400 美元，带来 300 个安装，0 个订阅
```

这时候广告团队就知道：

```text
Facebook 广告 A 可能值得继续投
Google 广告 B 要优化
TikTok 广告 C 看起来安装多，但质量差，可能要停
```

### 3.3 MMP 是什么

MMP 是 Mobile Measurement Partner，中文通常叫 **移动归因平台**。

你可以把 MMP 理解成广告数据里的“裁判”或“记账员”。

常见 MMP：

```text
Singular
AppsFlyer
Adjust
Branch
```

本表里写的 Singular，就是一个 MMP。

它的作用是：

```text
Meta 说：这个用户是我带来的
Google 说：这个用户也和我有关
Singular 根据规则判断：这次安装和付费到底算给谁
```

所以：

```text
归因 = 判断用户来源
MMP = 做归因的平台
Singular = 一个具体的 MMP 工具
```

## 4. 广告和数据平台分别干什么

### 4.1 Firebase

Firebase 是 Google 提供的一套 App 基础工具。

常见用途：

| 用途 | 小白理解 |
| --- | --- |
| Crashlytics | App 崩溃记录，帮开发知道哪里闪退 |
| Analytics | 统计用户打开、点击、注册、订阅等行为 |
| Remote Config | 不发新版 App，也能远程控制一些配置 |
| Cloud Messaging | 推送通知 |

Firebase 的意义：**让团队知道 App 是否稳定、用户在 App 里做了什么。**

### 4.2 Meta / FBID

Meta 是 Facebook、Instagram 背后的公司。

FBID 在这里一般指 **Meta App ID**，不是 Facebook 主页 ID，也不是个人账号 ID。

它的意义：

```text
让你的 App 和 Meta 广告系统对上号
```

如果广告团队要投 Facebook 或 Instagram 广告，客户端通常需要接 Meta SDK，并配置 FBID 和 FB 客户端口令。

### 4.3 Singular / MMP

Singular 是归因平台。

它主要回答：

```text
用户从哪个广告渠道来？
是哪条广告带来的？
这个用户有没有注册、订阅、付费？
广告投入和收入是否划算？
```

Singular 的意义：**帮助广告团队判断渠道质量和投放效果。**

### 4.4 AdMob

AdMob 是 Google 的 App 广告变现平台。

如果 App 里要展示广告赚钱，比如横幅广告、插屏广告、激励广告，就可能需要接 AdMob。

例子：

```text
用户打开 App
↓
看到一个广告
↓
广告展示或点击产生收益
↓
AdMob 后台统计收入
```

AdMob 的意义：**让 App 可以通过展示广告获得收入。**

### 4.5 它们之间的区别

| 平台 | 主要回答的问题 |
| --- | --- |
| Firebase | App 稳不稳定？用户在里面做了什么？ |
| Meta | Facebook/Instagram 广告有没有带来用户？ |
| Singular | 用户到底从哪个渠道来？广告效果怎么算？ |
| AdMob | App 里展示广告有没有赚钱？ |

一个完整用户路径：

```text
用户看到 Meta 广告
↓
点击并下载 App
↓
Singular 判断这个用户来自 Meta
↓
用户打开 App，Firebase 记录行为和崩溃
↓
App 展示广告，AdMob 记录广告收入
```

## 5. 原表字段逐项讲解

### 5.1 基础信息：心率包

是什么：

```text
这个项目的产品方向，是心率、血压、血糖、健康记录相关 App。
```

意义：

```text
让开发、设计、文案、投放知道这个 App 大概做什么。
```

注意：

健康类 App 不能随便承诺“诊断疾病”“治疗疾病”“替代医生”。更安全的说法是“健康记录”“一般健康参考”。

### 5.2 包名

是什么：

```text
iOS App 的唯一身份标识，英文常叫 Bundle Identifier。
```

例子：

```text
com.company.bettermy.hearthealth
```

意义：

```text
苹果、证书、推送、内购、第三方 SDK 都可能用它识别这款 App。
```

没有它会怎样：

```text
客户端无法正确打包、签名、创建证书，也无法稳定接入很多平台配置。
```

小白类比：

```text
包名 = App 的身份证号
```

### 5.3 APPLEID

是什么：

```text
App Store Connect 给某个 App 的数字 ID。
```

例子：

```text
1234567890
https://apps.apple.com/app/id1234567890
```

意义：

```text
用来拼 App Store 链接，也可能用于投放、归因、后台配置。
```

注意：

这里的 APPLEID 通常不是苹果账号邮箱，而是 App 的数字编号。

### 5.4 项目代号 HT01

是什么：

```text
公司内部给这个项目起的编号。
```

意义：

```text
方便大家沟通，不用每次都说完整 App 名。
```

例子：

```text
“HT01 的截图今天要给设计”
“HT01 的 Firebase 配置发给客户端”
```

### 5.5 应用名

当前应用名：

```text
Bettermy-Heart rate&Health
```

是什么：

```text
App Store 展示给用户看的名称。
```

意义：

```text
影响用户第一眼理解，也影响品牌和搜索。
```

注意：

应用名要注意拼写、大小写、符号、品牌一致性。健康类名称也要避免医疗承诺过重。

### 5.6 副标题

当前副标题：

```text
Blood Pressure&Blood Sugar
```

是什么：

```text
App Store 里应用名下面的一行补充说明。
```

意义：

```text
帮助用户快速理解 App 功能，也能覆盖一些关键词。
```

注意：

副标题不宜写成夸大医疗效果，比如“Lower Blood Pressure Fast”这类表达风险更高。

### 5.7 FBID

是什么：

```text
Meta 后台里的 App ID，用于 Meta SDK 和广告系统识别这款 App。
```

意义：

```text
如果要投 Facebook 或 Instagram 广告，FBID 能让广告平台和 App 数据对上。
```

谁需要：

```text
客户端开发、广告团队
```

### 5.8 FB 客户端口令

是什么：

```text
Meta 后台生成的 Client Token，用于 Meta SDK 配置。
```

意义：

```text
让 App 能正确初始化 Meta SDK。
```

注意：

它不像密码那么敏感，但也不应该到处公开发。

### 5.9 Firebase 配置

是什么：

```text
Firebase 项目给 App 的配置文件。
```

iOS 常见文件名：

```text
GoogleService-Info.plist
```

Android 常见文件名：

```text
google-services.json
```

意义：

```text
让 App 能连接到正确的 Firebase 项目，统计崩溃、事件、用户行为等。
```

### 5.10 mmp 应用 ID

是什么：

```text
Singular 这类 MMP 平台给当前 App 的应用 ID。
```

意义：

```text
客户端接入 Singular SDK 时要用它，广告团队也需要它看归因数据。
```

小白理解：

```text
mmp 应用 ID = 归因系统里这款 App 的编号
```

### 5.11 AdMob 应用 ID

是什么：

```text
AdMob 后台给当前 App 的应用 ID。
```

例子格式：

```text
ca-app-pub-xxxxxxxxxxxxxxxx~yyyyyyyyyy
```

意义：

```text
如果 App 要展示广告赚钱，客户端需要用它初始化 AdMob SDK。
```

### 5.12 隐私协议链接

是什么：

```text
一个网页链接，说明 App 收集什么数据、为什么收集、怎么使用、是否共享给第三方。
```

意义：

```text
App Store 审核需要，用户也需要知道自己的数据怎么被处理。
```

健康类 App 尤其要写清楚：

```text
健康数据
设备信息
广告标识符
订阅信息
崩溃日志
数据是否共享给 Firebase、Meta、Singular、AdMob 等第三方
```

### 5.13 服务条款

是什么：

```text
用户使用 App 时需要遵守的规则，以及订阅、退款、免责声明、责任限制。
```

意义：

```text
减少纠纷，也帮助平台审核理解产品边界。
```

健康类免责声明例子：

```text
本应用仅用于一般健康记录和信息参考，不属于医疗器械，不提供医学诊断、治疗或专业医疗建议。
```

### 5.14 反馈邮箱

是什么：

```text
用户、苹果审核、广告平台、隐私请求能联系到你的邮箱。
```

例子：

```text
support@example.com
feedback@example.com
```

意义：

```text
用户遇到问题能反馈；隐私协议和服务条款里也要放这个邮箱。
```

注意：

这个邮箱必须真实可用，不能写了没人看。

### 5.15 上线图

是什么：

```text
App Store 里展示的应用截图和宣传图。
```

意义：

```text
用户在商店看到截图后，决定要不要下载。
```

注意：

第一版可以参考头部竞品，但不能直接复制别人的素材、文案和设计。

### 5.16 产品链接

是什么：

```text
App Store 里这款 App 的链接。
```

例子：

```text
https://apps.apple.com/app/id1234567890
```

意义：

```text
项目群、广告团队、官网、测试都可能需要这个链接。
```

### 5.17 商店信息

是什么：

```text
App Store 上展示的所有文案和配置，包括标题、副标题、描述、关键词、隐私协议、服务协议、免责声明等。
```

意义：

```text
影响审核、搜索排名、用户转化和合规风险。
```

### 5.18 官网

是什么：

```text
项目的官方网站。
```

意义：

```text
可以放隐私协议、服务条款、app-ads.txt、产品介绍、下载入口。
```

如果要投 Web 测试，也要告诉投放团队，因为广告落地页可能会用到官网。

### 5.19 app 共享密钥

是什么：

```text
苹果开发者后台生成的订阅校验密钥。
```

意义：

```text
用户购买订阅后，后端可能需要用它向苹果验证这笔订阅是不是真的。
```

注意：

这是敏感信息。不能写进普通文档、项目群、截图或公开仓库。

正确写法：

```text
app 共享密钥：已生成，由后端负责人通过安全渠道接收
```

错误写法：

```text
app 共享密钥：abcdefg123456
```

### 5.20 app-ads.txt

是什么：

```text
一个放在官网根目录的广告授权文件。
```

常见链接：

```text
https://example.com/app-ads.txt
```

意义：

```text
告诉广告平台：哪些广告商被授权销售这个 App 的广告库存。
```

小白类比：

```text
app-ads.txt = 广告合作方授权名单
```

### 5.21 关键词

是什么：

```text
用户在 App Store 或搜索引擎里可能输入的词。
```

例子：

```text
heart rate
blood pressure
blood sugar
health tracker
pulse monitor
```

意义：

```text
影响 App Store 搜索曝光、苹果搜索广告 ASA、ASO 和 SEO。
```

几个缩写：

| 缩写 | 含义 | 小白理解 |
| --- | --- | --- |
| ASO | App Store Optimization | 让 App 在应用商店更容易被搜到 |
| ASA | Apple Search Ads | 苹果搜索广告 |
| SEO | Search Engine Optimization | 让网页在搜索引擎更容易被搜到 |

### 5.22 沙盒账号

是什么：

```text
苹果开发者后台创建的测试账号。
```

意义：

```text
用来测试订阅、内购，不会真的扣真实用户的钱。
```

注意：

尽量不要用真实个人邮箱，避免和正式 Apple ID 混淆。

### 5.23 苹果开发者邮箱

是什么：

```text
登录苹果开发者账号用的邮箱。
```

意义：

```text
用来管理 App、证书、内购、审核、App Store 信息。
```

注意：

密码必须保密，不能写进项目文档。

| 广告变现 | AdMob 应用 ID | 待补充 | AdMob 后台 | 客户端、广告团队 | 广告赚钱平台的 App ID |
| 法务支持 | 隐私协议链接 | 待补充 | 法务/部署 | 苹果审核、用户、投放 | 数据怎么收集和使用 |
| 法务支持 | 服务条款链接 | 待补充 | 法务/部署 | 苹果审核、用户 | 用户规则和免责声明 |
| 法务支持 | 反馈邮箱 | 待补充 | 新注册邮箱 | 用户、审核、法务 | 联系客服和处理投诉 |
| 官网 | 官网链接 | 待补充 | 开发部署 | 项目组、投放 | 项目官方网站 |
| 广告授权 | app-ads.txt | 待补充 | 部署/广告团队 | 广告平台 | 广告授权名单 |
| 商店增长 | 上线图 | 待补充 | 设计 | 商店、投放 | App Store 宣传截图 |
| 商店增长 | 商店信息 | 待补充 | 产品/ASO/法务 | 苹果审核、用户 | 商店里的完整文案 |
| 商店增长 | 关键词 | 待补充 | ASO/点点数据 | ASA、ASO、SEO | 用户搜索时可能用的词 |

## 7. 小白最该先记住的 12 句话

1. 包名是 App 的身份证。
2. APPLEID 是 App Store 给这个 App 的数字编号。
3. 客户端就是用户手机里的 App。
4. SDK 是第三方平台给 App 用的一包工具代码。
5. Firebase 主要看 App 稳不稳定、用户做了什么。
6. Meta 主要和 Facebook、Instagram 广告相关。
7. AdMob 主要用于 App 里展示广告赚钱。
8. Singular 是 MMP，用来做归因。
9. 归因就是判断用户从哪里来、广告钱花得值不值。
10. 隐私协议说明数据怎么收集和使用。
11. 服务条款说明使用规则、订阅规则和免责声明。
12. 密码、共享密钥、真实 Token 不能写进普通文档。

## 8. 上线前协作流程

### 第一步：先把 App 身份确定下来

需要确认：

```text
项目代号
应用名
副标题
包名
APPLEID
```

因为这些会影响客户端打包、苹果后台、商店链接和后续所有平台配置。

### 第二步：接入数据和广告平台

需要准备：

```text
Firebase 配置
Meta FBID 和客户端口令
Singular mmp 应用 ID
AdMob 应用 ID
```

因为这些决定了 App 能不能统计数据、做广告归因、展示广告赚钱。

### 第三步：准备合规资料

需要准备：

```text
隐私协议
服务条款
反馈邮箱
官网
健康类免责声明
```

因为健康类 App 审核和广告合规会比较敏感。

### 第四步：准备商店和增长资料

需要准备：

```text
上线图
商店描述
关键词
产品链接
app-ads.txt
```

因为这些影响 App Store 审核、搜索曝光、下载转化和广告投放。

## 9. 信息安全分级

| 安全等级 | 信息类型 | 是否可发项目群 | 建议 |
| --- | --- | --- | --- |
| 公开信息 | 应用名、副标题、官网、产品链接 | 可以 | 可在项目群同步 |
| 内部信息 | 项目代号、包名、APPLEID、关键词 | 看范围 | 发给相关岗位即可 |
| 受控信息 | Firebase 配置、FBID、AdMob ID、Singular ID | 谨慎 | 发给客户端、广告团队 |
| 敏感信息 | 账号密码、app 共享密钥、后台登录信息 | 不可以 | 通过安全渠道单独传 |

## 11. 最后一遍总结

这张表可以分成五类：

| 类别 | 解决的问题 |
| --- | --- |
| App 身份 | 这到底是哪一个 App |
| 数据统计 | App 稳不稳定，用户做了什么 |
| 广告归因 | 用户从哪里来，广告钱花得值不值 |
| 广告变现 | App 里展示广告能不能赚钱 |
| 法务商店 | 能不能过审，用户能不能看懂并信任 |

最关键的概念是：

```text
归因不是技术炫词。
归因就是帮团队判断：用户从哪里来，广告费有没有效果，下一笔钱应该投在哪里。
```
