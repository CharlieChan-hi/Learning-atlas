# 术语表

这一模块不是背英文缩写，而是把常见技术词翻译成产品经理能使用的判断语言。

读术语时只问三件事：它解决什么问题、它带来什么产品约束、我在评审里应该追问什么。

## 怎么读

1. 先读 `01_术语表.md`，建立基础词汇表。
2. 再做 `02_真实案例与练习.md`，把术语放回真实产品场景。
3. 遇到不懂的词，不要只查定义，要追问它影响了哪个用户体验。

## 本模块要解决的问题

很多 PM 听到 API、缓存、队列、幂等、SLO、召回、排序时，会误以为这是工程师内部语言。真正要学的是：这些词背后通常对应延迟、成本、可靠性、风险、体验一致性和团队协作边界。

## 你应该带走什么

| 术语类型 | 你要理解什么 |
|---|---|
| 系统基础 | 客户端、服务端、数据库、缓存、队列分别承担什么责任 |
| 接口协作 | API、Webhook、错误码、版本、幂等如何影响产品体验 |
| 数据与模型 | 事件、日志、特征、召回、排序如何支撑推荐和风控 |
| 可靠性 | SLO、监控、降级、重试、最终一致如何影响用户承诺 |
| 安全与风险 | 认证、授权、隐私、风控、审计分别解决什么问题 |

## 读完后的自测

拿到一个新术语时，你应该能说出：

- 它保护或提升了什么用户体验？
- 它依赖哪些数据或系统能力？
- 它失败时用户会看到什么？
- PM 在需求阶段应该补充什么边界条件？
