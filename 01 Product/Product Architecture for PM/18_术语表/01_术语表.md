# 术语表

## 使用方法

不要把术语当成考试名词。每个词都要转成一句产品语言：它影响什么体验，限制什么方案，需要什么验证。

## 系统基础

| 术语 | 人话理解 | PM 要追问 |
|---|---|---|
| 客户端 | 用户手里的 App、网页、小程序 | 离线、弱网、版本差异怎么处理？ |
| 服务端 | 接收请求、执行规则、读写数据的后台系统 | 成功、失败、超时分别返回什么？ |
| 数据库 | 存核心业务对象和状态的地方 | 对象状态是否清楚？谁能修改？ |
| 缓存 | 把常用结果临时放近一点，提高速度 | 数据旧了会不会误导用户？ |
| 队列 | 把任务排队异步处理，削峰和解耦 | 用户是否需要立刻知道结果？ |
| 搜索索引 | 为快速检索准备的数据结构 | 多久更新一次？新内容何时可搜到？ |

## API 与协作

| 术语 | 人话理解 | PM 要追问 |
|---|---|---|
| API | 系统之间约定好的能力入口 | 参数、错误、权限、版本是否清楚？ |
| Webhook | 外部系统状态变化后主动通知你 | 重复、延迟、失败重试怎么处理？ |
| 错误码 | 把失败原因表达给调用方 | 用户和开发者能否知道下一步？ |
| 版本兼容 | 新接口上线时不破坏旧调用方 | 老客户迁移成本是什么？ |
| 幂等 | 同一业务意图重复提交只产生一次副作用 | 幂等键由谁生成，保存多久？ |
| 限流 | 控制请求量，保护系统不被打爆 | 被限制时用户看到什么？ |

## 数据、推荐与风控

| 术语 | 人话理解 | PM 要追问 |
|---|---|---|
| 日志事件 | 用户和系统行为记录 | 曝光、点击、失败、取消是否都记录？ |
| 特征 | 模型或规则用于判断的信号 | 信号是否可靠、及时、合规？ |
| 召回 | 先找出可能相关的一批候选 | 候选来源是否覆盖新内容和长尾？ |
| 排序 | 给候选打分并决定展示顺序 | 目标是点击、停留、成交还是满意？ |
| 重排 | 在最终展示前加入规则和体验约束 | 是否保护多样性、安全和频控？ |
| 风险评分 | 对交易、账号或行为估计风险 | 分数如何转成验证、审核或拦截？ |

## 可靠性与异常

| 术语 | 人话理解 | PM 要追问 |
|---|---|---|
| SLO | 对用户体验稳定性的目标承诺 | 哪条用户旅程最需要稳定？ |
| 监控 | 系统持续观察指标和异常 | 只看系统指标，还是也看用户失败？ |
| 降级 | 系统压力大时保住核心体验 | 哪些能力可以暂时变弱？ |
| 重试 | 失败后再次尝试 | 会不会重复扣款、重复通知、重复下单？ |
| 最终一致 | 短时间不一致，但最终会对齐 | 用户等待期间看到什么状态？ |
| 人工处理 | 系统无法自动判断时转给运营或客服 | 入口、时效、记录和申诉怎么设计？ |

## 安全、隐私与权限

| 术语 | 人话理解 | PM 要追问 |
|---|---|---|
| 认证 | 确认你是谁 | 登录失败、设备变更如何处理？ |
| 授权 | 确认你能做什么 | 角色、资源、动作边界是否清楚？ |
| 隐私最小化 | 只收集完成目标所需的数据 | 这个字段是否真的必要？ |
| 审计日志 | 记录谁在什么时候做了什么 | 出问题后能否追溯？ |
| 反欺诈 | 识别和处置异常行为 | 误伤正常用户后如何解释和申诉？ |
| 安全边界 | 防止越权、泄漏和滥用 | API、后台、第三方是否都有边界？ |

## 最小背诵版

- API 是能力入口，错误码和版本决定协作成本。
- 缓存提速但可能变旧，队列解耦但会带来等待。
- 幂等解决重复副作用，重试解决偶发失败。
- 推荐系统不是一个模型，而是召回、排序、重排和反馈闭环。
- 风控不是只拦截，还要验证、审核、解释和申诉。
- 可靠性不是机器不坏，而是用户关键旅程稳定可用。
