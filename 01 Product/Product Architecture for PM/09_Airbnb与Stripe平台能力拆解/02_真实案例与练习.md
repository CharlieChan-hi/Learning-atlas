# Airbnb 与 Stripe 真实案例与练习

本练习分两段：一个练风控，一个练支付。目标是让你理解平台能力不是页面功能，而是事件、状态、规则、模型、API 和异常处理的组合。

## 练习 A：高风险预订审核

### 场景

Airbnb 上出现一笔高风险预订：新注册用户、临近入住、价格较高、房东历史较少，系统需要判断是否放行、审核或要求额外验证。

### 第 1 步：定义触发事件

写清楚风险系统什么时候介入：

- 注册后？
- 搜索房源时？
- 点击预订时？
- 支付前？
- 支付后但入住前？
- 房东确认前？

介入点越晚，用户打扰越少，但风险可能已经扩大。

### 第 2 步：列风险信号

| 信号类型 | 示例 |
|---|---|
| 账号信号 | 注册时间、历史评价、验证状态 |
| 交易信号 | 金额、入住时间、取消政策、房源风险 |
| 行为信号 | 搜索/预订频率、异常路径、设备变化 |
| 支付信号 | 支付方式、失败次数、账单信息一致性 |
| 关系信号 | 房客和房东历史、评价网络 |

不要写具体模型权重，公开资料没有披露。

### 第 3 步：设计处置动作

| 风险等级 | 动作 |
|---|---|
| 低 | 放行，后台记录 |
| 中 | 要求额外验证，或进入快速审核 |
| 高 | 暂停预订，人工审核，必要时拦截 |
| 不确定 | 限制部分动作，同时请求更多信息 |

关键是要有申诉和解释路径。只拦截不解释，会伤害正常用户。

### 第 4 步：设计指标

- 欺诈损失下降。
- 误拦正常订单比例。
- 审核时长。
- 用户申诉成功率。
- 预订转化率。
- 房东/房客投诉率。

## 练习 B：支付重复请求

### 场景

用户点击支付后网络卡住，连续点了三次。客户端重试，服务端收到多个相同支付请求。你要防止重复扣款，同时给用户清晰反馈。

### 第 1 步：定义业务意图

同一次订单支付应该对应同一个业务意图。重复点击、客户端重试、网络超时，不应该变成三笔支付。

### 第 2 步：设计状态

| 状态 | 用户看到什么 |
|---|---|
| 待确认 | 正在确认支付，请勿重复操作 |
| 成功 | 支付成功，订单确认 |
| 失败可重试 | 支付失败，可更换方式或重试 |
| 处理中 | 已收到请求，等待支付机构返回 |
| 需人工处理 | 状态异常，客服或后台处理 |

### 第 3 步：写幂等逻辑

- 每次业务支付意图生成一个幂等键。
- 重试时使用同一个幂等键。
- 服务端发现同一幂等键已经处理过，就返回原处理结果。
- webhook 重复到达时，也不能重复更新成错误状态。

### 第 4 步：设计指标

- 重复扣款事故数。
- 支付成功率。
- 支付处理中时长。
- 用户重复点击率。
- 客服咨询量。
- webhook 失败和重试次数。

## 区分事实、推断、未披露

| 类型 | 示例 |
|---|---|
| 公开事实 | Airbnb 公开资料描述风险 ML 链路；Stripe 文档公开 Radar、API、幂等等能力 |
| 合理推断 | 高风险预订需要事件触发、风险信号、模型/规则、处置动作和申诉路径；重复支付需要幂等和状态管理 |
| 公开资料未披露 | Airbnb 当前风险阈值、Stripe 当前 Radar 模型权重、内部状态机实现 |

## 自测问题

1. 风控为什么不能只做“通过/拒绝”？
2. 为什么误伤正常用户也是风控风险？
3. 支付状态为什么要围绕业务意图，而不是点击次数？
4. 幂等和重试分别解决什么问题？
5. API 错误码为什么会影响平台产品体验？
