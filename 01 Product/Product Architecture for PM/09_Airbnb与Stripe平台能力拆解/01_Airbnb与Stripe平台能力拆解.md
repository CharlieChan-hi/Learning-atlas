# Airbnb 与 Stripe 平台能力拆解

> 一句话：Airbnb 和 Stripe 不是只做“房源”和“支付”，它们的核心是高信任平台能力：风险识别、支付编排、API、规则、审核和异常处理。

## 本节真正要解决的问题

Airbnb 和 Stripe 都在处理高风险交易：

- Airbnb 要让陌生人之间完成住宿交易，必须处理身份、信任、风险、支付、取消、纠纷。
- Stripe 要让商户安全收款，必须处理 API、支付状态、欺诈、退款、争议、重复请求。

这类平台的架构重点不是页面，而是：**把复杂、危险、可复用的能力做成稳定系统，让不同业务可以调用。**

## 核心对象

| 对象 | Airbnb 里怎么体现 | Stripe 里怎么体现 |
|---|---|---|
| 用户 / 商户 | 房客、房东、账号、身份 | 商户、客户、账户、卡、支付方式 |
| 交易 | 预订、取消、退款、纠纷 | PaymentIntent、Charge、Refund、Dispute |
| 风险事件 | 异常预订、欺诈、账号风险 | 欺诈交易、盗刷、争议、异常行为 |
| 风险决策 | 放行、拦截、审核、验证 | allow、block、review、request 3DS |
| 支付状态 | 待支付、成功、失败、退款 | 授权、扣款、失败、回调、幂等 |
| 平台接口 | 内部服务和运营工具 | 对外 API、错误码、版本、幂等键 |

## Airbnb 风险处置流程图

```text
关键事件：注册、登录、预订、支付、取消、退款
        ↓
风险信号：账号、交易、行为、设备、支付、关系网络
        ↓
模型 / 规则：输出风险分数、风险类型、不确定性
        ↓
决策层：把风险判断转成产品动作
        ↓
放行 / 额外验证 / 人工审核 / 限制动作 / 拦截
        ↓
解释、申诉、客服处理、后续投诉和争议回流
```

这张图说明风控不是“通过/拒绝”开关。PM 要看清楚：风险在哪个事件点被触发，信号从哪里来，模型或规则如何转成用户可见动作，误伤后有没有申诉路径。

## Stripe 支付状态与幂等图

```text
用户重复点击支付
        ↓
客户端使用同一个幂等键提交同一业务意图
        ↓
服务端检查：这个幂等键是否已经处理过？
        ├─ 已处理：返回原结果或当前状态
        └─ 未处理：创建或确认支付
                ↓
        支付机构处理，webhook 可能延迟或重复到达
                ↓
        状态更新：处理中 / 成功 / 失败可重试 / 需验证 / 需人工处理
                ↓
        用户、商户、客服看到同一个最终状态
```

这张图的重点是“业务意图”。支付系统不能围绕用户点了几次按钮设计，而要围绕同一次订单支付意图设计，否则重试和 webhook 重复都会带来重复扣款或状态错乱。

## 系统能力怎么配合

### 1. Trust & Safety：风控不是一个开关

Airbnb 公开文章描述风险系统会在关键事件后调用欺诈预测服务，生成特征、调用模型评分，再根据阈值采取动作。

人话讲，风控不是“发现坏人就拦截”。它是一条链路：哪个事件触发、用哪些特征、模型怎么评分、阈值怎么判断、最后采取什么处置。

PM 要问：风险发生在哪个节点？注册、登录、预订、支付、取消、退款，每个节点能做的动作不同。

### 2. 处置动作：不是只有通过或拒绝

成熟风控通常不是二选一。可能动作包括：放行、阻止、进入人工审核、要求额外验证、延迟结算、限制功能、通知用户、开放申诉。

Stripe Radar 文档中可以看到规则、review、3DS 等能力，这说明风控产品需要自动化和可配置治理结合。

PM 要问：误伤正常用户怎么办？有没有解释、申诉和人工处理路径？

### 3. 支付编排：支付不是一次 API 调用

Airbnb 公开文章讨论支付编排。支付流程要管理状态、依赖、失败、重试、回调和下游系统。

人话讲，用户点“支付”只是开始。系统还要处理授权、扣款、失败重试、退款、账务、通知、风控、第三方返回慢或重复回调。

PM 要问：每个支付状态是什么意思？失败后用户看到什么？重试会不会重复扣款？

### 4. 幂等：防止重复副作用

Stripe API 文档强调幂等能力。支付系统里，网络抖动、用户重复点击、客户端重试都很常见。

幂等的产品含义是：同一个意图重复提交，不应该创建两笔订单或扣两次钱。

PM 要问：客户端重试时用什么幂等键？用户刷新页面时看到的是新请求，还是原请求状态？

### 5. API 平台：开发者体验也是产品体验

Stripe 的产品很大一部分是 API。资源命名、错误码、版本、认证、分页、限流、webhook、幂等，都会影响商户接入成本。

PM 要问：开发者出错时能不能知道怎么修？版本升级会不会破坏老商户？Webhook 失败怎么重试？

## 数据流：高风险预订

1. 用户触发预订或支付等关键事件。
2. 风险系统收集账号、交易、历史行为、设备、支付等特征。
3. 模型或规则输出风险分数和风险类型。
4. 决策层选择动作：放行、拦截、审核、验证或限制。
5. 用户、运营、客服、支付系统收到对应结果。
6. 后续完成、取消、投诉、争议等结果继续回流，改进规则和模型。

## 数据流：支付重复请求

1. 客户端发起支付请求，并带上同一业务意图对应的幂等键。
2. 服务端检查是否已经处理过同一幂等键。
3. 如果已处理，返回原结果；如果未处理，继续创建或确认支付。
4. 第三方支付和 webhook 返回状态。
5. 系统更新订单状态，并把最终状态展示给用户和商户。

重点：支付状态必须围绕“业务意图”管理，而不是围绕“用户点了几次按钮”。

## 关键权衡

| 权衡 | 解释 | PM 要盯什么 |
|---|---|---|
| 拦截欺诈 vs 误伤好用户 | 拦太少有损失，拦太多伤害转化 | 欺诈率、误拒率、申诉成功率 |
| 自动化 vs 人工审核 | 自动快但可能误判，人工准但慢 | 审核时效、队列、用户等待 |
| 支付成功率 vs 风险控制 | 放宽验证提高成功，也可能增加欺诈 | 支付成功率、争议、拒付 |
| API 稳定 vs 产品迭代 | 变更太快会伤害开发者 | 版本、兼容、迁移成本 |
| 用户解释 vs 风控保密 | 解释太少不透明，太多可能被攻击者利用 | 提示文案、客服、申诉 |

## PM 应该追问工程团队什么

- 风险系统在哪些事件点介入？每个点能做什么动作？
- 风险分数如何转成产品动作：放行、审核、验证还是拦截？
- 误伤正常用户后，用户如何知道原因、如何申诉？
- 支付状态机有哪些状态？哪些状态用户可见？
- 幂等键由谁生成，和业务订单号是什么关系？
- Webhook 失败或重复到达时，系统如何保证状态一致？
- API 错误码能否帮助开发者自己定位问题？

## 来源与证据边界

### 公开资料确认的事实

- Airbnb 公开文章描述风险 ML 系统会在关键事件后生成特征、调用评分服务、返回分数并由风险服务采取动作。
- Airbnb 公开文章讨论支付编排能力。
- Stripe Radar 文档说明 Radar 可评估欺诈风险，并支持规则、review、3DS 等动作。
- Stripe API 文档公开资源型 API、错误、版本和幂等等能力。
- Stripe 公开文章讨论 Radar 在风险评估中需要平衡欺诈和正常交易误伤。

### 基于公开资料的合理推断

- 高信任平台的 PM 需要把风险事件、模型/规则、处置动作、审核和申诉串成完整流程。
- 支付系统需求必须考虑幂等、状态、回调、重试和异常展示。
- API 产品的质量不仅是功能，还包括错误可理解、版本稳定和接入成本。

### 公开资料未披露

Airbnb 当前风险模型、支付编排内部状态机、风控阈值、人工审核策略、Stripe Radar 当前模型权重、风险评分公式、商户级规则效果、内部服务拓扑，公开资料未披露。
