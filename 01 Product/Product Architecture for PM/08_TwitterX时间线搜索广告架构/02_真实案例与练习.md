# Twitter/X 真实案例与练习

> 场景：你要设计一个“减少低质量推文曝光”的需求，但不能明显降低用户发现新内容的能力。

## 练习目标

训练你把信息流治理需求拆成候选、排序、日志、反馈、安全规则和广告护栏。

## 第 1 步：定义低质量

不能只写“低质量内容”。至少要拆成几类：

| 类型 | 可能信号 |
|---|---|
| 重复搬运 | 文本重复、媒体重复、异常发布频率 |
| 标题党 | 点击高但停留低、快速返回、负反馈高 |
| 垃圾营销 | 链接异常、账号行为异常、举报多 |
| 低相关 | 用户频繁跳过、隐藏、不互动 |
| 风险内容 | 举报、审核规则、安全分类 |

定义不清楚，工程只能猜。

## 第 2 步：拆时间线漏斗

| 阶段 | 你要回答的问题 |
|---|---|
| 候选 | 低质量内容是否在召回阶段就过滤？ |
| 排序 | 低质量信号是降权还是直接移除？ |
| 重排 | 是否限制同类低质内容连续出现？ |
| 日志 | 需要记录哪些负反馈？ |
| 实验 | 怎么证明不是把新内容也误伤了？ |

## 第 3 步：写关键取舍

- 治理强度 vs 内容多样性：过滤太狠可能误伤新作者或小众内容。
- 负反馈 vs 沉默用户：不点隐藏不代表喜欢，可能只是离开。
- 新鲜度 vs 质量：热点事件里最快的内容未必最可靠。
- 广告收入 vs 体验护栏：低质广告也会破坏信息流信任。

## 第 4 步：设计指标

| 指标类型 | 示例 |
|---|---|
| 主指标 | 低质量曝光下降、隐藏/举报率下降 |
| 护栏指标 | 新作者曝光、内容多样性、会话时长、留存 |
| 日志事件 | 曝光、停留、点击、隐藏、举报、屏蔽、关注/取关 |
| 分群 | 新用户、重度用户、不同兴趣圈层、不同语言地区 |

## 第 5 步：区分事实、推断、未披露

| 类型 | 示例 |
|---|---|
| 公开事实 | Twitter/X 公开资料介绍过时间线深度学习排序、日志管道、广告候选排序、搜索系统 |
| 合理推断 | 低质量治理需求需要改候选、排序、负反馈日志和实验护栏 |
| 公开资料未披露 | X 当前 For You 模型、低质量判定权重、审核规则细节、广告排序权重 |

## 参考答案骨架

一个合格答案应该包含：

- 先定义低质量类型，避免工程无法落地。
- 在候选、排序、重排层分别考虑过滤和降权。
- 记录足够的负反馈日志，才能判断治理是否有效。
- 保护新作者和小众内容，避免治理变成内容单一化。
- 广告也要纳入质量护栏，否则用户只会感知“信息流变差”。

## 自测问题

1. 搜索和时间线为什么不能用完全一样的排序目标？
2. 为什么日志管道是推荐系统学习的燃料？
3. 低质量治理为什么容易误伤新创作者？
4. 广告排序为什么必须有用户体验护栏？
5. 历史 Twitter 工程文章为什么不能直接写成 X 当前内部事实？
