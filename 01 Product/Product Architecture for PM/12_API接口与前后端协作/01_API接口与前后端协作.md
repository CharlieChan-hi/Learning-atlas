# API 接口与前后端协作

## API 不是技术细节

API 决定产品怎么协作。前端展示什么、失败怎么提示、能不能重试、第三方能不能接入，都和 API 设计有关。

## PM 需要问的六件事

### 1. 资源是什么

接口应该围绕业务对象设计。比如支付、订单、用户、房源、内容。对象不清，接口就会变成零散动作堆叠。

### 2. 动作是什么

是创建、查询、更新状态，还是触发一次业务动作？“取消订单”和“删除订单”不是一回事。

### 3. 成功返回什么

前端需要展示哪些信息？是否需要下一步动作？是否有处理中状态？

### 4. 失败怎么表达

错误不能只写“失败”。至少要区分：无权限、参数错、状态冲突、风控拦截、系统繁忙、第三方处理中。

### 5. 能不能安全重试

支付、下单、发券、提交审核这类动作必须考虑幂等。否则用户重复点击可能产生重复副作用。

### 6. 版本怎么演进

老客户端还在使用旧接口。接口变更要考虑兼容、废弃计划和迁移提示。

## API 协作图

```text
前端页面
  ↓ 请求：参数、权限、幂等键
API 接口
  ↓ 校验：身份、权限、参数、状态
业务服务
  ├─ 数据库：读写对象和状态
  ├─ 第三方：支付、短信、地图、风控
  ├─ 队列：通知、日志、重试、异步任务
  └─ 日志：记录成功、失败、耗时、错误码
  ↓
返回结果：成功 / 失败 / 处理中 / 需验证 / 不可重试
```

PM 看 API 不要只问“接口有没有”。要问前端怎么展示、服务端怎么判断、失败怎么表达、重复请求会不会产生副作用。

## Webhook 的产品含义

Webhook 是异步通知。比如支付成功不是马上返回，而是第三方稍后通知。PM 要设计：处理中页面、超时提示、重复通知、失败重试和最终状态查询。

## Webhook 时序骨架

```text
用户页面 → 服务端：发起支付
服务端 → 支付机构：创建或确认支付
服务端 → 用户页面：返回处理中
支付机构 → 服务端 Webhook：通知成功/失败，可能重复或延迟
服务端 → 数据库：按状态规则更新
用户页面 → 服务端：查询最终状态
```

这张图的重点是：用户页面拿到“处理中”不代表事情结束，最终状态通常由 webhook 或状态查询补齐。

## API 评审清单

- 资源名和业务对象一致吗？
- 状态是否清楚？
- 错误码能指导前端文案吗？
- 重复请求是否幂等？
- 权限在哪里校验？
- 异步结果怎么通知？
- 版本变化会不会破坏旧客户端？

## 典型坑

| 坑 | 后果 |
|---|---|
| 错误只有 500 | 前端无法给用户清楚提示 |
| 没有幂等 | 重复下单、重复扣款 |
| 状态不完整 | 用户看到“处理中”但系统无处查询 |
| 接口和对象不一致 | 后续功能扩展困难 |
