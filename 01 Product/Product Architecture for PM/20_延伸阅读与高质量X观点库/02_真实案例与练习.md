# 来源判断练习

## 练习目标

训练你判断一个链接到底能支持什么结论，避免把观点、推断和事实混在一起。

## 练习 A：X/Twitter 观点线索

### 场景

你看到一条 X/Twitter 观点说：“未来优秀工程师需要既懂架构，也懂产品。”

### 你要判断

| 问题 | 合格回答 |
|---|---|
| 它能支持什么？ | 学习动机、观点讨论、能力趋势线索 |
| 它不能支持什么？ | 不能证明某公司内部架构、模型、团队做法 |
| 需要补什么来源？ | 官方工程博客、公开演讲、方法论文档或可信访谈 |
| 写入正文时怎么标？ | 标为观点线索，不写成事实来源 |

## 练习 B：官方工程博客

### 场景

你读到 Meta Engineering 关于 Instagram Explore 推荐系统的公开文章。

### 你要拆分

| 类型 | 示例 |
|---|---|
| 公开事实 | 文章公开描述 Explore 推荐系统有候选生成和排序等链路 |
| 合理推断 | PM 设计 Explore 需求时要关注候选来源、排序目标和反馈闭环 |
| 未披露 | 当前线上模型权重、实时特征、团队内部权限边界 |

## 练习 C：官方 API 文档

### 场景

你读到 Stripe API Reference 和幂等文档。

### 你要拆分

| 类型 | 示例 |
|---|---|
| 公开事实 | Stripe 文档公开 API 资源、错误、版本、幂等键等能力 |
| 合理推断 | 支付产品设计需要围绕业务意图、状态和重复请求处理 |
| 未披露 | Stripe 内部服务拓扑、具体风控模型权重、商户级策略效果 |

## 练习 D：方法论文档

### 场景

你读 AWS Well-Architected、Google SRE、OWASP API Security Top 10。

### 你要判断

这些资料主要支持方法，不支持某个公司具体实现。

| 来源 | 可以转成什么 PM 问题 |
|---|---|
| AWS Well-Architected | 可靠性、安全、性能、成本、运维是否被评审？ |
| Google SRE | 用户旅程有没有 SLO、监控和事故复盘？ |
| OWASP API Security | API 是否存在越权、认证、暴露过度、限流不足等风险？ |

## 练习 E：给一章选择来源

你要写“支付重复请求”章节，请从下面选来源：

| 来源 | 是否适合 | 理由 |
|---|---|---|
| Stripe API 幂等文档 | 适合 | 直接支持幂等键和重复请求处理 |
| Stripe API Reference | 适合 | 支持资源、错误、版本和状态表达 |
| 某条“支付很复杂”的微博 | 不适合作事实 | 只能作为观点，不能支撑实现 |
| Airbnb 风控文章 | 部分适合 | 支持风险链路，不直接支持支付幂等 |

## 自测问题

1. 为什么 X/Twitter 不能单独当作公司内部架构事实？
2. 官方工程博客能证明哪些事实，不能证明哪些事实？
3. 为什么方法论文档适合变成评审问题，而不是写成公司案例？
4. 为什么每章不应该复制同一批延伸阅读？
5. 公开资料未披露时，为什么不能用行业常识补空白？
