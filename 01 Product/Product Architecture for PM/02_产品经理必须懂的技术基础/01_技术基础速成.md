# 技术基础速成

## 先看一张基础技术结构图

```text
用户
 ↓ 点击、浏览、支付、搜索
客户端 / 页面
 ↓ 发请求
API 接口
 ↓ 调用业务能力
服务端服务
 ├─ 数据库：保存真实状态
 ├─ 缓存：加快读取，但可能短暂变旧
 ├─ 队列：把通知、日志、重试等任务排队处理
 ├─ 第三方系统：支付、短信、地图、风控等外部能力
 └─ 日志 / 监控：记录行为、发现异常、支持实验
 ↓
返回用户可见状态：成功、失败、处理中、需验证
```

这张图要表达的是：一个按钮背后通常不是一个动作，而是一串系统协作。

## 用生活类比理解

| 技术结构 | 类比 | PM 要理解什么 |
|---|---|---|
| 客户端 / 页面 | 前台柜台 | 用户看到什么、能点什么、等待时怎么提示 |
| API | 办事窗口 | 前端和后端用什么格式沟通，失败怎么表达 |
| 服务端 | 业务规则中心 | 价格、权限、支付、风控等可信判断放这里 |
| 数据库 | 账本 | 谁是什么状态，历史是否可追溯 |
| 缓存 | 临时便签 | 读得快，但可能不是最新 |
| 队列 | 待办箱 | 不必马上完成的事排队处理 |
| 日志 / 监控 | 摄像头和报警器 | 记录发生什么，坏了能发现 |

## 一个典型功能的类代码结构

这不是任何公司的真实目录，只是帮助你理解工程师说“要改好几层”是什么意思：

```text
feature/
├── frontend/       # 页面、按钮、加载态、空态、错误态
├── api/            # 请求参数、返回结果、错误码、权限
├── services/       # 业务规则：推荐、支付、风控、派单
├── database/       # 业务对象、状态、关系、索引
├── queues/         # 异步任务、通知、重试、webhook 处理
├── logs/           # 曝光、点击、失败、取消、支付回调等事件
└── monitoring/     # 成功率、延迟、错误率、告警和仪表盘
```

PM 不需要写这些代码，但要知道需求会影响哪些层。只写“页面加一个按钮”，很可能漏掉 API、状态、异常和指标。

## 客户端和服务端

客户端负责用户看见和操作的部分，服务端负责业务规则和可信数据。比如价格、权限、支付状态这类不能只放客户端判断，否则容易被篡改或出现不一致。

PM 要问：这个判断应该在客户端、服务端，还是两边都要有？

## 数据库

数据库保存核心业务对象：用户、订单、内容、支付、权限、风险事件。数据模型一旦设计混乱，后面功能都会难做。

PM 要问：这个对象的唯一身份是什么？状态有哪些？谁能改？历史要不要保留？

## 缓存

缓存让读取更快，但可能读到旧数据。适合首页推荐、配置、热门内容，不适合随便处理支付余额这类强一致数据。

PM 要问：用户能不能接受几秒或几分钟不一致？如果不能，就别轻易靠缓存糊过去。

## 队列和异步任务

有些事不必同步完成，比如发邮件、生成报表、处理日志、重试 webhook。队列能削峰填谷，但会带来延迟和最终一致性。

PM 要问：用户是否必须马上看到结果？如果任务失败，谁重试，用户怎么知道？

## 日志和监控

日志记录发生了什么，监控告诉团队系统是否健康。推荐、风控、实验都依赖高质量事件日志。

PM 要问：这个功能上线后，哪些事件必须记录？哪些指标能发现体验变差？

## API

API 是系统之间的契约。好的 API 不只是能用，还要有清晰命名、错误码、版本、权限和幂等。

PM 要问：前端怎么知道失败原因？客户端重试会不会造成重复副作用？

## 最小记忆版

- 客户端管体验，服务端管可信业务规则。
- 数据库管状态，缓存管速度。
- 队列管异步，日志管学习，监控管健康。
- API 管协作契约。
