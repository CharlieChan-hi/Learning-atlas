# 产品经理架构学习资料库

这套资料的目标不是让产品经理变成工程师，而是帮你听懂架构讨论：一个产品承诺背后，需要哪些系统能力、数据流、协作边界和取舍。

这次重构后的原则很简单：**不要再用同一套模板撑每一章。每个案例必须讲它自己的真实问题。**

## 适合谁

- 完全不写代码，但需要和工程团队讨论方案的产品经理。
- 想理解推荐、搜索、支付、风控、实时匹配、权限、消息等系统能力的产品负责人。
- 已经会画流程图，但不知道流程图背后有哪些技术约束的人。

## 推荐学习顺序

1. 先读 `00_学习导航与路线` 到 `03_真实产品架构总览`，建立基础地图。
2. 再读 `04` 到 `09` 的真实产品案例，重点看具体产品怎么把业务问题拆成系统能力。
3. 最后读 `10` 到 `20`，把案例沉淀成 API、数据模型、权限、安全、推荐、消息、评审和练习方法。

## 先看图，再看文字

学架构不能只读段落。以后读每一章时，先找这几类图：

| 图类型 | 你要看懂什么 | 适合章节 |
|---|---|---|
| 流程图 | 用户动作按什么步骤发生 | 支付、预订、叫车、审核 |
| 系统结构图 | 哪些能力一起支撑体验 | 音乐首页、Twitter/X、平台能力 |
| 数据流图 | 数据从哪里来、被谁处理、影响什么结果 | 推荐、派单、风控、日志 |
| 对象关系图 | 用户、订单、内容、支付、风险事件之间怎么关联 | 数据模型、Netflix、Airbnb |
| 状态机图 | 一个对象会经历哪些状态，谁能改变状态 | 支付、订单、审核、退款 |

读图顺序固定为：先看用户承诺，再看核心对象，再看系统能力，再看数据流，最后看失败兜底和指标。

## 结构下面放什么

非技术 PM 不需要背代码，但要知道一个功能通常不是只改页面：

```text
用户看到的页面
  ↓
API：前端和后端约定怎么说话
  ↓
服务：执行业务规则，比如价格、权限、风控、推荐
  ↓
数据库：保存用户、订单、内容、支付、状态
  ↓
队列 / 任务：处理不必立刻完成的事，比如通知、日志、重试
  ↓
日志 / 监控：记录发生了什么，发现系统和体验变差
```

这不是任何公司的真实代码目录，只是帮助你理解“架构为什么不是一个页面功能”。

## 这套资料应该怎么写

每个真实案例章节都必须回答这些问题：

| 问题 | 必须讲清楚什么 |
|---|---|
| 本节解决什么产品问题 | 不能只说“讲架构”，要说清具体业务场景 |
| 核心对象是什么 | 用户、订单、内容、支付、风险事件、司机、房源等 |
| 系统能力怎么配合 | 推荐、搜索、匹配、定价、风控、支付、通知等如何分工 |
| 数据怎么流动 | 输入是什么，系统怎么处理，结果如何影响用户体验 |
| 关键取舍是什么 | 速度、准确率、成本、公平性、可靠性、安全、体验一致性 |
| 来源边界在哪里 | 哪些是公开事实，哪些是合理推断，哪些公开资料未披露 |

通用概念只在全局章节讲一次，不在每个案例里重复复制。

## 判断一章有没有写好

读完一章后，应该能回答：

- 如果删掉公司名，我还能不能看出这是哪类产品？
- 我能不能口头讲清这章的数据流？
- 我能不能说出至少 3 个具体权衡？
- 我知道哪些内容来自公开资料，哪些只是基于公开资料的合理推断吗？
- 我知道作为 PM 应该追问工程团队哪些问题吗？

如果答案是否定的，这章就是模板化失败，需要重写。

## 案例学习方法

不要只背“架构是拆分和连接”。更好的读法是：

1. 先找用户承诺：用户以为产品应该做到什么。
2. 再找系统压力：这个承诺为什么难。
3. 然后找核心对象：系统到底在处理什么东西。
4. 接着看数据流：哪些数据影响了结果。
5. 最后看取舍：为了一个目标，牺牲了什么。

例如 Uber 不是泛泛的“打车 App 架构”，而是实时双边市场：乘客需求、司机供给、价格、ETA、派单、激励和履约风险同时变化。

## 来源原则

- 官方工程博客、官方文档、公开论文、公开演讲可以支撑事实。
- 访谈、Newsletter、X/Twitter 更适合作为观点线索，不能单独证明公司内部架构。
- 公开资料未披露的模型权重、服务数量、内部指标口径、团队权限边界，必须写“公开资料未披露”。
- 不把行业常识写成某家公司内部事实。

## 记住这些

- 架构不是名词堆砌，而是产品承诺如何被稳定兑现。
- 产品经理学架构，重点是判断力，不是背系统名。
- 好案例必须有业务问题、核心对象、数据流、权衡和证据边界。
- 每章都复制同一段开场概念，不是学习资料，是模板噪音。
