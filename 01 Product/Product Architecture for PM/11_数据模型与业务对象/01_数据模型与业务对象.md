# 数据模型与业务对象

## 为什么 PM 要关心数据对象

因为很多产品问题不是页面问题，而是对象定义问题。

比如“订单取消”到底是订单状态变化，还是退款、库存、通知、风控都要跟着变化？如果对象关系没想清楚，功能会越做越乱。

## 四个基本问题

### 1. 这个对象是谁

每个对象都要有唯一身份。用户有 user_id，订单有 order_id，支付有 payment_id 或 payment_intent，内容有 content_id。

PM 要问：业务上同一个东西，系统里会不会被创建成多个东西？

### 2. 它有哪些状态

状态决定用户能做什么。

| 对象 | 常见状态 |
|---|---|
| 订单 | 待支付、已支付、已取消、已完成、退款中 |
| 支付 | 待确认、成功、失败、处理中、需验证 |
| 内容 | 草稿、已发布、下架、审核中 |
| 风险事件 | 待评分、放行、拦截、审核、申诉中 |

### 3. 谁能改变它

状态变化必须有责任边界。用户、客服、系统任务、第三方回调、风控规则，都可能改变状态。

PM 要问：这个状态由谁改？改了以后谁要收到通知？

### 4. 它和其他对象什么关系

一个订单可能关联用户、商品、支付、发票、优惠、风控事件。关系不清，异常处理会很痛苦。

## 对象关系图骨架

```text
用户
 ├─ 创建订单
 │    ├─ 关联商品 / 房源 / 内容
 │    ├─ 关联支付意图
 │    │    ├─ 支付成功 / 失败 / 处理中 / 需验证
 │    │    └─ 可能关联退款
 │    ├─ 关联风险事件
 │    │    └─ 放行 / 验证 / 审核 / 拦截 / 申诉中
 │    └─ 关联通知和客服工单
 └─ 后续行为：取消、评价、投诉、复购
```

这张图帮助 PM 看清：一个页面动作常常会牵动多个对象。比如“取消订单”可能同时改变订单、支付、退款、通知、库存、客服和风控记录。

## 谁能改状态

```text
用户：提交、取消、申诉、确认
系统任务：超时关闭、自动退款、批量同步
第三方回调：支付成功、支付失败、争议通知
客服 / 运营：人工审核、补偿、恢复、备注
风控系统：限制、拦截、要求验证
```

PM 要追问：每个状态由谁改，改完谁收到通知，是否需要审计和回滚。

## 真实例子

### Stripe PaymentIntent

Stripe 的 PaymentIntent 适合学习“支付意图”这个对象。它不是简单扣款请求，而是围绕一次支付意图管理状态和后续动作。

### Airbnb 预订

预订不只是日期和房源，还连接房客、房东、支付、取消、评价、风险和客服。

### Instagram 内容

内容对象不只是图片视频，还包括作者、发布时间、互动、标签、审核状态和推荐特征。

## PM 检查清单

- 对象唯一身份是什么？
- 状态有哪些？状态能不能回退？
- 谁能改状态？是否需要审计？
- 对象之间是什么关系？
- 哪些字段影响权限、风控、推荐或支付？
- 删除、下架、取消、退款是物理删除还是状态变化？
