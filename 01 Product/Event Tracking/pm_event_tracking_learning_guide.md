# 产品经理埋点学习指南

这份文档面向从 0 开始学习埋点的产品经理。你可以先用它理解“为什么要埋点、埋什么、怎么判断一条埋点是否有意义”，再基于具体 App 新建埋点需求。

参考材料：

- `/Users/mac/Desktop/Learning Studio/AuraThemes埋点方案-数数_updated_v4.xlsx`
- 本项目已有文档：`Healthofelderly/docs/thinkingdata_event_mapping.md`

## 1. 一句话理解埋点

埋点就是在用户使用产品的关键动作发生时，记录一条结构化数据。

一条埋点通常回答 5 个问题：

| 问题 | 对应字段 | 例子 |
| --- | --- | --- |
| 谁做的 | 用户 ID、设备 ID、用户属性 | `#account_id`、`#distinct_id`、`is_vip` |
| 在哪里做的 | 页面、入口、来源 | `belong_page=Home`、`source_page=Search` |
| 做了什么 | 事件名 | `paywall_show`、`start_search` |
| 对什么对象做的 | 业务对象属性 | `item_id`、`pack_id`、`sku` |
| 结果如何 | 结果属性 | `is_success=true`、`duration=12.5` |

产品经理设计埋点，本质不是“列很多事件”，而是先定义业务问题，再用事件和属性把问题拆成可计算的数据。

## 2. 埋点能帮 PM 解决什么问题

常见问题可以分成几类：

| PM 问题 | 需要的埋点 |
| --- | --- |
| 用户有没有完成新手引导 | `guide_show`、`guide_page_show`、`guide_complete` |
| 哪个入口带来的订阅最多 | `paywall_show`、`paywall_plan_click`、`paywall_subscribe_success`，属性带 `enter` |
| 用户为什么搜不到内容 | `start_search`、`search_result`，属性带 `keywords`、`has_result`、`result_num` |
| 哪些内容最吸引用户 | `explore_item_expose`、`explore_item_click`，属性带 `item_id`、`position_rank` |
| 广告是否影响体验或收入 | `ad_request`、`ad_load`、`ad_show`、`ad_click`、`ad_complete`、`ad_revenue` |
| 功能按钮有没有人点 | `page_function_click`，属性带 `belong_page`、`function_name` |
| 页面是否真的被浏览 | `page_enter`、`tab_expose_duration`，属性带页面和停留时长 |

一个好埋点的价值在于：它能被用来算指标、定位问题、比较版本、指导决策。

## 3. 埋点是 PM 做还是开发做

通常分工是：产品经理负责设计埋点方案，开发负责实现埋点代码，数据分析、运营或增长同学协助确认指标口径和验收结果。

产品经理不需要知道每一行代码怎么写，但必须说清楚：

- 用户做了什么行为
- 什么时候记录这条行为
- 这条行为属于哪个页面或入口
- 需要记录哪些属性
- 这条数据以后要用来分析什么
- 如何判断研发实现后没有漏报、误报、重复上报

开发需要负责：

- 在正确的代码位置上报事件
- 接入和初始化 SDK
- 保证触发时机准确
- 保证属性名、属性类型和枚举值正确
- 处理失败、取消、重复进入、后台切换等边界情况
- 在本地日志或数据后台协助验证事件是否成功上报

数据或运营同学通常负责：

- 检查这些事件能不能算出目标指标
- 确认指标口径是否统一
- 建看板、漏斗、留存、分群分析
- 上线后发现异常数据并反查埋点或产品路径

最理想的协作流程是：

1. PM 提出业务问题。
2. PM 和数据同学定义指标口径。
3. PM 写埋点需求表。
4. 开发评估触发时机和实现成本。
5. 开发实现埋点。
6. PM、数据和开发一起验收。
7. 上线后持续看数据，发现异常再修正口径或补点。

所以，埋点不是单纯 PM 的工作，也不是单纯开发的工作。PM 是“定义要记录什么和为什么记录的人”，开发是“让这些记录准确发生的人”。

## 4. 先区分 4 类数据

### 4.1 事件数据

事件数据记录“某个时刻发生了什么”。

例子：

- `guide_show`：用户进入引导页
- `paywall_show`：订阅页曝光
- `start_search`：用户发起搜索
- `ad_show`：广告成功展示

事件是行为流水，适合分析转化、路径、频次、留存、漏斗。

### 4.2 事件属性

事件属性描述这次行为的上下文。

例子：

- `paywall_show` 的 `enter`：订阅页从哪里打开
- `ad_show` 的 `ad_position`：广告点位
- `search_result` 的 `result_num`：返回结果数量
- `explore_item_click` 的 `item_id`：用户点了哪个素材

事件属性必须和这一次行为绑定。它回答“这次行为具体是什么情况”。

### 4.3 公共事件属性

公共事件属性是每条事件都尽量带上的上下文。

AuraThemes 表里有：

- `language`：语言
- `country`：国家
- `user_status`：用户身份
- `vip_plan`：会员套餐
- `audit_mode`：审核模式
- `custom_type`：服务 AB test 分组

公共属性适合做所有事件的分组分析，比如“美国用户订阅转化是否更高”“A 版本和 B 版本搜索点击率是否不同”。

### 4.4 用户属性

用户属性描述“这个用户当前是什么状态”，不是某一次行为。

AuraThemes 表里有：

- `channel`：渠道
- `first_login_time`：首次登录时间
- `is_vip`：是否会员
- `vip_plan`：会员套餐类型
- `total_ad_watched`：累计观看广告次数
- `total_wallpaper_downloaded`：累计下载壁纸数量

用户属性适合做人群分层，比如“新用户 vs 老用户”“会员 vs 非会员”“高活跃用户 vs 低活跃用户”。

## 5. 一条埋点应该怎么设计

设计一条埋点时，不要从事件名开始。按这个顺序来：

1. 写出业务问题  
   例：我想知道用户是否看完新手引导。

2. 写出指标  
   例：引导完成率 = `guide_complete` 用户数 / `guide_show` 用户数。

3. 找到关键动作  
   例：进入引导页、轮播页曝光、完成引导。

4. 定义事件  
   例：`guide_show`、`guide_page_show`、`guide_complete`。

5. 定义属性  
   例：`page`、`duration`、`total_duration`。

6. 定义触发时机  
   例：页面首次展示时上报；轮播切换时上报上一页停留时长；点击最后一页 Continue 后上报完成。

7. 定义验收方式  
   例：本地日志能看到事件；后台能按页面看到漏斗；重复进入页面不会误报。

## 6. 埋点表每一列怎么看

AuraThemes 的 `#事件数据` sheet 是典型埋点表。核心列含义如下：

| 列名 | PM 应该怎么理解 |
| --- | --- |
| 事件名 | 给研发和数据平台用的英文唯一标识，建议小写下划线 |
| 事件显示名 | 给人看的中文名 |
| 事件说明 | 什么时候触发，越具体越好 |
| 事件标签 | 事件所属业务模块，比如订阅、广告、搜索 |
| 属性名 | 事件携带的字段 |
| 属性显示名 | 属性中文名 |
| 属性类型 | 文本、数值、布尔、时间、列表 |
| 属性说明 | 属性怎么取值、单位是什么、枚举有哪些 |
| 备注 | 特殊口径、第三方限制、研发注意事项 |

同一个事件常常占多行：第一行写事件本身，后面几行的事件名为空，只是在补充这个事件的其它属性。

## 7. 事件命名怎么判断好不好

推荐格式：`业务对象_动作_结果` 或 `页面/模块_动作`。

好例子：

- `paywall_show`：订阅页曝光
- `paywall_plan_click`：套餐点击
- `paywall_subscribe_success`：订阅成功
- `search_result_click`：搜索结果点击
- `widget_save`：小组件保存

不建议：

- `click1`：不知道点了什么
- `button_click`：太泛，无法分析业务
- `user_action`：太抽象
- `paywall`：不知道是曝光、点击还是成功

产品经理写事件名时，要让研发、数据分析、未来的自己都能看懂。

## 8. 属性怎么设计

### 8.1 属性要能支持分析

属性不是越多越好。判断一个属性是否需要加，可以问：

- 它会不会用于分组？例如按 `enter` 看不同入口订阅转化。
- 它会不会用于筛选？例如只看 `is_premium=true` 的内容。
- 它会不会用于排序或计算？例如 `position_rank`、`duration`、`price`。
- 它会不会用于定位问题？例如 `fail_reason`。

如果一个属性未来完全不会用于分析，它可能不该放进埋点。

### 8.2 属性类型要稳定

数数这类平台通常会根据首次上报确定字段类型，所以类型不要随意变。

| 类型 | 适合场景 | 注意点 |
| --- | --- | --- |
| 文本 | 枚举、ID、名称、渠道 | 不要把要计算的数字写成文本 |
| 数值 | 价格、时长、次数、排序 | 要写清单位，比如秒、美元 |
| 布尔 | 是否首次、是否成功、是否会员 | 用 true/false，不要混用 0/1 |
| 时间 | 首次登录、会员到期 | 统一格式和时区 |
| 列表 | 标签、兴趣集合 | 使用前先确认平台支持和查询方式 |

### 8.3 枚举要提前写清楚

比如 `plan_type` 不要只写“套餐类型”，要写：

- `weekly`
- `monthly`
- `yearly`

比如 `close_method` 不要只写“关闭方式”，要写：

- `close_button`
- `back`
- `subscribe`

枚举不清楚，后面数据会出现大小写混乱、中文英文混用、同义词重复。

## 9. 页面类埋点

页面类埋点回答两个问题：

- 用户去了哪些页面？
- 用户从哪里来到这个页面？

常用事件：

- `page_enter`：进入页面
- `tab_expose`：Tab 曝光
- `tab_expose_duration`：Tab 浏览时长
- `tab_expose_change`：Tab 切换
- `page_function_click`：页面功能点击

关键属性：

- `belong_page`：当前页面
- `source_page`：来源页面
- `function_name`：点击的功能名
- `duration` 或 `expose_duration`：停留时长

设计页面值时要统一，比如 AuraThemes 的页面值表里统一用：

- `Home`
- `Widgets`
- `Mine`
- `Search`
- `ThemePackDetail`
- `Subscribe`

不要一会儿写 `Home`，一会儿写 `home_page`，一会儿写 `首页`。

## 10. 曝光、点击、成功为什么要分开

很多功能都需要三段式埋点：

| 阶段 | 例子 | 意义 |
| --- | --- | --- |
| 曝光 | `paywall_show` | 用户是否看到了 |
| 点击 | `paywall_plan_click` | 用户是否有意愿 |
| 成功 | `paywall_subscribe_success` | 用户是否完成结果 |

只埋成功事件的问题是：你不知道用户在哪一步流失。

只埋点击事件的问题是：你不知道点击后是否成功。

只埋曝光事件的问题是：你只能知道流量，不能知道转化。

## 11. 用 AuraThemes 表举几个完整案例

### 11.1 新手引导

目标：分析用户是否完成引导、卡在哪一页。

事件：

- `guide_show`：进入引导页
- `guide_page_show`：轮播图曝光
- `guide_complete`：完成引导

关键属性：

- `page`：第几页轮播图
- `duration`：当前页停留时长
- `total_duration`：引导总时长

能回答的问题：

- 引导页整体完成率是多少？
- 哪一页停留时间过长？
- 用户是否在某一页大量流失？

### 11.2 订阅页

目标：分析订阅漏斗和不同入口转化。

事件：

- `paywall_show`
- `paywall_plan_click`
- `paywall_subscribe_popup_show`
- `paywall_subscribe_popup_close`
- `paywall_subscribe_success`
- `paywall_close`
- `paywall_restore`

关键属性：

- `enter`：订阅入口
- `ui_type`：订阅 UI 类型
- `plan_type`：套餐类型
- `sku`：商品 SKU
- `price`、`currency`：价格和币种
- `is_trial`：是否试用
- `duration`：停留时长
- `close_method`：关闭方式

能回答的问题：

- 哪个入口打开订阅页最多？
- 哪个入口的订阅成功率最高？
- 用户更偏好周订阅、月订阅还是年订阅？
- 是页面没吸引力，还是商店弹窗阶段流失严重？

### 11.3 搜索

目标：分析搜索意图、结果质量、点击转化。

事件：

- `click_search_button`
- `trending_keyword_click`
- `start_search`
- `search_result`
- `search_result_click`
- `you_might_like_click`

关键属性：

- `keywords`：搜索词
- `has_result`：是否有结果
- `result_num`：返回条数
- `position_rank`：点击位置
- `item_id`、`item_name`、`item_type`：点击对象

能回答的问题：

- 用户最常搜什么？
- 哪些搜索词没有结果？
- 搜索结果是否被点击？
- 第几位结果最容易被点？

### 11.4 广告

目标：分析广告加载、展示、点击、完成和收入。

事件：

- `ad_request`
- `ad_load`
- `ad_show`
- `ad_click`
- `ad_complete`
- `ad_fail`
- `ad_revenue`

关键属性：

- `ad_position`：广告点位
- `ad_type`：激励视频、插屏、横幅、原生
- `ad_platform`：广告平台
- `loading_duration`：加载时长
- `ad_is_load`：是否加载成功
- `fail_reason`：失败原因
- `ad_ecpm`、`ad_revenue`：收入相关

能回答的问题：

- 哪个广告点位加载失败多？
- 哪种广告类型收入最高？
- 激励视频看完率如何？
- 某个版本是否导致广告加载变慢？

### 11.5 内容浏览

目标：分析内容曝光、点击、安装/下载。

事件：

- `explore_item_expose`
- `explore_item_click`
- `theme_pack_detail_show`
- `theme_pack_install_click`
- `theme_pack_install_success`
- `wallpaper_download`

关键属性：

- `category_type`
- `tag_name`
- `item_id`
- `item_name`
- `item_type`
- `position_rank`
- `is_premium`
- `unlock_method`
- `is_success`

能回答的问题：

- 哪类内容曝光多但点击低？
- 免费内容和会员内容点击差异如何？
- 列表第几个位置点击最好？
- 详情页到安装成功的转化率是多少？

## 12. 首次事件、普通事件、用户属性更新

### 12.1 首次事件

只需要记录第一次发生的事件，比如：

- `new_device`
- 首次权限弹窗出现
- 首次联系人权限弹窗

意义：用于新增、激活、首次触达分析。

### 12.2 普通事件

每次发生都记录，比如：

- `page_enter`
- `ad_show`
- `start_search`
- `paywall_plan_click`

意义：用于行为次数、路径和转化。

### 12.3 用户属性更新

用户状态变化时更新，比如：

- 订阅成功后更新 `is_vip=true`
- 会员到期后更新 `vip_expiration_time`
- 看完广告后累加 `total_ad_watched`

意义：用于用户分群和长期画像。

## 13. 客户端埋点和服务端埋点

客户端埋点适合记录用户在 App 内发生的行为：

- 页面曝光
- 按钮点击
- 搜索
- 广告展示
- 本地弹窗关闭

服务端埋点适合记录更可靠或客户端拿不到的数据：

- 广告投放消耗：`server_ad_cost`
- 订阅续费、退款、取消：`server_subscription_status`

原则：

- 发生在用户设备上的交互，优先客户端。
- 涉及收入、退款、投放成本、服务端通知，优先服务端。
- 支付成功这类关键数据，客户端可以记录体验路径，但最终收入口径应以服务端或商店回调为准。

## 14. PM 设计埋点的标准模板

你之后新建埋点时，可以用这个模板：

| 字段 | 填写说明 |
| --- | --- |
| 业务目标 | 为什么需要这条埋点 |
| 指标 | 这条埋点参与计算什么指标 |
| 事件名 | 英文小写下划线 |
| 事件显示名 | 中文名称 |
| 触发时机 | 用户做什么、系统到什么状态时上报 |
| 去重规则 | 是否只上报一次、页面重复出现是否重复上报 |
| 属性 | 每个属性的名称、类型、取值、单位 |
| 关联页面 | `belong_page`、`source_page` 怎么填 |
| 异常情况 | 失败、取消、超时、无结果怎么记录 |
| 验收方式 | 如何在日志和后台验证 |

## 15. 设计一组埋点的步骤

当你要给一个新功能设计埋点，可以按这个流程：

1. 画出用户路径  
   例：入口点击 -> 页面曝光 -> 选择内容 -> 开始 -> 完成 -> 结果页。

2. 找关键转化点  
   每一步是否需要衡量流失。

3. 找关键对象  
   用户操作的是课程、商品、文章、广告、任务还是弹窗。

4. 定义事件集合  
   曝光、点击、开始、成功、失败、关闭。

5. 定义公共属性  
   页面、来源、用户状态、语言、国家、AB 分组。

6. 定义业务属性  
   对象 ID、对象名称、类型、排序、价格、是否会员。

7. 对齐研发触发时机  
   页面生命周期、按钮回调、接口返回、播放器状态、支付回调。

8. 设计验收用例  
   正常路径、取消路径、失败路径、重复进入、后台切换。

## 16. 埋点常见坑

### 16.1 事件太泛

只写 `button_click`，后续不知道点击了什么。应该写具体事件或至少带 `function_name`。

### 16.2 属性类型混乱

同一个 `price` 有时上报数字，有时上报 `"9.99 USD"`。这样无法求和、均值和分组。

### 16.3 触发时机不清楚

“页面展示时上报”要明确是：

- 页面创建时？
- 页面真正出现在屏幕时？
- 每次返回页面都上报？
- 只首次上报？

### 16.4 成功和点击混在一起

点击安装不等于安装成功。应拆成：

- `theme_pack_install_click`
- `theme_pack_install_success`

### 16.5 缺少失败原因

只知道失败，不知道为什么失败。关键失败事件应带：

- `fail_reason`
- `error_code`
- `is_success`

### 16.6 页面值不统一

同一个页面不要出现多种写法。应维护统一页面字典。

### 16.7 只按功能埋，不按问题埋

埋点不是“每个按钮都记录一下”。如果一条数据不能服务指标、漏斗、分群或问题定位，就要谨慎。

## 17. 如何判断一个埋点方案是否完整

可以用这 8 个问题检查：

1. 是否覆盖了核心用户路径？
2. 是否能计算曝光到点击、点击到成功的转化？
3. 是否能区分不同入口、不同页面、不同内容？
4. 是否能区分成功、失败、取消？
5. 是否有统一用户 ID 和设备 ID？
6. 是否有公共属性支持分群？
7. 属性类型和枚举是否稳定？
8. 研发能否明确知道什么时候上报？

## 18. 套到老年人健身 App 的思路

如果后续要给老年人健身 App 新建埋点，可以先从这些业务问题出发：

| 业务问题 | 可能需要的事件 |
| --- | --- |
| 用户是否完成初始问卷/引导 | 引导曝光、步骤曝光、选项点击、完成 |
| 用户是否进入健康计划 | 页面进入、卡片点击 |
| 用户是否记录喝水/用药/体重等健康数据 | 功能页进入、保存成功、修改目标 |
| 用户是否开始运动课程 | 课程详情曝光、开始播放、暂停、完成 |
| 哪些提醒有效 | 提醒展示、提醒点击、提醒后完成行为 |
| 会员/订阅是否转化 | 订阅页曝光、套餐点击、支付弹窗、订阅成功 |

不要一开始就把所有页面都埋满。建议先选 1 条主路径，例如：

首页 Exercise 卡片点击 -> 运动详情页曝光 -> 点击 Start Workout -> 视频播放 -> 完成训练 -> 返回首页活跃分钟更新。

这条路径能回答“用户是否真的开始并完成运动”，比单纯记录一个按钮点击更有价值。

## 19. 新建埋点前的 PM 检查清单

写埋点需求前，先确认：

- 这个埋点要回答什么问题？
- 这个问题能否转化成指标？
- 指标分母是什么，分子是什么？
- 事件是曝光、点击、开始、成功、失败还是关闭？
- 属性是否足够支持分组和定位？
- 属性是否有单位、枚举和类型？
- 是否需要用户属性同步？
- 是否应该由客户端、服务端或第三方 SDK 自动采集？
- 是否涉及隐私、权限、支付、健康数据等敏感信息？
- 研发是否能根据触发时机准确实现？

## 20. 一条埋点需求示例

示例：运动详情页曝光。

| 字段 | 示例 |
| --- | --- |
| 业务目标 | 看用户是否从首页进入运动课程详情 |
| 指标 | 运动详情页访问人数、首页运动入口点击到详情页曝光转化 |
| 事件名 | `exercise_detail_show` |
| 事件显示名 | 运动详情页曝光 |
| 触发时机 | 运动详情页首次真实展示到屏幕时 |
| 去重规则 | 同一次进入只上报一次；退出后再次进入可再次上报 |
| 属性 | `workout_id` 文本；`workout_name` 文本；`duration_minutes` 数值；`source_page` 文本 |
| 关联页面 | `belong_page=ExerciseDetail`，`source_page=Plan` |
| 异常情况 | 课程 ID 缺失时不上报或上报 `unknown`，需提前约定 |
| 验收方式 | 从首页点击运动卡片后，日志出现 1 条事件；返回再进入出现新一条 |

示例本身不是让你立刻实现，而是展示 PM 该怎样把一个想法写成研发可执行、数据可分析的需求。

## 21. 下一步建议

学习顺序建议：

1. 先看懂 `#事件数据`：事件名、触发时机、属性。
2. 再看 `#公共事件属性`：理解所有事件都应该带哪些上下文。
3. 再看 `#用户数据`：理解用户状态和行为事件的区别。
4. 最后看 `页面相关对应的值`：学习维护页面字典。
