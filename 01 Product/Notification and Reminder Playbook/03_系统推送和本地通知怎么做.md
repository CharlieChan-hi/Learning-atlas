> 一句话版本：系统通知分成本地通知和远程推送。用户自定义运动时间，优先用本地通知；服务器运营消息，才用远程推送。

## 1. 本地通知是什么

本地通知由 App 在设备本地创建，由系统在指定时间展示。比如用户设置“每天早上 8 点提醒我运动”，这个场景不需要服务器也能完成。

适合本地通知的场景：

- 每天固定时间运动。
- 每周一、三、五提醒训练。
- 用户主动设置的药物、饮水、体重记录提醒。
- 完成一次训练后，次日同一时间提醒。

优势：

- 稳定，不依赖网络。
- 保护隐私，不需要把用户日程发到服务器。
- 更符合“用户自定义提醒”的心理预期。

限制：

- 必须有系统通知权限。
- 用户关闭权限后无法弹出系统通知。
- 重复规则要自己维护，例如星期、时区、夏令时变化。

## 2. 远程推送是什么

远程推送由服务器发出，通过 Apple APNs 或 Firebase Cloud Messaging 等通道送到设备。

适合远程推送的场景：

- 新训练计划上线。
- 用户连续多天未打开 App 的温和召回。
- 后台根据用户状态生成的个性化建议。
- 版本活动、订阅权益、内容更新。

不适合远程推送的场景：

- 用户已经设置好的固定每日提醒。
- 每个用户都一样的频繁运营消息。
- 没有明确点击去向的泛泛提示。

## 3. iOS 常见技术链路

iOS 的通知通常涉及：

1. App 请求通知权限。
2. 用户同意后，App 可以创建本地通知。
3. 如果要做远程推送，设备需要注册 APNs token。
4. 使用 Firebase Cloud Messaging 时，还需要将 APNs token 与 FCM 关联。
5. 用户点击通知后，App 根据 payload 或本地 identifier 跳转到对应页面。

对当前项目来说，源码里已经有 `FirebaseManager`，但 `AppDelegate` 里的 Firebase 配置目前被注释。也就是说远程推送有骨架，但还没有真正启用。

## 4. 一条系统通知通常包含什么

| 字段 | 说明 | 健康运动示例 |
|---|---|---|
| identifier | 通知唯一 ID，用于取消或更新 | `exercise-reminder-mon-0800` |
| title | 标题，短而明确 | `Time for gentle movement` |
| body | 简介，说明价值或动作 | `Your 8-minute plan is ready.` |
| trigger | 触发时间或条件 | 每天 8:00 AM |
| sound | 声音 | 默认或轻柔音 |
| badge | 角标 | 健康类不建议滥用 |
| userInfo | 点击后跳转信息 | `route=today_exercise` |

## 5. 通知点击后的去向

推送不能只把用户带回首页。用户点击通知时，心里期待的是“处理这条通知对应的事”。

健康运动 App 推荐去向：

- 每日运动提醒：今日训练页。
- 未设置提醒提示：运动提醒设置页。
- 完成后复盘：训练完成页或运动记录页。
- 新计划推荐：计划详情页。

## 6. 本地通知和远程推送的产品分工

| 任务 | 推荐方式 | 原因 |
|---|---|---|
| 用户每天 8 点运动提醒 | 本地通知 | 用户自定义、重复稳定、隐私友好 |
| 服务器推荐新内容 | 远程推送 | 内容来自服务端 |
| 用户刚完成训练后引导设置提醒 | App 内弹窗 | 用户正在 App 内 |
| 首页提醒今天没运动 | 首页卡片 | 低打断 |
| 连续 7 天未打开 App | 远程推送，低频 | 召回场景 |

## 7. 对当前 App 的建议

第一阶段不要急着启用 Firebase 远程推送。先做：

1. 运动提醒设置模型。
2. 本地通知管理器。
3. More 页入口。
4. 首页 Exercise 卡显示提醒状态。
5. 训练完成后引导用户设置提醒。

等这些基础体验稳定后，再考虑远程推送、分群、运营活动和服务端策略。
