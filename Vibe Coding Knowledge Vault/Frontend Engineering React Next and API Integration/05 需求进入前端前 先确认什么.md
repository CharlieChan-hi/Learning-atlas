---

## 📖 这一节在讲什么

这页不是继续讲概念，而是把前端工程化里最实用的检查维度整理成一份可直接拿来用的清单。

## 🎯 需求进入前端前，先确认什么

- 页面入口是什么
- 主要用户动作是什么
- 页面需要哪些状态
- 依赖哪些接口
- 错误和空态怎么处理

## 🎯 联调前要确认什么

- 请求参数是否清楚
- 成功返回结构是否明确
- 失败结构是否明确
- 鉴权和角色影响是否清楚
- 是否有异步处理中状态

## 🎯 联调时要盯什么

- 页面有没有在正确时机发请求
- 返回字段是否和页面逻辑一致
- loading、成功、失败状态是否都能走通
- 是否有明显的旧数据残留或状态错乱

> 🌐 **In English:**
> Frontend delivery quality depends on aligning entry flow, states, API contracts, and validation before and during integration.

## ⚠️ 最常见的前端实战失误

### 失误一：只关心主流程，不关心异常路径

### 失误二：字段能显示就算完成

### 失误三：接口联调完成就不回归旧路径

### 失误四：页面逻辑和设计稿对齐了，但和真实后端状态没对齐

## 🧪 一份最实用的前端交付检查单

你可以在任何页面功能上快速过一遍：

1. 路由入口清楚吗
2. 状态定义完整吗
3. 请求和响应契约清楚吗
4. 失败和空态处理了吗
5. 有最小验证动作吗

## 📝 记住这些

- 前端工程化的真正难点，在于把状态、接口和用户体验一起拉齐。
- 一份好的清单，会比很多零散经验更稳定。
- PM 很适合用清单方式参与联调和验收。
