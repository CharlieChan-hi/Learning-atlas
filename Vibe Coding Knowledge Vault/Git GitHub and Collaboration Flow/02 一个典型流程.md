> 📌 **一句话版本：** 从需求到提交 PR，不只是“改完代码再点提交”，而是一条包含理解需求、控制范围、形成改动、验证结果和发起协作的完整链路。

---

## 📖 这一节在讲什么

很多 PM 会看到工程同学说“我提个 PR 你看一下”，但不知道这句话背后到底经历了什么。这一节把从需求到 PR 的过程拆清楚，帮助你理解工程协作的节奏和关键判断点。

## 🧠 一个典型流程

1. 明确需求与边界
2. 找到相关代码或模块
3. 新建分支
4. 做局部改动
5. 本地验证
6. 查看 diff
7. 提交 commit
8. push 到远端
9. 发起 PR
10. review、修改、合并

## 🎯 每一步对 PM 的意义

### 明确需求与边界

这一步决定后面是不是会“越改越大”。如果边界不清，AI 和工程师都容易脑补。

### 新建分支

这是在隔离本次工作，避免把实验性改动直接污染主线。

### 本地验证

这一步是在回答：“我不是只改了，我还确认它至少基本能跑。”

### 查看 diff

这是你真正看到“这次到底改了什么”的地方。

### 发起 PR

PR 不只是提交流程，它也是一次协作沟通。里面应该能让人看懂：

- 为什么改
- 改了哪些地方
- 如何验证
- 还有什么风险

> 🌐 **In English:**
> A pull request is not just a code upload. It is a collaboration checkpoint that explains what changed, why it changed, and how it was verified.

## 💡 如果是 AI 帮你改代码，这条链路更重要

因为 AI 非常容易出现两类问题：

- 改动超范围
- 改完没有验证

所以当 AI 参与时，更要坚持：

- 先开分支
- 只改局部
- 看 diff
- 跑验证
- 再发 PR

## ⚠️ 最容易出问题的地方

### 需求没拆清

最后 PR 里混进很多无关改动。

### 没看 diff

以为只改了一个按钮，结果顺手动了别的配置和逻辑。

### 没写验证说明

review 的人看不懂你是怎么确认功能正常的。

### 提交太大

一个 PR 同时混了样式、逻辑、重构、配置，review 成本会非常高。

## 🧪 PM 看 PR 时可以关注什么

- 改动是否对准需求
- 有没有明显超范围修改
- 风险和验证是否写清楚
- 关联模块是否需要额外回归
- 是否涉及接口、权限、配置、埋点、日志等隐藏影响

## 🎯 一个更成熟的 PR 描述应该包含

- 背景：为什么要做
- 变更点：具体改了什么
- 不改什么：明确边界
- 验证方式：怎么证明它有效
- 风险提示：可能影响哪里

## 📝 记住这些

- PR 是工程协作的沟通节点，不只是代码上传动作。
- 从需求到 PR 的全过程里，PM 最该关心的是边界、验证和影响范围。
- AI 参与越多，Git 和 PR 流程越应该更严格。
