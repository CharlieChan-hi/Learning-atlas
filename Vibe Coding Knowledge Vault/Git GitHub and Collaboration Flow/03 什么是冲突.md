> 📌 **一句话版本：** 冲突和回滚不是 Git 的“事故模式”，而是多人协作和快速迭代里的正常机制；关键不在于别遇到，而在于你能不能理解边界和影响。

---

## 📖 这一节在讲什么

很多人一看到冲突就慌，一听到回滚就觉得“出大事了”。其实这些动作本身并不罕见，真正重要的是你能不能看懂它们意味着什么。

## 🧠 什么是冲突

冲突通常意味着两边都改了同一段内容，Git 不知道应该自动保留哪一边。

## 🧠 什么是回滚

回滚不是简单“后悔了”，而是在当前改动引发问题时，把系统尽量带回更稳定的状态。

## 🎯 PM 最值得关注什么

- 冲突发生在哪个模块
- 回滚会影响哪些功能
- 这次冲突是文本层，还是业务逻辑层
- 回滚后是否要补数据或配置修正

> 🌐 **In English:**
> Merge conflicts and rollbacks are normal collaboration mechanisms. The real question is whether you understand the impact and scope behind them.

## 🧩 为什么冲突并不等于“代码坏了”

冲突很多时候只是说明：

- 两边都在积极推进
- 两边都改到了同一片区域
- Git 不能替你做业务判断

也就是说，冲突不是事故本身，而是一个“需要人工决策”的提醒。

## 🎯 文本冲突和逻辑冲突有什么区别

### 文本冲突

Git 能明显看出同一段文本两边都改了，所以要求你手动选择。

### 逻辑冲突

更隐蔽。代码可能已经成功合并了，但业务逻辑其实互相打架。

这也是为什么“没有冲突”不代表“没有风险”。

## 💡 回滚真正难的地方

很多人把回滚想成“点一下返回上一版”，但真实世界里往往没这么简单。因为除了代码，还有可能涉及：

- 数据结构变化
- 缓存状态变化
- 外部服务状态变化
- 用户已经产生的新数据

所以 PM 理解回滚时，最好把它看成“恢复稳定性”的动作，而不是简单的“反向操作”。

> 🌐 **In English:**
> A rollback is often a stability decision, not merely a reverse code action, because data and side effects may already exist.

## ⚠️ PM 在冲突和回滚时最容易忽略什么

### 忽略一：只看技术合并，不看业务意图

代码能合，不代表产品逻辑就对。

### 忽略二：只关心能不能退，不关心退完会怎样

退完后用户状态、数据和页面体验可能仍然需要收口。

### 忽略三：把回滚当失败

很多时候回滚恰恰是成熟团队在做理性风险控制。

## 🧪 一个简单判断模板

以后遇到冲突或回滚，你可以先问：

1. 这次争的是哪块逻辑
2. 影响哪些用户路径
3. 如果退回去，哪些状态会变化
4. 退完之后还需要补什么动作

只要能答出这四句，理解就会比单纯“看见 Git 弹红字”深很多。

## 📝 记住这些

- 冲突本身不是失败，关键是边界和意图要判断清楚。
- 回滚不是羞耻动作，而是风险控制动作。
- PM 理解这层之后，会更会判断发布风险和协作节奏。
