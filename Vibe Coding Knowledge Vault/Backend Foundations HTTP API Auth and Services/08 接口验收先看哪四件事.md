## 📖 这一节在讲什么

很多 PM 到“接口验收”这一步时，不知道自己到底应该看什么。这页就是给你一套面向 PM 的接口验收框架。

## 🎯 接口验收先看哪四件事

### 1. 契合需求吗

返回内容和业务目标是否真的一致。

### 2. 状态清楚吗

成功、失败、处理中这些状态，前后端是不是都说得清楚。

### 3. 错误可理解吗

失败时，前端和用户是否能拿到足够明确的反馈。

### 4. 风险边界清楚吗

重复请求、权限、配额、异步任务和历史兼容这些高风险点是否考虑了。

> 🌐 **In English:**
> API acceptance for PMs is about requirement fit, state clarity, understandable failures, and clear risk boundaries.

## 💡 PM 和工程师的验收视角有什么不同

工程师更容易关注：

- 实现是否通
- 错误是否可复现
- 技术路径是否合理

PM 更应该补上：

- 需求有没有被误实现
- 状态和反馈是否适合用户理解
- 边界是否会影响交付节奏

## ⚠️ 接口验收最容易漏掉的几类情况

### 漏项一：只测成功路径

### 漏项二：只看字段，不看业务语义

### 漏项三：没有看异步和历史状态

### 漏项四：没有验证重复请求和权限边界

## 🧪 一个很实用的接口验收清单

你可以快速问：

1. 这个接口成功到底意味着什么
2. 失败时前端和用户各自看到什么
3. 是否有处理中或异步后续状态
4. 是否存在重复执行风险
5. 是否有权限、配额或限流边界

## 📝 记住这些

- 接口验收不是只看“有没有回数据”。
- PM 的独特价值，在于把业务目标、状态表达和风险边界一起看。
- 接口验收看得越清楚，后面的联调和上线返工越少。
