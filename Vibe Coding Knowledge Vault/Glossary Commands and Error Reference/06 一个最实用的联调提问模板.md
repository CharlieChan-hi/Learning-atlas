> 📌 **一句话版本：** 联调效率低，很多时候不是大家不努力，而是问题描述方式太散；这页给你一套适合 PM 在接口联调里直接使用的提问模板。

---

## 📖 这一节在讲什么

联调时最怕的不是“有问题”，而是问题被描述成：

- “这里不对”
- “接口好像有问题”
- “前端没拿到”

这种说法很难让任何人快速定位。这页就是把联调沟通结构化。

## 🎯 一个最实用的联调提问模板

你可以按这个顺序去问：

1. 用户做了什么动作
2. 当前页面入口是什么
3. 发出的请求长什么样
4. 接口返回了什么
5. 页面最终展示成了什么
6. 和预期差异在哪

> 🌐 **In English:**
> Integration discussions become much more effective when user action, request, response, UI state, and expected behavior are all made explicit.

## 💡 适合直接拿来用的句式

- “这一步用户的动作是什么？”
- “页面发出的请求参数能确认一下吗？”
- “接口返回成功在这里具体意味着什么？”
- “前端拿到这个字段之后，页面应该展示成什么？”
- “当前是接口错位，还是状态映射错位？”

## ⚠️ 联调时最应该避免的说法

- “感觉不太对”
- “你们那边看一下”
- “接口不是都通了吗”

这些说法会让问题停留在模糊层。

## 📝 记住这些

- 联调提问结构越清晰，定位速度越快。
- PM 在联调中最有价值的能力，不是替代工程师，而是把问题整理得足够清楚。
- 一套稳定模板，会比临场发挥更有效。
