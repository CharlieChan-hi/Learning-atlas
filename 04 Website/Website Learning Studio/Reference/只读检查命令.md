## 这一节在讲什么

这一节提供 command cheatsheet。它帮助你理解常见命令，不鼓励你盲目复制执行。

执行命令前，先确认路径、作用和是否会修改文件。

## 只读检查命令

| Command | 作用 | 是否修改文件 |
|---------|------|--------------|
| `ls -la "path"` | 查看目录内容 | 否 |
| `find "path" -maxdepth 2 -type d -print` | 查看目录结构 | 否 |
| `git -C "path" status --short` | 查看 Git 工作区状态 | 否 |
| `find "path" -type f -name "*.md"` | 查找 Markdown 文件 | 否 |
| `grep -n "text" file.md` | 在文件中查找文字 | 否 |

## 项目常见脚本

| Script | 常见作用 | 学习提醒 |
|--------|----------|----------|
| `pnpm dev` | 启动本地开发环境 | 可能占用端口 |
| `pnpm build` | 构建发布文件 | 会生成输出目录 |
| `pnpm check` | 做类型或框架检查 | 通常用于发现错误 |
| `pnpm test` | 运行测试 | 可能需要依赖完整 |
| `pnpm verify` | 综合验证 | 时间更长，范围更广 |

## Charlie 项目中的命令

Charlie 根目录 `package.json` 里有 `dev`、`build`、`check`、`verify`、`test`。学习时先读这些脚本，不要在不理解影响范围时执行。

例如：

```bash
git -C "/Users/mac/Desktop/charliechanstudio.com" status --short
```

这个命令是只读的，用来查看已有变化。

## 小练习

1. 选择一个只读命令，解释它的作用。
2. 选择一个可能生成文件的命令，说明为什么要谨慎。
3. 写一句执行命令前的自查提醒。

## 检查问题

1. `ls` 会修改文件吗？
2. `build` 可能会产生什么结果？
3. 为什么命令里的路径要加引号？

## 常见误解

常见误解是速查表里的命令都可以无脑执行。速查表的作用是帮助理解，不是替代判断。

另一个误解是只读命令一定没有输出风险。只读命令通常不改文件，但输出可能很长，仍要控制范围。

## 记住这些

- 先理解，再执行。
- 分清只读命令和会写入的命令。
- 路径和当前目录会影响结果。
