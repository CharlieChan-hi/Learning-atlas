## 这一节在讲什么

这一节收集学习个人网站时常见错误。它不是报错全集，而是帮助你建立排查方向。

遇到问题时，先分类，再深入。

## 常见错误

| 现象 | 可能原因 | 先做什么 |
|------|----------|----------|
| 网站打不开 | DNS、部署、服务或网络问题 | 先确认本地和线上分别是否正常 |
| 页面空白 | 构建失败、路径错误、脚本错误 | 查看构建输出和控制台 |
| 样式丢失 | CSS 未加载、路径错误、构建配置问题 | 检查资源路径 |
| 项目没显示 | 内容文件缺失、frontmatter 错误、读取函数报错 | 检查内容文件和 schema |
| 中文页缺失 | locale 文件缺失或 route 未配置 | 检查 `zh` 页面和内容 |
| 部署失败 | build command 错误、依赖版本不匹配 | 先读部署日志 |
| 本地能跑线上不行 | 环境差异、输出目录错误 | 比较本地构建和平台设置 |
| 页面难读 | 字号、行高、对比度、信息密度问题 | 回到设计审查清单 |
| 首页没重点 | 信息层级混乱 | 重排第一屏内容 |

## 现实例子

如果项目卡片不显示，不要先改样式。你要先问内容有没有被读到，数据是否通过 schema，页面是否调用了读取函数。

如果首页看起来不专业，不要马上加颜色。先检查标题、说明、间距、层级和文案是否清楚。

## Charlie 项目案例

Charlie 项目里 `packages/content/src/schema.ts` 会校验项目 frontmatter。如果内容文件的 `slug`、`locale` 或 `date` 格式不对，读取内容时可能报错。

`packages/content/src/projects.ts` 还会检查文件名是否符合 `<slug>.<en|zh>.mdx`。这说明内容错误可能来自文件名，也可能来自 frontmatter。

## 小练习

1. 选一个错误现象，写出 3 个可能原因。
2. 判断它属于内容、代码、设计还是部署。
3. 写出第一步检查动作。

## 检查问题

1. 页面问题是否一定是 CSS 错？
2. 内容不显示时为什么要检查 schema？
3. 部署失败时应该先读什么？

## 常见误解

常见误解是看到错误就随便改代码。排查要先定位层级。

另一个误解是忽略文件名规则。真实项目常通过文件名生成 route 或 locale。

## 记住这些

- 先分类，再排查。
- 内容、设计、代码、部署都有可能出问题。
- 第一反应不要乱改正式项目。
