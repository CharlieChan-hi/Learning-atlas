> 一句话版本：项目故事不是罗列功能，而是说明背景、问题、方法、结果和你在其中的判断。

---

## 这一节在讲什么

这一节教你写 project storytelling。你会学习如何把项目写成读者能理解的故事，而不是技术清单或作品截图。

个人网站里的项目承担证明作用，所以项目故事要证明你的能力和思考。

## 概念解释

一个项目故事通常包含五部分：背景、问题、你的角色、解决方式、结果或学习。

背景说明项目为什么存在。问题说明你面对什么挑战。角色说明你负责什么。解决方式说明你如何判断和执行。结果说明项目带来了什么，或者你学到了什么。

如果项目还没完全完成，也可以写当前阶段和下一步。诚实的进度比夸张包装更可信。

## 现实例子

弱项目介绍：这是一个专注训练平台，有很多功能。

更好的介绍：这个项目从毕业设计出发，用游戏化任务、奖励和行为报告帮助用户理解自己的专注习惯。当前版本先建立可运行容器和核心流程，后续再扩展评估模型。

第二种写法更能体现项目动机和阶段。

## Charlie 项目案例

`content/projects/focus-training.en.mdx` 里写到：

```text
The purpose of this project page is to separate portfolio storytelling from live product experience. The project page explains the thinking; the app path carries the runnable demo.
```

这句话很适合学习。它把项目故事和可运行 demo 分开：项目页解释思考，app 路径承载体验。

这也是个人网站项目页的重要原则。页面不只是放链接，还要告诉读者为什么值得看。

## 小练习

1. 选一个项目，写出它的背景和问题。
2. 写一句你在项目里的角色。
3. 写一句这个项目证明了你什么能力。

## 检查问题

1. 项目故事是否等于功能列表？
2. 为什么要说明你的角色？
3. 项目未完成时是否完全不能展示？

## 常见误解

常见误解是项目页必须只展示结果。过程、判断和限制也能证明能力。

另一个误解是把项目写得越大越好。读者更关心你是否具体、真实、有判断。

## 记住这些

- 项目故事要有背景和问题。
- 写清你的角色和判断。
- 项目页解释思考，demo 承载体验。
