> 📌 **一句话版本：** 接口联调真正难的，不只是“字段对不对”，而是当字段、状态、日志和用户现象对不上时，你怎么快速判断问题落在哪一层。

## 📖 这一节在讲什么

这页专门讲“接口联调出问题时，怎么靠日志、状态和用户现象把问题缩小范围”。它和前面的接口设计页不同，更偏排查与协作。

## 🧠 联调时最常见的错位

- 前端认为参数没问题
- 后端认为字段没问题
- 用户仍然看到错误结果

这时真正需要的，不是双方继续“感觉式争论”，而是把现场拆开看。

## 🎯 一条联调排查链

你可以顺着这条链去看：

1. 用户做了什么动作
2. 前端发了什么请求
3. 接口收到了什么参数
4. 后端执行到了哪一步
5. 返回了什么状态和结构
6. 前端最后展示成了什么

> 🌐 **In English:**
> API integration debugging works best when you trace one full chain: user action, request, backend processing, response, and frontend rendering.

## 💡 日志在联调里最有价值的地方

日志最适合回答：

- 这个请求有没有真的进来
- 参数长什么样
- 执行在哪一步失败
- 后续任务有没有被触发

这会比单纯看页面现象更可靠。

## ⚠️ 最常见的联调误判

### 误判一：看到 200 就当成功

有时接口只是“接收成功”，业务并没有真正完成。

### 误判二：前端参数看着对，就默认后端一定按同样理解处理

中间可能存在字段名、类型或默认值理解偏差。

### 误判三：后端返回了数据，就默认前端一定能正确展示

很多问题会卡在状态映射或界面收口上。

## 🧪 一个很适合 PM 用的联调排查模板

你可以这样整理问题：

- 现象：用户看到了什么
- 动作：用户做了什么
- 请求：发了哪个接口
- 结果：接口回了什么
- 展示：页面最终怎么呈现
- 线索：日志或状态码里有什么关键信息

## 📝 记住这些

- 接口联调的问题，常常不是单边问题，而是整条链对不上。
- 日志和状态，是把“感觉有问题”变成“能定位问题”的关键。
- PM 只要会沿着链路问问题，就能显著提升联调效率。
