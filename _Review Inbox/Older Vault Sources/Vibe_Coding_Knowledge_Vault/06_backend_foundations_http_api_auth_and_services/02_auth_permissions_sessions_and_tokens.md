> 📌 **一句话版本：** 鉴权解决“你是谁”，权限解决“你能做什么”，而会话、Cookie、Token 则是系统记录和传递身份信息的常见方式。

---

## 📖 这一节在讲什么

为什么有的用户能看、有的不能看？为什么登录后还能记住身份？为什么有时页面明明打开了，但某个按钮点不了？这些都和鉴权和权限系统有关。

## 🧠 先分清两个概念

### 认证 Authentication

确认“你是谁”。

### 授权 Authorization

确认“你能做什么”。

很多人把两者混在一起，但在产品和技术设计里，它们不是一回事。

> 🌐 **In English:**
> Authentication answers who the user is. Authorization answers what the user is allowed to do.

## 🎯 会话、Cookie、Token 在做什么

- `Session`：服务端保存登录状态
- `Cookie`：浏览器帮忙保存和传递一部分信息
- `Token`：客户端携带的一段身份凭证，服务端据此识别用户

你不需要一开始就卷底层实现，但要知道这些是“登录状态为什么能持续存在”的关键。

## 🧩 为什么这块对 PM 特别重要

因为只要一个系统里存在：

- 登录
- 角色
- 操作权限
- 管理后台

你就迟早会遇到：

- 为什么这个账号能看那个不能看
- 为什么重新登录后状态不一样
- 为什么某个接口前端看不到但后端还要做校验

这些问题本质上都和鉴权体系有关。

## 🎯 一个很实用的理解顺序

### 先确认身份

系统先要知道你是谁。

### 再确认权限

系统再判断你能做哪些动作、能看到哪些数据。

### 最后落实到页面和接口

页面决定展示什么，接口决定最终是否允许执行。

> 🌐 **In English:**
> Identity comes first, permissions come next, and both eventually shape what the UI shows and what the backend allows.

## 💡 为什么“前端隐藏按钮”绝对不够

因为隐藏按钮只解决“界面不让点”，并没有解决“后端会不会放行”。

成熟系统一定还要有后端权限校验，因为真正的安全边界不在页面上，而在服务端。

## ⚠️ 最容易出问题的几类场景

### 场景一：角色切换

角色变了，但缓存、旧页面或历史链接还沿用旧权限状态。

### 场景二：管理员功能

管理员路径通常风险高，但最容易被觉得“内部用就没关系”。

### 场景三：历史数据兼容

新权限规则上线后，旧数据和旧用户状态是否仍然成立，往往是隐患点。

## ⚠️ PM 最容易忽略的权限问题

- 页面隐藏了按钮，不代表后端就安全
- 用户角色变化后，历史数据和旧链接怎么办
- 接口是否对越权访问做了校验
- 管理员和普通用户看到的状态是否一致

## 💡 为什么权限设计和产品体验强相关

权限设计不是只影响安全，也影响：

- 页面展示逻辑
- 按钮状态
- 空态提示
- 引导文案
- 错误反馈

一个权限设计不清楚的功能，往往连体验都做不好。

## 📝 记住这些

- 认证和授权不是一回事。
- 登录状态的持续，通常依赖会话、Cookie 或 Token 机制。
- 权限设计既是安全问题，也是体验问题。
