> 📌 **一句话版本：** 一个 AI 系统的效果，常常不是由模型本身单独决定，而是由 Prompt、上下文质量、记忆策略和工具调用方式共同决定。

---

## 📖 这一节在讲什么

为什么同一个模型，有时很聪明，有时像失忆？因为 AI 系统的实际表现，很大程度取决于你给了它什么上下文、如何组织任务、是否允许它调用工具，以及如何保留历史信息。

## 🧠 Prompt 不只是“提问话术”

在工程场景里，Prompt 更像任务说明书，往往需要包含：

- 目标
- 边界
- 输入材料
- 输出格式
- 验证标准

## 🎯 上下文决定上限

上下文不只是聊天记录，它还可能包括：

- 代码仓内容
- 当前文件
- 设计稿
- 文档
- 日志
- 历史决策

AI 的很多错误，本质上是上下文不完整或噪音太多。

## 🎯 记忆为什么重要

如果 agent 要长期协作，它需要某种方式保留用户偏好、项目规则和历史决策，否则每轮都像重新开始。

## 🎯 工具调用为什么能把 AI 从“会说”变成“会做”

因为只靠模型本身，它只能生成文本。接上工具之后，它才能：

- 搜索文件
- 查文档
- 操作浏览器
- 读写设计和知识库
- 运行命令或脚本

> 🌐 **In English:**
> AI systems become much more useful when prompts, context, memory, and tool use are designed together instead of treated separately.

## 🧩 为什么很多人会误以为“Prompt 写好就够了”

因为在简单问答场景下，好 Prompt 确实能立刻改善结果。但一旦进入真实工作流，效果往往还取决于：

- 给了什么上下文
- 有没有接到对的工具
- 历史记忆是否还有效
- 输出有没有被验证

所以 Prompt 很重要，但它只是系统中的一层。

## 🎯 上下文为什么常常比模型名更重要

一个强模型如果拿不到关键材料，也会乱猜。相反，一个不那么夸张的模型，如果拿到了：

- 正确的代码片段
- 准确的需求边界
- 相关日志和报错
- 当前项目约束

往往就能做得更稳。

## 💡 记忆最适合保存什么，不适合保存什么

### 适合保存

- 用户长期偏好
- 项目固定规则
- 稳定工作方式

### 不适合盲目累积

- 已经过时的临时需求
- 某次例外场景的特殊规则
- 未验证的猜测

> 🌐 **In English:**
> A memory system should preserve durable preferences and rules, not every temporary detail from past interactions.

## ⚠️ 工具调用最常见的误区

### 误区一：能接就都接

工具越多，不一定越聪明，也可能越乱。

### 误区二：查到了资料就当答案

资料只是原料，不代表已经完成理解和验证。

### 误区三：记忆越来越多就越来越聪明

记忆如果不清理，反而容易污染判断。

## ⚠️ 最常见的问题

- Prompt 很长，但目标不清
- 上下文很多，但关键材料缺失
- 工具很多，但没有验证闭环
- 记忆很多，但混入过时规则

## 📝 记住这些

- Prompt 是任务说明，不只是礼貌表达。
- 上下文质量和结构，往往比模型名更影响结果。
- 工具调用能放大能力，但也会放大风险。
