> 📌 **一句话版本：** 一个能落地的 PRD，必须被继续翻译成页面流程、接口边界和输入输出结构，否则它很难进入真实实现。

---

## 📖 这一节在讲什么

案例第一步，是把“做一个 AI 周报功能”从概念需求拆成工程能接的东西。

## 🧠 PRD 里至少要说清楚什么

- 谁在什么场景下使用
- 输入是什么
- 输出是什么
- 成功标准是什么
- 失败和异常如何处理

## 🎯 页面层要拆什么

- 入口页或弹窗
- 表单项
- 提交按钮
- 处理中状态
- 结果展示区
- 错误提示区

## 🎯 接口层要拆什么

- 提交生成任务的接口
- 查询生成结果的接口
- 历史记录接口

> 🌐 **In English:**
> A good PRD becomes actionable only when it is translated into UI flow, API boundaries, and explicit input-output expectations.

## 🧩 这一步最关键的，不是“写得完整”，而是“让别人能接着做”

如果 PRD 只是让人觉得“方向挺对”，但工程师或 AI 工具依然不知道下一步该从哪里开始，那它还不够可执行。

所以这一步真正的目标是：把需求翻译成一组能继续被页面、接口和数据设计消费的中间材料。

## 🎯 先把用户路径画清楚

针对“AI 自动生成周报”这个案例，用户路径至少可以拆成：

1. 进入功能入口
2. 选择时间范围和输入素材
3. 点击生成
4. 看到处理中状态
5. 成功后查看结果
6. 失败时得到提示并可重试
7. 可查看历史记录

这条路径越清楚，后面页面状态和接口边界越容易定。

## 💡 页面草图阶段最值得先定什么

### 区块划分

- 输入区
- 配置区
- 操作区
- 状态区
- 结果区
- 历史区

### 核心状态

- 初始空态
- 正在提交
- 正在生成
- 成功展示
- 失败提示

### 关键按钮

- 生成
- 重试
- 复制结果
- 查看历史

也就是说，页面草图不只是长什么样，而是要提前承载状态变化和动作路径。

> 🌐 **In English:**
> UI planning should define not only layout, but also state transitions, user actions, and what happens after each action.

## 🎯 接口草图阶段最值得先定什么

### 接口一：创建生成任务

你要先想清楚：

- 输入参数有哪些
- 哪些是必填
- 是否立即返回完整结果，还是先返回任务 ID

### 接口二：查询任务结果

你要想：

- 轮询还是主动刷新
- 返回哪些状态字段
- 失败时返回怎样的错误信息

### 接口三：历史记录

你要想：

- 是否分页
- 是否按时间排序
- 历史项展示哪些核心字段

## ⚠️ 这一阶段最容易漏掉的点

### 漏点一：只写 happy path

没写失败和处理中状态，后面产品体验一定会补得很痛苦。

### 漏点二：接口和页面没一起想

结果页面状态设计和接口返回结构对不上。

### 漏点三：没定义成功标准

最后团队不知道“生成成功”到底指的是返回文本、结构化结果，还是历史记录也要写入成功。

## 🧪 一个可复用的小模板

以后你要做任何类似功能，都可以先写：

- 用户路径 5 到 7 步
- 页面区块 4 到 6 个
- 核心状态 4 到 5 个
- 关键接口 2 到 4 个
- 每个接口的输入 / 输出 / 异常

这一版哪怕还是草图，也已经足够支撑后面的实现讨论了。

## 📝 记住这些

- PRD 只是起点，不是实现说明书的终点。
- 页面和接口拆解，是把产品语言翻成工程语言的关键一步。
