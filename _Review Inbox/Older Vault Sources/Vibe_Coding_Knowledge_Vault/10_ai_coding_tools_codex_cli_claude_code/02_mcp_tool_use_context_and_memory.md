## 📖 这一节在讲什么

你前面已经允许我为了项目需要自己接 `MCP`。这页就是把为什么要接、接了以后能做什么、风险在哪里讲清楚。

## 🧠 MCP 对 AI Coding 的价值

在没有外部工具时，AI 主要依赖你贴进去的内容和模型自身知识。接上工具后，它可以更直接访问真实环境、真实文档和真实工作材料。

## 🎯 典型工具能力

- 查官方文档
- 看设计稿
- 操作浏览器
- 读知识库或 Notion
- 跑测试或脚本

## 🎯 为什么上下文和记忆很关键

工具越多，信息源越多。如果没有上下文治理，很容易发生：

- 查到很多信息，但没对准任务
- 引用了过时规则
- 混入与当前项目无关的历史记忆

> 🌐 **In English:**
> MCP and tools can greatly improve an agent’s usefulness, but they also increase the need for good context management and clear boundaries.

## 🧩 为什么“接工具”经常是双刃剑

一旦 agent 可以访问更多系统，它确实会更有用，但同时也更容易出现这几类问题：

- 找到了很多信息，却没找到真正关键的那份
- 拿到了旧规则，误以为那是当前规则
- 可以做的动作变多了，误操作风险也变大了

所以工具接入不是“越多越好”，而是“是否真的对当前任务有帮助”。

## 🎯 上下文治理到底在管什么

你可以把它理解成三件事：

### 管输入范围

这轮任务到底允许读取哪些信息源。

### 管历史记忆

哪些历史规则应该沿用，哪些已经过时。

### 管输出约束

最终汇报要不要带风险、验证结果和待确认点。

## 💡 PM 为什么尤其要理解这层

因为当你开始把 AI 用到真实工作流时，你经常需要判断：

- 这次要不要接设计稿工具
- 这次要不要查官方文档
- 这次要不要让它跑浏览器或命令
- 这次是否需要保留项目记忆

这些都不是纯技术细节，而是工作流设计。

> 🌐 **In English:**
> Context management is about deciding what the agent can read, what prior rules still apply, and what kind of output discipline is required.

## ⚠️ 最容易被忽略的记忆问题

如果记忆一直累积，但没人清理，就容易出现：

- 老规则压过新规则
- 某个项目经验误导另一个项目
- 工具持续引用已经不适用的习惯

所以“记忆”不只是存下来，更重要的是持续校准。

## 🧪 一个很有用的自检问题

以后你让 agent 接更多工具前，可以问：

1. 这次任务最需要什么外部信息
2. 哪些工具是必须的，哪些只是锦上添花
3. 这些信息是否有版本和时效风险
4. 我最后怎么验证工具拿回来的信息没错

这四问能显著降低“工具越多越乱”的概率。

## ⚠️ 风险点

- 工具权限过大
- 引用错误数据源
- 改动范围失控
- 依赖外部信息却不做验证

## 📝 记住这些

- MCP 让 AI 更像真实工作助手，而不是纯聊天机器人。
- 工具能力越强，上下文控制和权限控制越重要。
- 好的 agent 工作流不是“接越多工具越好”，而是“对任务真正有帮助的工具才接入”。
