> 🌐 **In English:**

## 📖 这一节在讲什么

本节整理15个AI PM高频面试题，按类型分为：
- 技术理解题（5个）
- 产品设计题（5个）
- 行业认知题（3个）
- 行为面试题（2个）

每题包括：**考察点 + 参考回答框架 + 加分点**。

不要死记答案。理解框架，用自己的经历填充。

## 🧠 一、技术理解题（5题）

### T1. 请解释RAG是什么，它解决了什么问题？

**考察点：** 是否理解LLM的核心局限，以及工程化解决方案

**参考回答框架：**

```
第一层（问题）：
LLM有两个核心局限：
- 知识截止日期（训练数据有限）
- 无法访问私有/实时数据

第二层（方案）：
RAG = Retrieval Augmented Generation
工作流程：
用户提问 → 向量检索相关文档片段
→ 将文档片段+问题一起送给LLM
→ LLM基于上下文生成答案

第三层（产品价值）：
- 减少幻觉（有来源可溯）
- 知识可实时更新
- 成本比全量微调低很多
```

**加分点：** 能说出RAG的局限性（检索质量依赖embedding模型、长文档chunk策略影响效果），并提出你会如何评测RAG系统的质量（推荐用RAGAS框架）。

---

### T2. LLM的幻觉是什么？作为PM你如何处理？

**考察点：** 对AI核心风险的认知，以及产品层面的应对策略

**参考回答框架：**

```
定义（30秒）：
幻觉 = 模型生成听起来合理但事实错误的内容。
根本原因是LLM是概率预测机器，
不是事实数据库。

产品层面的应对策略：
1. 设计层：
   - 对高风险场景（医疗/法律/金融）
     强制显示"AI生成，请核实"
   - 提供原始来源链接（RAG场景）
2. 评测层：
   - 建立幻觉率基准，每次迭代回归测试
   - 分类测试：事实型/推理型/创意型
3. 兜底层：
   - 设计人工审核流程
   - 建立用户反馈-模型迭代闭环
```

**加分点：** 能区分"事实幻觉"（说错了具体信息）和"推理幻觉"（逻辑链条出错），并说明在你的具体产品场景中如何权衡幻觉风险和产品体验流畅度。

---

### T3. Fine-tuning和Prompt Engineering有什么区别？什么时候用哪个？

**考察点：** 是否具备AI产品技术路线的判断能力

**参考回答框架：**

```
对比维度：

               Prompt工程    Fine-tuning
成本           极低           高（算力+标注）
速度           即时           需要训练周期
数据需求       无需标注数据    需要高质量标注数据
效果天花板     受基础模型限制  可超越基础模型上限
适用场景       通用任务调整    垂直专业领域

决策规则：
- 先试Prompt Engineering
- Prompt效果不稳定且有大量
  高质量标注数据 → 考虑微调
- 效果差但没有数据 → 先构建数据飞轮
```

**加分点：** 提到"RLHF"（从人类反馈中强化学习）是微调的一种形式，以及"LoRA"等参数高效微调方法可以大幅降低微调成本。

### T4. 你会如何评估一个AI功能是否值得上线？

**考察点：** AI产品质量评估框架，是否有系统性思维

**参考回答框架：**

```
评估维度（四象限）：

1. 质量维度
   - 准确率/F1/BLEU（依任务而定）
   - 幻觉率基准测试
   - 边界case覆盖率

2. 体验维度
   - 响应延迟（P50/P95）
   - 输出一致性（同问题多次回答）
   - 与产品整体体验的融合度

3. 成本维度
   - 每次调用的Token成本
   - 规模化后的成本预测
   - 成本收益比（ROI）

4. 安全维度
   - 违规内容率
   - 对抗性输入测试（红队测试）
   - 隐私数据泄露风险

上线决策：以上四个维度都达到预设阈值，
且有回滚方案，才可以灰度上线。
```

**加分点：** 能说出"灰度发布 + 指标监控 + 快速回滚"是AI功能上线的标准流程，并举一个你实际操作过的案例。

### T5. Token、上下文窗口、Embedding分别是什么？

**考察点：** LLM基础概念掌握程度

**参考回答框架：**

```
Token：
LLM处理文本的最小单位，约1个英文词
= 1-2个token，1个中文字 = 1-2个token。
直接影响API调用成本和速度。

上下文窗口（Context Window）：
模型一次能"看到"的最大Token数量。
GPT-4是128k token，Claude是200k token。
产品设计时要考虑长对话的截断策略。

Embedding（向量表示）：
将文本转化为高维数字向量，
语义相似的文本在向量空间中距离近。
RAG系统的核心技术，用于语义检索。
```

**加分点：** 能说出"上下文窗口大小影响产品设计"——比如长文档摘要的策略选择，以及"Embedding模型的选择影响RAG效果"。

> 🌐 **In English:**
> Technical questions test your understanding of LLM architecture (RAG, hallucination, fine-tuning, evaluation, tokens). Know the concepts AND their product implications.

## 🧠 二、产品设计题（5题）

### P1. 请为一款面向律师的AI助手设计核心功能。

**考察点：** AI产品设计能力，垂直场景理解，功能优先级判断

**参考回答框架：**

```
第一步：澄清问题（1分钟）
- 律师类型？诉讼律师/合规律师？
- 工作流哪个环节最耗时？
- 主要在哪里工作？PC还是移动？

第二步：用户核心痛点分析
律师最耗时的工作：
1. 案例检索（找相关判例）
2. 合同审查（找风险条款）
3. 文件起草（撰写法律文书）
4. 庭审准备（整理证据材料）

第三步：功能设计（聚焦TOP1）
核心功能：智能案例检索+分析
- 输入：案件描述/关键词
- 处理：语义检索相关判例
         + LLM提炼裁判要点
- 输出：相关案例列表+可复用论点
- 安全：所有结论标注来源，
         律师可一键跳转原文

第四步：非功能性要求
- 幻觉控制：法律场景容错率极低，
  必须有来源链接
- 数据安全：案件数据不可用于模型训练
```

**加分点：** 主动提出"为什么先做检索而不是起草"——因为检索的准确性可验证，风险更低，是最好的信任建立切入点。

---

### P2. 你如何设计一个AI功能的A/B测试？

**考察点：** 数据驱动能力，AI产品验证方法论

**参考回答框架：**

```
AI功能A/B测试的特殊性：
传统A/B测试对比两个确定性版本，
AI功能的输出是概率性的，
需要额外处理"输出质量"的评估。

设计步骤：
1. 定义核心指标
   - 北极星指标（如：任务完成率）
   - AI质量指标（如：用户接受率）
   - 护栏指标（如：违规内容率）

2. 样本量计算
   - 确定最小可检测效果（MDE）
   - 计算所需样本量（通常7-14天）

3. 实验分组
   - 控制组：旧版本/无AI功能
   - 实验组：新AI功能
   - 避免样本污染（用户ID哈希分桶）

4. AI特有：质量评估
   - 自动评估：LLM-as-Judge
   - 人工评估：随机抽样100-200条
   - 用户反馈：点赞/踩/举报率

5. 上线决策
   - 所有指标正向，且无护栏指标恶化
   - 置信度 > 95%
```

**加分点：** 提到"LLM-as-Judge"（用另一个LLM评估输出质量）是AI产品评测的特有技术，并指出其局限性（评估模型本身也可能有偏差）。

### P3. 如何设计一个AI聊天机器人的对话状态管理？

**考察点：** 对话产品设计能力，工程意识

**参考回答框架：**

```
核心挑战：
多轮对话需要"记忆"，但LLM本身是无状态的。

解决方案层次：

层次1：对话历史管理
- 全量历史（短对话有效）
- 滚动窗口（超过窗口截断）
- 摘要压缩（长对话自动摘要）

层次2：用户偏好记忆
- 短期记忆：本次对话的偏好
- 长期记忆：跨对话的用户画像
- 实现：结构化数据库存储

层次3：任务状态追踪
- 对话进行到哪一步了？
- 用户意图是否变化？
- 需要调用哪个工具？

产品设计考量：
- 给用户"清除记忆"的控制权
- 显示AI"记住"了什么（透明度）
- 跨设备记忆同步
```

### P4. 现有AI产品留存率低，你会怎么分析原因？

**考察点：** 数据分析能力，AI产品特有的留存问题识别

**参考回答框架：**

```
AI产品留存低的特有原因：

1. 首次体验失败（幻觉/答非所问）
   诊断：查看新用户D1会话的反馈率

2. 期望管理失败（用户期望太高）
   诊断：查看用户流失前的最后一次对话

3. 使用场景单一（只有一个用途）
   诊断：分析高留存用户的使用场景多样性

4. 学习成本高（不知道怎么用好）
   诊断：对比新老用户的任务完成率

5. AI效果提升后用户没感知
   诊断：主动推送改进通知的效果

分析路径：
用户分群（流失 vs 留存）
→ 对比关键行为差异
→ 定性访谈10个流失用户
→ 找到最高杠杆的改进点
```

### P5. 请评价一款你最近使用的AI产品，并提出改进建议。

**考察点：** 产品洞察力，AI产品审美

**回答策略（以豆包为例）：**

```
评价结构：
1. 核心定位是什么？做到了吗？
2. 用户路径中最关键的3个体验节点
3. 哪个功能做得好，为什么？
4. 哪个功能可以改进，如何改？

示例答案框架：
强项：
- [功能X]在[场景Y]的体验非常顺滑，
  原因是[具体设计决策]

弱项+改进：
- [功能A]在[场景B]存在[具体问题]，
  根本原因是[分析]
- 改进方向：[具体方案]，
  预期效果：[量化目标]
```

**加分点：** 能从"AI能力边界"角度分析产品决策，而不只是从用户体验角度。

> 🌐 **In English:**
> Product design questions test your ability to apply AI thinking to real scenarios: handling hallucination risk, designing AI A/B tests, managing conversation state, and diagnosing AI-specific retention issues.

## 🧠 三、行业认知题（3题）

### I1. 你怎么看GPT-4o/Claude 3.5 Sonnet的竞争格局？

**考察点：** 行业视野，是否持续关注AI前沿

**回答要点：**

```
不要只背参数对比，要说出：
1. 技术差异化点（多模态/代码/推理/速度）
2. 商业模式差异（API/订阅/企业版）
3. 对产品开发者的影响
   - 选模型的考量（成本/能力/稳定性）
   - API兼容性风险
   - 依赖单一模型的风险管控

加分框架：
"对于[具体场景]，我会选择[模型X]，
因为[技术原因]，同时考虑到[成本/风险]"
```

### I2. 你认为AI PM和传统PM最大的能力差距是什么？

**考察点：** 自我认知，学习能力

**回答要点：**

```
诚实回答，展示成长心态：

独特能力要求：
1. 技术直觉：能快速判断"这个能力AI能不能做"
2. 概率思维：接受AI输出的不确定性
3. 评估设计：知道怎么衡量AI效果好不好
4. 人机协作设计：知道哪些部分该AI做，哪些该人做

我的成长路径：
- 我在[X方面]已有积累：[举例]
- 我在[Y方面]还在补课：[具体行动]
```

### I3. AI产品面临哪些伦理风险？你如何在产品中处理？

**考察点：** AI伦理意识，产品责任感

**回答要点：**

```
核心风险清单：
1. 偏见与歧视（训练数据偏差导致）
2. 隐私泄露（用户数据用于训练）
3. 信息茧房（推荐算法过滤）
4. 幻觉风险（在高风险场景中）
5. 滥用风险（生成有害内容）

产品层面的应对：
- 建立内容安全过滤层
- 数据使用条款透明化
- 高风险功能强制人工审核
- 建立用户举报-快速响应机制

不要说"AI不会有偏见"——
这是最大的红旗。
```

> 🌐 **In English:**
> Industry questions test whether you follow AI trends closely, understand the capability gap between traditional and AI PM, and take AI ethics seriously.

## 🎯 AI产品经理视角

**面试备考的正确方式：**

不要死背答案模板。要做到：

1. **理解原理，不背答案** — 每个题型背后的考察逻辑比标准答案更重要
2. **用自己的真实经历填充** — 哪怕是个人项目、副业、学习中的实验
3. **主动暴露思考过程** — 面试官更想看你怎么思考，而不是你有没有正确答案
4. **对不懂的技术坦诚** — "我目前还不完全理解X的实现，但我的判断是…"比装懂更好

## 📝 记住这些

- 技术题考的是"理解力"，不是"记忆力"
- 产品题必须有结构：定义问题 → 分析场景 → 设计方案 → 评估风险
- 行业题要有自己的真实观点，不要说官方话
- 行为题必须有真实案例，不能只讲理论
- 所有回答都要有AI PM特有的思维角度
