# 四、核心工作原则

## 原则 1：循序渐进，不要一上来塞满干货

按"从建立感觉 → 建立概念 → 建立框架 → 建立判断"的顺序来写。

错误方式（禁止）：
- 上来就讲 Transformer 数学细节
- 上来就讲 attention 公式
- 上来就讲 embedding 维度、梯度下降、loss 设计
- 默认用户知道 API、数据库、前后端是什么

正确方式：
- 先解释 AI 这几年到底发生了什么
- 再解释为什么大家都在谈大模型
- 再解释大模型和传统软件有什么不同
- 先解释"程序"和"接口"这种最基本概念
- 再逐步过渡到模型、工具调用、Agent、MCP、Skills

## 原则 2：每出现一个新术语，都要补齐解释链

任何新术语，至少要解释这几层：

- 它是什么（中文 + 英文）
- 为什么会出现
- 它解决了什么问题
- 它和相近概念的区别
- 它和 AI 产品经理有什么关系
- 给一个具体例子
- 给一个常见误解

必须覆盖的核心术语（不限于）：
- 程序（Program）、接口（API）、服务器（Server）
- Token、Context Window、Embedding
- 模型（Model）、训练（Training）、推理（Inference）
- Fine-tuning、RAG、Prompt Engineering
- Function Calling、Agent、Workflow
- MCP、Skills
- Evals、Benchmark
- Hallucination、Latency、Inference Cost
- Guardrails、RLHF

## 原则 3：所有内容都要贴近 AI 产品经理

不能只写技术定义，必须持续补充：

- 对产品经理来说，为什么要懂这个
- 在需求讨论中会遇到什么问题
- 在方案评审中会问什么问题
- 在落地时会踩什么坑
- 哪些是用户可感知问题，哪些是工程端问题
- 哪些指标需要关注
- 哪些问题会影响体验、成本、可靠性、上线速度

## 原则 4：中英双语段落规则

每个核心概念段落，采用"中文正文 + 英文精简版"的格式：

- 正文用中文写，自然、像人话
- 每个核心段落后面跟一个英文引用块
- 英文使用中等难度词汇，句子清晰，不过度简单也不过度复杂
- 英文段落是中文段落的精简版，不是逐字翻译
- AI 术语在中文正文中首次出现时标注英文：如"大语言模型（Large Language Model, LLM）"

格式示例：
```
你发一段话给 ChatGPT，它不是一个字一个字看的。
它会先把你的话切成一小块一小块的「Token」来处理。

> In English:
> When you send text to ChatGPT, it doesn't read
> character by character. Instead, it breaks your
> input into small chunks called "tokens."
```

## 原则 5：中文表达必须自然

请避免：
- 机械堆术语
- 过度学术化
- 空泛鸡汤式表达
- 自嗨式技术炫耀

要做到：
- 先让初学者看懂
- 再逐步让他理解深一层
- 用类比，但不要类比过头
- 一旦类比可能误导，明确提醒"这只是帮助理解，不是严格等价"

## 原则 6：资料来源优先级

第一优先级：
- 官方文档、官方博客、官方研究发布、原始论文

第二优先级：
- 高质量技术博客、工程实践文章、高质量 X 线程

第三优先级：
- 一般媒体报道（只用于补充背景）

引用 X 内容时：
- 优先选择长期输出稳定、业内有认可度的作者
- 优先选择解释清楚、有实践价值的内容
- 区分"事实"与"作者判断"

## 原则 7：不能偷懒

- 先搭好骨架
- 先完成最关键模块的第一版
- 再逐步补深
- 不能只给目录不给内容

## 原则 8：每个文件独立可读

任何一个 md 文件，至少包含模板中要求的所有章节。

# 五、排版与格式规范（极其重要）

## 设计理念

采用 **杂志感排版**：视觉层次丰富、大量留白、图标分区、卡片高亮。

## 兼容性要求

资料库必须在以下三个场景中都能良好呈现：

1. **Obsidian** — 日常阅读与笔记
2. **Notion 导入** — 团队分享
3. **纸质打印（A4）** — 用户最喜欢的阅读方式

因此遵循以下兼容性规则：

| 语法 | 是否使用 | 原因 |
|------|---------|------|
| 标准 `>` 引用块 | 使用 | 三端兼容 |
| Obsidian Callout `> [!note]` | 不使用 | Notion 和打印不识别 |
| ``内链`` | 不使用 | Notion 不识别 |
| `<details>` 折叠块 | 不使用 | 打印时无法展开 |
| Emoji 标题 / 图标 | 使用 | 三端兼容，打印时也可显示 |
| 标准表格 | 使用 | 三端兼容 |
| `---` 分隔线 | 使用 | 三端兼容 |
| 代码块 ` ``` ` | 使用 | 三端兼容 |
| 粗体 / 斜体 | 使用 | 三端兼容 |

## 打印友好规则（重要）

纸质打印是用户最喜欢的阅读方式，必须遵守：

1. **表格最多 4 列** — 超过 4 列在 A4 纸上会挤
2. **代码块宽度不超过 60 字符** — 避免打印时换行错乱
3. **每节之间用 `---` 分隔** — 打印时视觉层次清晰
4. **段落不要过长** — 每段控制在 3-5 行，方便纸上阅读
5. **Emoji 要配文字标签** — 如 `💡 提示：` 而不是只放 💡，打印时如果 emoji 显示异常，文字标签仍可读
6. **引用块内容不要超过 5 行** — 打印时引用块太长视觉上会很重
7. **留白充足** — 段落之间空行，章节之间分隔线
8. **每个文件控制在打印 8-12 页以内** — 太长不利于单篇阅读

## Emoji 图标规范

统一使用以下图标体系，保持全库一致：

| 图标 | 用途 | 文字标签 |
|------|------|---------|
| 📌 | 一句话总结 | 📌 **一句话版本** |
| 💡 | 提示 / 补充说明 | 💡 **提示：** |

- 什么是云服务

这些是理解后续所有 AI 概念的地基。

## 8.3 AI 产品经理模块

不能空泛，必须覆盖以下现实问题：

**1. AI PM 和传统 PM 的差别：**
- 传统互联网产品 = 确定性系统
- AI 产品 = 概率型系统
- 导致需求写法、验收方式、上线策略都变化

**2. AI 需求分析时必须补充的问题：**
- 这个需求真的需要模型吗
- 需要什么级别的模型能力
- 失败时用户体验怎么办
- 模型输出错了怎么办
- 是否需要人工兜底
- 成本能不能接受
- 延迟能不能接受
- 数据从哪里来
- 是否需要记忆 / 检索 / 工具调用

**3. AI 产品核心指标：**
- 质量、延迟、成本、成功率
- 用户满意度、留存、任务完成率
- 幻觉率 / 错误率

**4. AI 产品中的常见错觉：**
- 把模型能力问题误判成交互问题
- 把工程接入问题误判成模型不行
- 把 demo 体验误判成可规模化产品
- 把一次惊艳误判成长期价值

## 8.4 高质量 X 信息源

1. 能把复杂问题讲清楚的人

4. 对 Agent / Evals / AI Engineering / AI PM 有长期观察的人
5. 不是只会喊口号、追热点、贩卖焦虑的人

每个作者写清楚：

- 不能把热闹当趋势
- 不能把 demo 当成熟产品
- 不能把单条线程当行业共识

## 8.5 动手实践模块（新增）

# 九、质量标准

每个文件都必须达到以下标准：

**A. 初学者能读懂**
- 不默认读者懂技术黑话
- 不在没有解释的情况下连续抛缩写
- 不一段里塞太多新概念

**B. 有逻辑递进**
- 前文和后文能接上
- 基础概念先于进阶概念
- 不允许知识跳跃严重

**C. 不空泛**
- 不能只有概念没有例子
- 不能只有趋势没有原因
- 不能只有观点没有判断依据

**D. 面向 AI 产品经理**
- 不写成纯工程师教程
- 不写成纯学术综述
- 持续回到"产品工作中有什么用"

**E. 有资料来源意识**
- 能标注来源就标注
- 区分事实、推测、观点
- 有争议的地方说明

**F. 可持续扩展**
- 文件结构利于追加内容
- 命名清晰，层级清楚

**G. 双语质量（新增）**
- 英文段落语法正确、表达自然
- 英文用中等难度词汇，避免过度简单
- 英文是精简版，不是逐字翻译

**H. 排版质量（新增）**
- 符合杂志感排版规范
- 打印友好
- Emoji 图标统一
- 三端兼容（Obsidian / Notion / 打印）

# 十、写作时必须避免的行为

1. 一上来讲得太技术
2. 堆砌术语不解释
3. 只会给目录，不真正写内容
4. 只讲技术，不讲产品经理为什么要懂
5. 只讲热点，不讲底层逻辑
6. 大量空洞形容词，没有实质信息
7. 默认用户理解代码、训练流程、系统架构
8. 把复杂概念讲得像正确废话
9. 只写论文感内容，缺少"人话"
10. 没有建立"知识之间的连接"
11. 英文段落和中文段落内容完全脱节（新增）
12. 排版不统一，图标使用混乱（新增）
13. 内容太长不适合打印阅读（新增）
