Structure Standard

优雅结构标准

先建立判断项目结构是否优雅的四个标准，读结构前先有判断尺子。 4 条判断

01

可预测性 Predictable

你不用问别人，就知道 API、页面、业务逻辑各自应该放在哪一层。

新人 3 分钟能读懂结构，通常就已经接近优雅。

02

单一职责 Separation of Concerns

每一层只负责一件事，UI、请求、业务逻辑和数据结构不要混在一个目录里。

services 负责业务，components 负责展示，api 负责请求。

03

低耦合 Low Coupling

页面之间不要互相调用，跨页面能力应该经由 service 或 api 中转。

页面只组织展示和交互，不直接依赖别的页面实现细节。

04

易扩展 Scalable

新模块应该可以在原结构上扩展，而不是每次加功能都推翻重来。

加用户系统、加新接口、换 UI 框架，都应当是扩展，不是重建。

Frontend

前端结构模板

前端更推荐按业务模块组织，而不是只按技术类型切目录。这样一个功能会成为一个闭环，而不是散在各处。 18 行模板
frontend/

前端工程容器

这一层是整个前端工程的最上层容器，负责把源码、静态资源和前端配置收在一起。

代码里通常怎么用/真实团队里通常会把前端代码收进 frontend/，和 backend/ 分开治理。

src/=source

主源码目录

项目最核心的实现通常都从这里开始往下展开，看到它就能判断“主代码在这里”。

代码里通常怎么用/常见写法是 src/components、src/pages、src/hooks，这些都属于真正会被打包和运行的代码。

api/=Application Programming Interface

接口请求层

适合集中封装请求函数，让页面和组件不要直接写 fetch 或 axios 细节。

代码里通常怎么用/例如 export async function getUser() { return request.get("/user") }，页面只调用 getUser()。

assets/

静态资源层

这一层适合收图片、图标、插图和其他不会直接执行的静态资源。

代码里通常怎么用/像 logo、banner、插图和占位图，一般会放在 assets/。

components/

可复用组件层

放按钮、卡片、弹窗这类可复用界面片段，重点是复用而不是承接整页业务。

代码里通常怎么用/如果一个文件里主要是 Button、Card、Modal、FormField，这一层通常就叫 components。

features/

业务模块层

按业务闭环组织代码，把一个功能相关的组件、hooks、service 放在一起。

代码里通常怎么用/像 focus-session、task-system 这种模块名，比单纯把所有代码分散到 pages/components 更容易维护。

focus-session/

专注会话模块

这是按业务命名的功能模块，围绕一次专注训练过程组织自己的组件、状态和服务。

代码里通常怎么用/业务模块名越贴近真实产品能力，团队就越容易理解这个目录在负责什么。

task-system/

任务系统模块

这是任务相关能力的业务模块，适合把任务规则、视图和交互闭环收在一起。

代码里通常怎么用/比起 module-a 这类抽象名字，用 task-system 这种业务词更容易维护。

reward-system/

奖励系统模块

这是奖励逻辑的业务模块，适合沉淀积分、激励和奖励反馈相关能力。

代码里通常怎么用/奖励规则、奖励面板、奖励计算通常会在同一个业务模块里协作。

report-analytics/

报告与分析模块

这是分析和报告能力的业务模块，重点是把结果展示和分析逻辑聚在一起。

代码里通常怎么用/像训练报告、趋势分析、周报面板，适合收在 report-analytics 这类模块里。

hooks/

状态逻辑复用层

常见于 React 语境，适合放 useXxx 这类带状态和副作用的复用逻辑。

代码里通常怎么用/例如 useUserProfile、useCountdown、useSessionTimer 这类函数，通常就会放在 hooks/。

layouts/

布局层

这一层负责页面骨架和共用布局，不直接承接具体业务规则。

代码里通常怎么用/像 DashboardLayout、AuthLayout 这种页面框架通常会放在这里。

pages/

页面层

页面层负责路由对应的完整视图，它更像一个装配入口，而不是所有逻辑都堆在里面。

代码里通常怎么用/页面文件里会组合组件、调用 hooks、触发 api，但不会把请求和状态实现都写在同一个大文件里。

router/

路由配置层

这一层负责定义页面路径和页面之间的导航关系。

代码里通常怎么用/路由表、导航守卫和跳转配置通常会放在 router/。

store/

全局状态层

适合放跨页面共享的状态容器，例如登录信息、主题、全局筛选条件。

代码里通常怎么用/如果多个页面都要读写同一份状态，通常会放到 store/ 而不是互相 import 页面里的变量。

styles/

样式层

这一层收全局样式、设计变量和基础视觉规则，帮助页面保持一致。

代码里通常怎么用/reset、theme、token 或全局 css 往往会放在 styles/。

utils/

纯工具层

只放没有业务语义的通用函数，例如时间格式化、字符串处理、数组转换。

代码里通常怎么用/如果一个函数叫 formatDate、groupByMonth、clampNumber，它更适合放 utils；如果叫 getUserInsight，就更像 service。

main.ts=TypeScript

前端入口文件

这是应用真正启动的入口文件，通常从这里挂载应用、引入路由和全局状态。

代码里通常怎么用/React / Vue 项目里常会在 main.ts 里创建应用实例并挂到页面根节点。

Backend

后端结构模板

后端的优雅感来自清楚的请求流和稳定的业务层，controller 不要吞掉 service 的职责。 10 行模板
backend/

后端工程容器

这一层是整个后端工程的最上层容器，负责把接口、服务和配置收成一套稳定结构。

代码里通常怎么用/把前后端分开后，团队会更容易定位请求入口和数据处理边界。

src/=source

主源码目录

这一层是后端核心实现的真正入口，后续的 routes、services、models 都从这里继续展开。

代码里通常怎么用/如果你想快速看懂后端结构，通常先从 backend/src/ 往下读。

controllers/

接口入口处理层

负责接住请求、整理参数、返回响应，不应该把真正的业务规则全塞在这里。

代码里通常怎么用/controller 最好只做“收参数、调 service、回结果”这三件事，避免直接查数据库。

services/

业务逻辑层

承接真正的业务规则、数据组合和流程判断，是后端最像“大脑”的一层。

代码里通常怎么用/如果你在写积分发放、训练报告生成、奖励计算，这类逻辑更应该进 services/。

models/

数据模型层

负责描述数据结构、数据库映射或 ORM 模型，让数据层保持稳定和清楚。

代码里通常怎么用/像 User、FocusSession、RewardRecord 这类结构，一般会落在 models/。

routes/

路由入口层

定义 URL 或接口路径，把请求分发给对应 controller，是后端的入口索引。

代码里通常怎么用/例如 router.get('/reports', reportController.listReports)，这一步先决定谁来接请求。

middlewares/

中间件层

处理鉴权、日志、限流、错误捕获这类所有请求都会经过的通用流程。

代码里通常怎么用/如果一段逻辑要在多个接口前统一执行，例如校验 token，就更适合放 middleware。

utils/

纯工具层

这一层只放通用工具函数或基础能力，不应该堆业务规则。

代码里通常怎么用/例如日期处理、签名工具、格式转换，适合放在 utils/。

config/

配置层

统一管理环境变量、端口、数据库地址、第三方密钥的读取和组织方式。

代码里通常怎么用/这样切换开发、测试、生产环境时，不需要去很多业务文件里手改值。

app.js=JavaScript

后端入口文件

这是后端启动入口，通常会在这里组装路由、中间件和服务启动逻辑。

代码里通常怎么用/常见写法会在 app.js 里初始化 express 或其他服务框架，再挂载 routes。

Request Flow

请求流

请求链越清楚，项目结构越容易维护。读懂 route -> controller -> service -> model -> database 这条线，基本就能快速定位问题。 5 步
route -> controller -> service -> model -> database

每一步在做什么

route

路由入口

先定义“哪个 URL / 接口地址”由哪个处理函数接住。

代码里通常怎么用/例如 app.get('/users', userController.listUsers)，route 先负责分发。

controller

请求接待层

把请求参数整理干净，调用 service，再把结果回给前端。

代码里通常怎么用/controller 里常见 req.params、req.body、res.json 这类代码，但不适合堆业务判断。

service

业务判断层

真正处理规则、计算和流程，是“为什么这样做”的地方。

代码里通常怎么用/例如根据专注时长发奖励、根据用户状态生成训练建议，都应该在 service。

model

数据读写层

把业务逻辑转成数据库可理解的结构，负责查、写、更新数据。

代码里通常怎么用/service 调 model 拿数据，而不是 controller 直接拼 SQL 或直接碰数据库。

database

数据库

最终把数据落到持久化存储里，例如 MySQL、PostgreSQL、SQLite。

代码里通常怎么用/这层关注的是存储，不负责业务语义，所以业务规则不要写在数据库表名里。

知识库工作流

01

需求确认与用户画像

先确认读者是谁、目标是什么、主要阅读场景是什么，再决定目录和命名。

结构启发/结构不是给自己炫技的，是为了让目标读者更快找到入口。

02

模块结构设计

把资料拆成可预测、可逐步阅读的模块，再定义每个文件的职责边界。

结构启发/目录先有层级，再谈内容；不然文件再多也只是平铺堆积。

03

排版规范统一

先定标题、摘要、提示、语言切换和命名规则，再批量生产内容。

结构启发/统一规范能让知识库像一个系统，而不是一堆风格混乱的单篇文档。

04

分批生产与验证

按模块分批推进，每批都先检查缺失、格式和命名，再继续扩写。

结构启发/增量验证比一次性堆完再返工，更适合长期维护和协作。

05

质量审查与适配

抽检是否有未解释术语、标题重复、格式失真，以及 Obsidian / Notion / 打印是否都能读。

结构启发/真正优雅的结构，不只是好看，还要在不同阅读场景下都稳定可用。

06

交付与记忆沉淀

交付时同时保留项目记忆、规则和统计信息，方便后续复用和迭代。

结构启发/结构做好以后，未来的增量更新才不会把整个系统越改越乱。

质量检查提醒

  • 高频缩写第一次出现时要能看懂，不要只写缩写不解释。
  • 目录词和内容词分开，结构位不要和业务主题混成一个长名字。
  • 文档入口词要稳定，例如 readme、guide、notes、workflow。
  • 一层目录最好只承担一种职责，避免既放源码又放历史归档。
  • 输出前检查是否有明显技术噪音目录被误当成命名模板。

高频缩写速查

src=source

源码

通常表示主源码目录，看到 src 基本可以理解成“真正运行的代码在这里”。

api=Application Programming Interface

接口

常表示请求层或接口层，前端会调它，后端会实现它。

ui=User Interface

用户界面

更偏界面呈现、组件样式和视觉层，不等于完整体验。

ux=User Experience

用户体验

更偏流程、理解成本和交互感受，通常比 UI 更上层。

prd=Product Requirements Document

产品需求文档

常表示需求说明稿，是产品定义问题和范围的入口文件。

md=Markdown

Markdown 文档

轻量文档格式，适合知识库、说明、笔记和 README。

tsx=TypeScript + JSX

React 组件文件

多用于 React 页面或组件文件，说明这个文件既有 TypeScript 又有界面结构。

json=JavaScript Object Notation

结构化数据文件

适合配置、映射表和结果输出，通常是“给程序读”的，不是“给人直接写长文”的。

db=Database

数据库

路径或字段里看到 db，通常就是在指数据存储层，而不是页面层或业务规则层。

cli=Command-Line Interface

命令行界面

通常表示终端工具、脚本入口或本地执行命令的交互层，偏开发工具，不是用户页面。

orm=Object-Relational Mapping

对象关系映射

常见于数据库工具语境，作用是把代码里的对象结构映射到数据库表，减少手写 SQL。

sdk=Software Development Kit

软件开发工具包

通常是一套被别人调用的能力包，比如支付 SDK、地图 SDK、AI SDK。

npm=Node Package Manager

Node 套件管理器

常出现在 JavaScript 项目的依赖安装、脚本执行和套件发布语境里。

cms=Content Management System

内容管理系统

如果项目里出现 cms，多半在说内容录入、编辑、发布后台，而不是前台展示页。

seo=Search Engine Optimization

搜索引擎优化

多出现在流量、内容分发和页面元信息语境，和标题、描述、索引能力有关。
