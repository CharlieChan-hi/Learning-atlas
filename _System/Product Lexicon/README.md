# 🧠 Product Lexicon

> 把行业黑话，翻译成真正可用的产品理解。

---

# 1. 这个文件是做什么的？

这是一个用于长期维护的 **行业术语 / 产品黑话 / 工作表达 / 英文词汇** 的知识系统文件。

它的目标不是简单记笔记，而是把你在工作里听到的内容，逐步沉淀成一个：

- 可阅读
- 可检索
- 可维护
- 可扩展
- 可被 AI 持续接管和更新
- 可进一步生成结构化 HTML

的知识文档系统。

这个文件默认服务于 **C 端产品场景**。

当前已经补入一份更适合日常速查的桌面命名词表：

- [Desktop Workspace Naming Glossary](desktop_workspace_naming_glossary.md)

也就是说，所有术语、表达、词汇，最终都尽量要回到以下几个维度来理解：

- 用户体验
- 用户行为
- 转化
- 留存
- 增长
- 商业化
- 路径阻力
- 产品决策

# 2. 这个文件夹要做什么？

这个文件夹的职责，是作为一个长期使用的 **术语解码器 + 产品认知仓库**。

你后续会持续往里面塞入：

- 产品术语
- 数据术语
- 增长术语
- 商业化术语
- UX 相关表达
- 团队里常说的黑话
- 中文工作表达

- 它一般出现在哪些场景

- 这个词放在 C 端产品里到底意味着什么

- 这个词有哪些常见误解和伪用法

# 7. AI 协作规则

## 7.1 目的

另一个 AI 读取本文件后，需要理解：

- 这个系统要做什么
- 应该如何写入内容
- 应该如何更新内容
- 如何维护长期记忆和工作记忆
- 如何避免重复和混乱
- 如何根据本文件进一步生成 HTML

## 7.2 AI 必须遵守的规则

### 规则 1：保留结构
不得破坏标题层级，不得随意改乱章节结构。

### 规则 2：避免重复
如果某个术语已经存在，优先补充和优化原条目，而不是重复创建。

### 规则 3：同步更新
每次新增术语或表达时，AI 应至少同步考虑：

- 是否需要加入“工作记录区”
- 是否已经具备进入“长期记忆区”的条件

### 规则 4：统一口径
同一个术语的解释要尽量保持一致，不要产生冲突版本。

### 规则 5：适合长期维护
新增内容要便于检索、阅读、扩展和后期渲染。

### 规则 6：写给人和 AI 同时看
既要让人看懂，也要让 AI 能准确识别结构。

# 10. 标准条目模板

后续所有正式术语条目，原则上都要按下面结构来写。

```md
## 词条：［术语名称］

### 基础信息
- 语言：中文 / 英文 / 中英混合
- 分类：产品 / 数据 / 增长 / 商业化 / UX / 工作沟通 / 英文词汇 / 混合黑话
- 相关词：
- 状态：草稿 / 已整理 / 稳定

### 一句话解释
用最直白的人话解释这个词。

### 常见场景
说明这个词通常会出现在什么场景里。
比如：需求评审、数据复盘、增长讨论、商业化分析、跨部门沟通等。

### C 端产品视角
说明这个词放在 C 端产品里，真正意味着什么。
尽量结合：用户体验、行为路径、转化、留存、商业化、信任、阻力。

### 工作中的真实含义
解释当别人说这个词时，对方通常真正想表达什么。

### 常见用法
给出 1～3 个真实工作场景中的表达例子。

### 面试 / 汇报里怎么说
如果适合，补充这个词在面试、复盘、汇报中可以如何更专业地表达。

### 延伸思考
记录围绕这个词，值得继续思考的问题。

### 常见误区
列出这个词最容易被误用、滥用、说空话的地方。

### 备注
补充任何有价值的细节、边界、提醒。
```

# 13. 更新流程

每当新增一个术语或表达时，建议按下面步骤处理：

## 第一步：先收录原词
先把原始内容记录下来，不丢失。

## 第二步：判断分类
判断它更偏：

- 产品
- 数据
- 增长
- 商业化
- UX
- 工作沟通
- 英文词汇
- 混合黑话

## 第三步：先写人话解释
先用最简单的话说明它是什么。

## 第四步：补上工作语境
说明它一般在哪些情况下被提到。

## 第五步：补上 C 端视角
回答：这个词放到 C 端产品里究竟意味着什么？

## 第六步：补常见用法和误区
让它真正能被拿来工作中使用。

## 第七步：成熟后放入长期记忆
如果条目已经足够完整，就进入“长期记忆区”。

# 14. HTML 输出要求

本 Markdown 文件未来可能会被另一个 AI 用来生成 HTML。

因此，需要明确以下要求。

## 14.1 输出目标
生成的 HTML 应当是：

- 有结构的
- 可维护的
- 可扩展的
- 可长期使用的
- 不是一次性拼出来的单页杂糅页面

## 14.2 不接受的输出方式
不希望最终产物只是：

- 一个无层级的大长页
- 一个没有导航逻辑的静态单文件
- 一个只能看、不能扩展的临时 HTML

## 14.3 希望的 HTML 特征
希望 HTML 系统具备：

- 首页 / 总览页
- 分类导航
- 词条索引
- 词条详情结构
- 标签或分类视图
- 后续可继续追加内容的结构能力
- 可维护、可拆分、可重组的页面逻辑

## 14.4 结构优先于视觉
HTML 的重点首先是信息结构正确，而不是只做一个视觉漂亮但难维护的单页。

## 词条：留存

### 基础信息
- 语言：中文
- 分类：产品 / 数据 / 增长
- 相关词：流失、激活、活跃、回访
- 状态：稳定

### 一句话解释
留存，指的是用户在第一次使用之后，是否还会继续回来使用产品。

### 常见场景
常见于：

- 数据复盘
- 用户行为分析
- 增长讨论
- 功能上线复盘
- 产品健康度判断

### C 端产品视角
对于 C 端产品来说，留存本质上是在看：

- 产品有没有真正给到用户价值
- 用户有没有形成继续回来的理由
- 产品体验是不是足够顺
- 用户是否建立了使用习惯
- 价值是否出现得足够早

### 工作中的真实含义
当别人说“留存不好”时，很多时候真正的意思是：

- 用户用了就走
- 产品不够有粘性
- 价值没有及时被感知到
- 早期体验存在问题
- 获得的用户质量本身不高

### 常见用法
- “我们这次更关注首周留存。”
- “这个版本提升了激活，但没有明显改善留存。”
- “下载量不错，但留存一般，说明增长质量不够健康。”

### 面试 / 汇报里怎么说
可以表达为：

“我会把留存看作产品真实价值是否被持续感知的核心信号，而不只是一个结果指标。”

### 延伸思考
- 用户为什么没有回来？
- 是价值不足，还是价值出现得太晚？
- 是产品问题，还是用户来源问题？
- 是否应该结合分群和行为路径一起分析？

### 常见误区
- 只报留存数字，不分析原因
- 不分用户群体，笼统讨论留存
- 以为多发通知就能解决留存问题

### 备注
留存通常需要结合激活、用户路径、分群、使用频次一起看，不能孤立判断。

## 词条：漏斗

### 基础信息
- 语言：中文
- 分类：产品 / 数据 / 增长
- 相关词：转化率、流失、路径分析、用户旅程
- 状态：稳定

### 一句话解释
漏斗，就是用户为了完成某个目标，按步骤往下走的一条路径。

### 常见场景
常见于：

- 注册流程分析
- 支付流程分析
- 新手引导分析
- 订阅转化分析
- 功能转化分析

### C 端产品视角
对于 C 端产品来说，漏斗不是单纯看“掉了多少人”，而是在看：

- 用户在哪一步被卡住了
- 哪一步太麻烦
- 哪一步不够清楚
- 哪一步让用户不信任
- 哪一步让用户失去继续往下走的动力

### 工作中的真实含义
当别人说“看一下漏斗”时，很多时候真实要看的其实是：

- 用户到底卡在哪一步
- 哪一环阻力最大
- 哪个环节拖累了转化
- 是体验问题、策略问题、价格问题还是性能问题

### 常见用法
- “先把注册漏斗拆出来看一下。”
- “支付漏斗最后一步掉得很厉害。”
- “这次改版后，漏斗中段的损耗明显下降。”

### 面试 / 汇报里怎么说
可以表达为：

“我会把漏斗作为用户路径损耗分析工具，不只看数据结果，还要定位用户阻力的具体来源。”

### 延伸思考
- 掉失最严重的是哪一步？
- 这一步的问题是理解成本高，还是操作成本高？
- 能否减少步骤，或者提前建立信任？

### 常见误区
- 只看数字，不看原因
- 一看到漏斗掉失就觉得一定是 UI 问题
- 只优化局部步骤，却破坏整体体验

### 备注
漏斗不只是一个图表，而是一种理解用户路径损耗的方式。

# 19. 模式库

这一部分用于沉淀比“单个术语”更高一级的规律。

## 模式 1：很多黑话其实是压缩过的决策语言
团队里很多词，不只是概念说明，而是在快速传递：

- 优先级
- 风险
- 阻力
- 决策倾向
- 资源取舍

## 模式 2：很多数据词背后其实是产品问题
例如：

- 留存 = 用户为什么留下来
- 流失 = 用户为什么离开
- 转化 = 用户为什么行动
- 激活 = 用户有没有足够快地感受到价值

## 模式 3：同一个词，只有放回 C 端语境里才真正有意义
一个词只有回到：

- 用户感受
- 用户行为
- 产品路径
- 信任建立
- 决策阻力

这些维度上，才会真正变成能指导工作的认知。

# 23. 推荐使用方式

每次听到一个新词时：

1. 先丢进“工作记录区”或“词汇收集区”  
2. 再补一句最简单的人话解释  
3. 后续有更多理解时，再整理进“长期记忆区”  
4. AI 每次写完后，都需要同步检查：
   - 是否更新了工作记录
   - 是否需要更新长期记忆
   - 是否有重复条目需要合并

这样，这个系统才会越用越强，而不是越记越乱。
