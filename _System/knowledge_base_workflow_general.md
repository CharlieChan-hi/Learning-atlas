# 知识体系资料库生产工作流

> 适用场景：用户要求系统学习**任意领域**的知识，需要产出一套完整的、
> 可在 Obsidian/Notion/纸质打印 三端使用的 Markdown 知识库。
> 适用于但不限于：AI、金融、医疗、设计、法律、营销、管理等任何行业。
> 已验证项目：AI产品经理知识体系（62篇，19000+行）

---

## 一、总体流程（7个阶段）

```
阶段1：需求确认与用户画像
    ↓
阶段2：模块结构设计
    ↓
阶段3：排版规范确定
    ↓
阶段4：分批内容生产（并行Agent）
    ↓
阶段5：质量审查与修复
    ↓
阶段6：Obsidian适配检查
    ↓
阶段7：交付与记忆存档
```

---

## 二、阶段详解

### 阶段1：需求确认与用户画像（必须先完成）

**目标：** 明确用户是谁、要学什么、学到什么程度、怎么阅读。

**必须确认的8个问题：**

| 序号 | 问题 | 示例答案 |
|------|------|---------|
| 1 | 用户基础水平 | 完全零基础 / 有一定基础 / 专业人士 |
| 2 | 学习目标 | 求职 / 能力建设 / 转型 / 独立做产品 |
| 3 | 语言偏好 | 纯中文 / 中英双语 / 纯英文 |
| 4 | 英文水平（如选双语） | 初级 / 中等 / 高级 |
| 5 | 阅读工具 | Obsidian / Notion / 纸质打印 / 组合 |
| 6 | 排版风格偏好 | 杂志感 / 简洁 / 学术 |
| 7 | 领域范围 | 整个领域 / 某个子方向 |
| 8 | 特殊要求 | 如"中国生态简单提及"等 |

**输出物：** 用户画像文档（嵌入到主提示词中）

**关键原则：**
- 不要假设用户水平，直接问
- 阅读工具决定排版约束（见阶段3）
- 用户说"最喜欢"的方式要重点优化

---

### 阶段2：模块结构设计

**目标：** 设计整个知识库的模块划分和文件清单。

**设计原则：**
1. **循序渐进**：从"建立感觉→建立概念→建立框架→建立判断"
2. **模块独立**：每个模块可独立阅读，但有推荐顺序
3. **覆盖完整**：基础→进阶→实操→求职→日常应用→持续学习
4. **数量合理**：10-15个模块，每模块3-8篇文件

**推荐的通用模块结构（可按领域调整）：**

```
00_导航与学习方法/     ← 怎么用这套资料
01_领域发展时间线/     ← 发生了什么、为什么
02_基础概念补课/       ← 零基础入门
03_核心知识体系/       ← 核心理论/技术脉络
04_核心岗位能力/       ← 职业技能与思维框架
05_案例与拆解/         ← 真实产品/项目/案例分析
06_高质量信息源/       ← 去哪里持续学习
07_词汇表/             ← 术语速查
08_动手实践/           ← 练习任务
09_求职与面试/         ← 求职全流程
10_工具提效/           ← 日常工作应用
11_学习计划/           ← 4周/8周/12周路线
12_待补充与持续迭代/   ← TODO和扩展方向
```

**领域适配示例：**

| 领域 | 模块01 | 模块03 | 模块05 |
|------|--------|--------|--------|
| AI产品经理 | AI发展时间线 | 从LLM到Agent | ChatGPT/Claude拆解 |
| 金融分析 | 金融市场演进 | 估值与财务建模 | 特斯拉/茅台案例 |
| UX设计 | 设计趋势变迁 | 设计系统与组件 | Apple/Figma拆解 |
| 法律实务 | 法律制度演进 | 诉讼与非诉实务 | 经典判例分析 |
| 数据分析 | 数据驱动发展史 | SQL到Python | 增长黑客案例 |

**每个模块的文件命名规范：**
- 格式：`编号_标题.md`（如 `00_实践模块使用指南.md`）
- 编号两位数，从00开始
- 标题用中文，简洁明了
- 特殊字符用下划线替代

**输出物：**
1. 完整目录结构（含每个文件的一句话说明）
2. README.md（资料库总览）
3. 建议学习路线（至少2-3条路线）

### 阶段3：排版规范确定

**目标：** 建立统一的排版标准，确保全库风格一致。

#### 3.1 Emoji图标系统

| 图标 | 用途 | 使用位置 |
|------|------|---------|
| 📌 一句话版本 | 最简总结 | 每篇开头第一个元素 |
| 📖 这一节在讲什么 | 章节概述 | 紧跟一句话版本之后 |
| 💡 提示 | 补充说明 | 正文中穿插 |
| 🧠 概念标题 | 核心概念 | 概念首次出现 |
| 🎯 视角关联 | 实战关联 | 每个概念后 |
| ⚠️ 常见误解 | 易错点 | 概念讲解后 |
| 🧪 试试看 | 动手练习 | 实操环节 |
| 📝 记住这些 | 本节要点 | 每篇结尾 |
| 📚 延伸阅读 | 推荐资料 | 每篇最后 |
| 🌐 In English | 英文版 | 核心段落后 |
| ⚖️ 区别与对比 | 概念对比 | 需要对比时 |

#### 3.2 双语段落格式

```markdown
中文正文段落写在这里，3-5行。

> 🌐 **In English:**
> English summary goes here. Keep it concise,
> using intermediate-level vocabulary.
```

**规则：**
- 中文是正文，英文是精简版（不是逐字翻译）
- 每个核心概念块后都要有英文段落
- 英文难度根据用户水平调整
- 英文段落用 blockquote 格式

#### 3.3 打印友好约束（如用户需要打印）

| 约束 | 规则 | 原因 |
|------|------|------|
| 表格列数 | ≤4列 | A4纸宽度限制 |
| 代码块宽度 | ≤60字符/行 | 避免打印截断 |
| 段落长度 | 3-5行 | 阅读舒适度 |
| 每篇总长 | 300-500行 | 控制在8-12页A4 |
| 图片 | 避免或用文字描述 | 打印黑白友好 |

#### 3.4 三端兼容规则

**禁止使用（会导致兼容问题）：**
- `&nbsp;` HTML实体 → 用空行替代
- `> [!note]` Obsidian专用callout → 用标准blockquote
- `[[wikilinks]]` Obsidian双链 → 用标准markdown链接
- `<details>` HTML折叠 → 用标准标题分级
- YAML frontmatter → Obsidian会显示"无效属性"

**安全使用：**
- 标准 Markdown 语法（标题/列表/表格/blockquote/代码块）
- Emoji 图标（三端都能显示）
- 标准 `[text](url)` 链接

#### 3.5 Obsidian 专项适配

**文件开头禁止：**
- 不要以 `# 标题` 开头 → Obsidian自动用文件名做标题，H1会重复
- 不要以 `---` 开头 → Obsidian会误认为YAML frontmatter，显示"无效属性"

**正确的文件开头格式：**
```markdown
> 📌 **一句话版本：** 这里写一句话总结。

---

## 📖 这一节在讲什么

正文从这里开始...
```

**错误示例（会出问题）：**
```markdown
# 标题名称        ← 和文件名重复！
---               ← 被误认为frontmatter！
```

### 阶段4：分批内容生产（并行Agent策略）

**目标：** 高效批量生产内容，同时保证质量。

#### 4.1 分批策略

**原则：每批3-5个模块，每批启动3-4个并行Agent。**

```
第1批：基础模块（导航 + 时间线 + 基础概念）
第2批：核心模块（技术演进 + 核心能力）
第3批：应用模块（案例 + 信息源 + 词汇表）
第4批：实操模块（实践 + 求职 + 工具 + 计划）
第5批：收尾模块（迭代方向 + 质量审查）
```

**每批之间要做：**
1. 验证文件是否全部创建成功
2. 检查 `&nbsp;` 等格式问题
3. 更新项目记忆文件
4. 向用户汇报进度

#### 4.2 Agent提示词模板

每个Agent必须包含以下信息：

```
1. 角色定义：你是一个[领域]知识体系的内容写作者
2. 目标目录：完整路径
3. 排版规则：完整复制阶段3的规则
4. 文件清单：每个文件名 + 内容要求（3-5行描述）
5. 禁止事项：
   - 禁止使用 &nbsp; HTML实体
   - 禁止以 # 标题开头（Obsidian会重复）
   - 禁止以 --- 开头（会被误认为frontmatter）
6. 质量要求：每篇300-500行，零基础友好
```

#### 4.3 Agent失败处理

```
Agent返回502/错误
    ↓
检查哪些文件已创建、哪些缺失
    ↓
只对缺失文件启动新Agent重试
    ↓
重试仍失败 → 换用sonnet模型 / 拆分为更小批次
```

**关键：不要重试已成功的文件，只补缺失的。**

#### 4.4 每批交付后的验证清单

```bash
# 1. 检查文件是否全部存在
ls 目标目录/

# 2. 检查 &nbsp; 问题
grep -rl '&nbsp;' 目标目录/ || echo "clean"

# 3. 检查是否以 # 开头（Obsidian标题重复）
for f in 目标目录/*.md; do
  head -1 "$f" | grep -q "^# " && echo "需修复: $f"
done

# 4. 检查是否以 --- 开头（frontmatter误判）
for f in 目标目录/*.md; do
  [ "$(head -1 "$f")" = "---" ] && echo "需修复: $f"
done

# 5. 统计文件数和总行数
find 目标目录/ -name "*.md" | wc -l
find 目标目录/ -name "*.md" -exec wc -l {} + | tail -1
```

### 阶段5：质量审查与修复

**目标：** 确保全库内容和格式质量达标。

#### 5.1 抽检策略

- 每个模块至少抽检1篇
- 用Agent做批量审查（只读不改）
- 审查维度见下方检查表

#### 5.2 单篇质量检查表

| 检查项 | 要求 | 优先级 |
|--------|------|--------|
| 📌 一句话版本 | 文件开头第一个元素 | 高 |
| 📖 这一节在讲什么 | 紧跟其后 | 高 |
| 🌐 英文段落 | 每个核心概念块后都有 | 高 |
| 📝 记住这些 | 文件结尾有要点总结 | 高 |
| 📚 延伸阅读 | 文件最后有推荐资料 | 中 |
| 表格列数 | ≤4列 | 高 |
| &nbsp; | 不能出现 | 高 |
| 段落长度 | 3-5行 | 中 |
| 零基础友好 | 无未解释的术语 | 高 |
| 文件开头 | 不以#或---开头 | 高 |

#### 5.3 常见问题修复方法

| 问题 | 修复命令 |
|------|---------|
| &nbsp; | `sed -i '' 's/&nbsp;/ /g' file.md` |
| H1标题重复 | `sed -i '' '1d' file.md`（删第一行） |
| ---开头 | `sed -i '' '1d' file.md`（删第一行） |
| 表格超4列 | 手动合并列（如"速度"+"成本"→"速度与成本"） |
| 缺少英文段落 | 在对应中文段落后补充blockquote |
| 缺少延伸阅读 | 在文件末尾补充📚板块 |

### 阶段6：Obsidian适配检查

**目标：** 确保在Obsidian中打开无报错、显示正常。

**检查清单：**

1. **用URL scheme打开vault：**
   ```bash
   open "obsidian://open?path=/完整路径/知识库目录"
   ```

2. **逐一检查：**
   - [ ] 左侧目录树结构正确
   - [ ] 打开文件无"无效属性"红色提示
   - [ ] 标题不重复显示
   - [ ] 表格正常渲染
   - [ ] Emoji图标正常显示
   - [ ] 英文blockquote正常缩进
   - [ ] 内部链接可跳转

3. **常见Obsidian问题：**
   - "无效属性" → 文件开头有被误认为frontmatter的 `---`
   - 标题重复 → 文件内有H1和文件名重复
   - 链接失效 → 用了wikilink而非标准markdown链接

### 阶段7：交付与记忆存档

**目标：** 完成交付，存档项目记忆供未来复用。

#### 7.1 交付汇报模板

```
## 交付完成！

### 最终成果
| 指标 | 数据 |
|------|------|
| 总文件数 | XX 个 |
| 总行数 | XX 行 |
| 模块数 | XX 个 |

### 模块总览
（每模块一行：名称 + 篇数 + 状态）

### 质量检查结果
- 无 &nbsp; ✅
- 无标题重复 ✅
- 无frontmatter误判 ✅
```

在 `memory/` 目录创建项目记忆文件，记录：

- 关键决策（排版规则、语言规则等）
- 交付进度（每批的完成状态）
- 最终统计数据

## 三、已验证的经验教训

### 3.1 必须避免的坑

| 坑 | 后果 | 预防 |
|----|------|------|
| 文件以 `# 标题` 开头 | Obsidian标题重复显示 | Agent提示词中明确禁止 |
| 文件以 `---` 开头 | Obsidian显示"无效属性" | Agent提示词中明确禁止 |
| 使用 `&nbsp;` | Obsidian显示为原始文本 | Agent提示词+批量grep检查 |
| 表格超过4列 | 打印时截断 | Agent提示词+抽检 |
| Agent 502错误 | 部分文件缺失 | 检查→只补缺失文件→重试 |
| 英文段落覆盖不均 | 部分内容无英文 | 审查时重点检查 |

### 3.2 效率最大化技巧

1. **并行Agent**：每批启动3-4个Agent，每个负责一个模块
2. **sonnet模型**：内容生产用sonnet（快+便宜），审查用sonnet
3. **批量验证**：用bash脚本一次性检查所有已知问题
4. **增量修复**：只修有问题的文件，不重写正常文件
5. **提示词复用**：排版规则在每个Agent提示词中完整复制

### 3.3 文件大小参考

| 文件类型 | 建议行数 | 建议大小 |
|---------|---------|---------|
| 导航/概述类 | 200-300行 | 5-8KB |
| 概念讲解类 | 300-500行 | 8-15KB |
| 案例分析类 | 300-500行 | 8-15KB |
| 词汇表 | 400-600行 | 15-25KB |
| 学习计划 | 500-800行 | 15-20KB |
| TODO/收尾 | 200-400行 | 5-10KB |

## 五、Agent提示词完整模板

```
你是一个[领域]知识体系的内容写作者。
请在以下目录创建[N]个Markdown文件。

**目标目录：** [完整路径]

**必须遵守的排版规则：**
- 中文正文，核心段落后附英文精简版
  （用 `> 🌐 **In English:**` 引用块）
- Emoji图标分区（📌/📖/💡/🧠/🎯/⚠️/🧪/📝/📚/🌐）
- 表格≤4列，代码块≤60字符/行，段落3-5行
- 禁止使用 &nbsp; HTML实体
- 禁止以 # 标题开头（Obsidian会与文件名重复）
- 禁止以 --- 开头（会被Obsidian误认为frontmatter）
- 每篇300-500行，适合A4打印
- 面向[用户画像描述]

**正确的文件开头格式：**
> 📌 **一句话版本：** ……
然后是 ---，然后是 ## 📖 这一节在讲什么

**[N]个文件及内容要求：**
1. **文件名.md** — 内容要求描述
2. ...

每个文件都用Write工具创建。
```
