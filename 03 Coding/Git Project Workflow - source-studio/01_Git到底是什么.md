> 一句话版本：Git 是给项目做“可回退、可对比、可同步”的版本管理工具，像游戏存档，但比游戏存档更强。

---

## 这一篇解决什么问题

你先不用记命令。先建立感觉：Git 到底是什么？它为什么不是普通文件夹？它为什么对你的网站重要？

读完这一篇，你应该能说清楚：Git 管的是项目变化历史，不只是当前文件。

## Git 像什么

你可以先把 Git 理解成“项目存档系统”。

但它比游戏存档更强：

- 它能记录每次改了哪些文件；
- 它能显示具体哪几行变了；
- 它能写下这次修改的原因；
- 它能回到以前的版本；
- 它能让 A 电脑和 B 电脑同步；
- 它能和 GitHub、Cloudflare Pages 连接起来。

所以 Git 不是只给程序员用的“黑色命令行工具”。对你的个人网站来说，它是长期维护系统。

## Git 管的不是“一个文件”，而是一整个项目

你的 `charliechanstudio.com` 不是一个 Word 文档，而是一整个项目：

- 页面代码；
- 内容文件；
- 样式；
- 图片资源；
- 配置；
- 文档；
- 部署说明。

Git 能把这些文件作为一个整体来管理。你每次提交，不是只保存一个文件，而是保存“这次项目整体发生了什么变化”。

## Git 的四个位置

Git 最重要的是四个位置：

| 位置 | 英文 | 人话解释 |
| --- | --- | --- |
| 工作区 | Working Tree | 你电脑里正在编辑的文件夹 |
| 暂存区 | Staging Area | 准备放进下一次存档的清单 |
| 本地仓库 | Local Repository | 你电脑里的正式版本历史 |
| 远程仓库 | Remote Repository | GitHub 上的云端版本历史 |

你可以想象成：

```text
正在改文件
  → 选中这次要保存的文件
  → 在本机生成一个正式存档
  → 上传到 GitHub 云端
```

对应 Git 动作是：

```text
修改文件
  → git add
  → git commit
  → git push
```

## commit 是什么

commit 是 Git 里的正式存档点。

一个 commit 通常包含：

- 这次改了哪些文件；
- 每个文件改了哪些行；
- 提交时间；
- 提交人；
- 提交说明。

好的 commit 就像一条清楚的历史记录：以后你看到它，就知道当时为什么这么改。

## Git 为什么对你的网站重要

你的个人网站是长期项目，不是一次性作业。没有 Git 的话，容易出现这些问题：

- 不知道哪些文件改过；
- 不知道某个问题是哪次改出来的；
- 改坏后不知道怎么回去；
- 换电脑后不知道哪个版本最新；
- 部署到 Cloudflare Pages 时没有可靠来源。

有 Git 后，你的网站可以变成：

```text
本机开发
→ Git 保存历史
→ GitHub 云端备份
→ Cloudflare Pages 自动部署
```

## 这一篇记住这些

- Git 是版本管理工具，不是普通网盘。
- commit 像正式存档点。
- push 才是上传到 GitHub。
- 你的项目已经有本地 Git，但还没有远程 GitHub 仓库。
- 先理解工作区、暂存区、本地仓库、远程仓库，后面命令就容易很多。
