> 一句话版本：回滚不要乱用 reset；未提交改动用 restore，已提交历史优先用 revert，危险命令先不要碰。

---

## 这一篇解决什么问题

Git 最容易出事的地方就是“我想撤回”。

读完这一篇，你应该知道不同阶段要用不同方法，而不是一看到问题就 `reset --hard`。

## 先记住安全原则

```text
还没提交的改动，用 restore 处理。
已经提交的历史，优先用 revert。
不熟悉时，不要用 reset --hard。
```

回滚前永远先看状态：

```bash
git status --short --branch
```

## 情况 1：文件改坏了，但还没 add

如果你只是修改了文件，还没有放进暂存区，可以用：

```bash
git restore 文件名
```

这会把这个文件恢复到上一次 commit 的样子。

注意：这会丢掉这个文件当前未提交的修改。所以只有在你确认不要这些修改时才用。

## 情况 2：已经 add 了，但还没 commit

如果你已经 `git add` 了，但后悔了，可以取消暂存：

```bash
git restore --staged 文件名
```

这只是把文件从暂存区拿出来。文件内容还在，不会删除你的修改。

之后如果你也想丢掉文件内容，再用：

```bash
git restore 文件名
```

## 情况 3：已经 commit，但还没 push

如果已经 commit，但还没有 push，先不要乱操作。

初学阶段更安全的做法通常是：再做一个修正 commit。

例如：

```text
第一次 commit：改错了
第二次 commit：修正刚才的问题
```

这样历史虽然多一条，但安全、清楚、不容易丢东西。

## 情况 4：已经 push 到 GitHub

如果已经 push 到远程仓库，优先用：

```bash
git revert 某个提交ID
```

revert 的意思是：创建一个新的提交，用来抵消之前那个提交。

它的好处是：

- 不重写远程历史；
- 不容易影响其他电脑；
- 更适合已经上传的项目。

## 危险命令：先不要碰

这些命令不是永远不能用，而是初学阶段非常容易造成损失：

```bash
git reset --hard
git clean -fd
git push --force
```

它们可能造成：

| 命令 | 风险 |
| --- | --- |
| `git reset --hard` | 丢掉未提交修改，移动历史指针 |
| `git clean -fd` | 删除 Git 还没跟踪的新文件 |
| `git push --force` | 覆盖远程历史，影响其他电脑或协作者 |

你的 `charliechanstudio.com` 当前有很多未提交改动，所以尤其不能乱用这些命令。

## 回滚前检查清单

在做任何撤回前，先问自己：

```text
□ 我看过 git status 了吗？
□ 我要撤回的是一个文件，还是一次 commit？
□ 这个改动有没有 push 到 GitHub？
□ 我确定不要这些未提交内容了吗？
□ 有没有可能先复制一份文字内容再操作？
```

## 这一篇记住这些

- 未 add 的文件：`git restore 文件名`。
- 已 add 未 commit：`git restore --staged 文件名`。
- 已 push 的 commit：优先 `git revert`。
- 初学阶段不要乱用 `reset --hard`、`clean -fd`、`push --force`。
- 回滚前先看 status，不要凭感觉操作。
