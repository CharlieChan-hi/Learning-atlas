> 一句话版本：初学 Git 不需要先学一百个命令，先会 `status`、`diff`、`add`、`commit`、`log` 就够。

---

## 这一篇解决什么问题

你会看到很多 Git 命令，但一开始不要被吓到。真正每天都用的命令很少。

读完这一篇，你应该知道：每个命令是为了回答哪个问题。

## 1. 看当前状态

```bash
git status --short --branch
```

它回答的是：

```text
我现在在哪个分支？
哪些文件变了？
哪些文件还没被 Git 跟踪？
```

你可能看到：

```text
## main
 M README.md
 D docs/old.md
?? docs/new.md
```

含义：

| 符号 | 意思 |
| --- | --- |
| `M` | modified，文件被修改了 |
| `D` | deleted，文件被删除了 |
| `??` | untracked，新文件，Git 还没跟踪 |
| `A` | added，新文件已加入暂存 |

这是你每天最应该先看的命令。

## 2. 看具体改了什么

```bash
git diff
```

它回答的是：

```text
文件到底哪几行变了？
```

你要养成习惯：提交前一定看 diff。

如果你已经 `git add` 了，想看暂存区里的改动：

```bash
git diff --staged
```

## 3. 选择文件进入下一次提交

```bash
git add README.md
```

它回答的是：

```text
我想把哪些改动放进下一次 commit？
```

初学阶段，优先写明确文件名，不要习惯性用：

```bash
git add .
```

因为 `git add .` 会把当前目录下很多你没注意的文件一起加入暂存区。

## 4. 创建一次提交

```bash
git commit -m "说明这次为什么改"
```

commit 是正式存档点。

好的 commit message 应该说“为什么改”，例如：

```text
整理部署文档，降低 Cloudflare 配置出错风险
```

不要只写：

```text
update files
```

## 5. 看历史树

```bash
git log --oneline --graph --decorate --all
```

它回答的是：

```text
项目过去有哪些 commit？
分支像树一样怎么分出来？
我现在的位置在哪里？
```

这就是命令行里看“树状历史”的方式。

## 一个最小流程

日常最小流程是：

```bash
git status --short --branch
git diff
git add 文件名
git diff --staged
git commit -m "说明修改原因"
```

如果以后有 GitHub 远程仓库，再加：

```bash
git push
```

## 这一篇记住这些

- `status` 看状态。
- `diff` 看具体变化。
- `add` 选择要提交的内容。
- `commit` 创建本地版本点。
- `log --graph` 看历史树。
- 真实项目里不要没看 diff 就提交。
