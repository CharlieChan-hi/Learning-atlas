> 📌 **一句话版本：** 学终端最重要的不是多背几十条命令，而是形成一套“先看位置、再看状态、再看输出、最后再改动”的排错思路。

## 📖 这一节在讲什么

这页是终端模块的收口页。前面你已经见过路径、命令和环境，现在要再补一层：当命令不工作时，怎么不慌。

## 🧠 高频排错顺序

1. 先看自己在哪：`pwd`
2. 再看这里有什么：`ls`
3. 再看命令是否存在：版本命令或 `which`
4. 再看输出和报错细节
5. 最后才决定是否安装、删除或重试

## 🎯 常见报错类型

- `command not found`：程序不存在或 `PATH` 没配好
- `no such file or directory`：路径错了
- `permission denied`：权限不够
- 依赖版本不匹配：环境差异问题

> 🌐 **In English:**
> A stable terminal workflow depends less on memorizing commands and more on checking location, state, and output in the right order.

## 🧩 终端排错为什么容易让人慌

因为终端通常不会给你图形化提示，它只会吐几行字。只要你还没有形成“先分类、再缩小范围”的习惯，就很容易觉得这些字都像天书。

其实大多数终端问题，最终都能归到几个常见方向：

- 位置不对
- 程序没装或找不到
- 权限不够
- 参数写错
- 环境变量或依赖有问题

## 🎯 一个更完整的排错顺序

### 第一步：确认当前位置

你现在是不是在正确目录里。很多命令本身没问题，但执行位置错了。

### 第二步：确认目标对象是否存在

文件、目录、脚本、配置文件是否真的在你要操作的位置。

### 第三步：确认命令本身是否可用

是没装、装了但不在 PATH 里，还是版本不对。

### 第四步：读完整输出

不要只看最后一句，前后上下文通常才决定根因。

### 第五步：只做一个最小修复动作

先试一个最可能的修复方向，再看结果，不要一口气乱试很多命令。

> 🌐 **In English:**
> Good terminal troubleshooting follows an order: verify location, confirm targets, check command availability, read output context, and make one controlled fix at a time.

## 💡 排错本质上是在缩小可能性

排错不是拍脑袋想到“神奇答案”，而是一步步排除：

- 如果目录是对的，就排除位置问题
- 如果程序存在，就排除安装问题
- 如果权限没问题，就继续看参数和环境

你每确认一件事，问题范围就会缩小一点。

## ⚠️ 最容易让问题更乱的三种做法

### 做法一：不看输出，直接继续下一步

### 做法二：一口气复制很多命令

### 做法三：没搞清根因就重装一切

这三种做法都会让你最后更难知道真正问题在哪。

## 🧪 一个适合 PM 的排错模板

你可以快速写下：

1. 我现在在哪
2. 我要操作什么
3. 当前报错更像哪一类
4. 我下一步只试一个什么动作

这个模板很朴素，但已经足够让终端排错变得清楚很多。

## 📝 记住这些

- 大多数终端问题都有线索，不是纯黑盒。
- 先看状态，再做动作，能减少大量误操作。
- 对 PM 来说，会排最基础的命令问题已经很有价值。
