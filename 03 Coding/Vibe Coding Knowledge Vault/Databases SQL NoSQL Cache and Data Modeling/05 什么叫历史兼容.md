> 📌 **一句话版本：** 数据结构和查询逻辑一旦进入真实系统，后面每次改动都不只是“改一行 SQL”，而可能影响历史数据、接口返回、报表口径和上线风险；这就是兼容性和迁移风险。

---

## 📖 这一节在讲什么

很多需求改动，最终都会落到查询和结构变更上。比如：

- 新增字段
- 拆分表
- 改状态值
- 改查询条件
- 改排序逻辑

这些动作看起来像后端或数据库自己的事，但它们往往会直接影响产品功能和历史数据表现。

## 🧠 什么叫历史兼容

历史兼容的意思是：新逻辑上线以后，旧数据还能不能被正确理解、正确展示、正确统计。

如果不能，就会出现：

- 老记录打不开
- 新旧状态混在一起
- 统计结果突然跳变
- 页面某些旧数据结构渲染异常

> 🌐 **In English:**
> Schema or query changes are risky because they affect not only new data but also how old data is interpreted, displayed, and counted.

## 🎯 什么叫迁移风险

迁移风险通常出现在：

- 字段改名
- 数据类型变化
- 枚举值变化
- 一张表拆成多张表
- 旧查询逻辑切到新逻辑

这些变化如果没有明确的迁移方案，就很容易产生“系统升级了，但历史世界没跟上”的问题。

## 💡 PM 在这类改动里最该问什么

1. 旧数据怎么处理
2. 新旧逻辑会不会并存一段时间
3. 接口和页面是否要同时兼容
4. 报表和统计口径是否要同步调整
5. 上线失败时怎么回退

## ⚠️ 最常见的误判

### 误判一：只要新数据正常就算成功

### 误判二：接口改好了，页面自然会兼容

### 误判三：历史数据量不大，就可以手动补

## 🧪 一个很实用的迁移自检清单

每次遇到数据结构或查询变更，都可以先过这几句：

- 影响的是新数据、旧数据，还是两者都影响
- 是否需要回填或迁移脚本
- 是否需要灰度期
- 是否需要临时兼容逻辑
- 验收时要不要单独看历史样本

## 📝 记住这些

- 数据变更的真正难点，常常在历史兼容而不是新逻辑本身。
- 一次看似很小的字段或查询改动，可能影响多个模块。
- PM 如果能在变更前先问兼容性，很多返工和线上事故都能提前避免。
