> 📌 **一句话版本：** `Codex CLI` 和 `Claude Code` 都是在终端里工作的 AI 编码助手，但真正决定产出质量的，不是你选哪个工具，而是你能不能提供清晰上下文、控制改动范围并验证结果。

---

## 📖 这一节在讲什么

既然你的目标是把 `Vibe Coding` 真正用起来，那 `Codex CLI` 和 `Claude Code` 就一定是核心主题。它们都不是“自动生成一段代码然后结束”的小工具，而是可以在代码仓里理解上下文、读写文件、运行命令、调用工具并与你持续协作的终端型 agent。

## 🧠 先建立正确认知

你可以把这类工具理解成：

- 会读代码的对话式搭子
- 会执行命令的工程助手
- 会在仓库上下文里工作的 agent
- 但仍然需要你持续约束和验证的自动化协作者

这意味着它们不是“替代你思考”，而是“放大你的问题定义能力和验证能力”。

> 🌐 **In English:**
> Codex CLI and Claude Code are terminal-based coding agents. They can inspect code, edit files, run commands, and collaborate in context, but they still require human guidance and verification.

## 🎯 它们共同擅长什么

- 阅读代码仓结构
- 搜索相关文件和调用链
- 解释报错和技术细节
- 提出修改方案
- 局部改代码
- 运行测试、命令和验证步骤
- 协助文档、脚本和自动化工作流

## ⚖️ 你真正该比较的维度

### 上下文理解能力

看它能不能在一个真实仓库里把需求、历史文件、依赖关系串起来。

### 工具调用能力

看它能不能配合终端、浏览器、MCP、设计、文档、测试工具一起工作。

### 改动可控性

看它是否会说明自己改了什么、为什么改、风险在哪，能不能只做局部变更。

### 验证意识

看它会不会主动跑测试、检查报错、提醒风险，而不是只生成“看起来像对的代码”。

## 💡 对 PM 来说，最有价值的不是“让它写整站”，而是这些高频场景

- 快速理解一个旧项目
- 根据 PRD 拆成工程任务
- 看懂某个报错和影响范围
- 帮你生成脚本、接口样例、测试思路、调试步骤
- 先做一个可运行的原型或最小功能
- 帮你 review 需求是否容易引发技术风险

## 🧭 一个更稳的使用方式

### 第一步：把目标说清楚

不是“帮我改好”，而是“请定位登录页报错来源，只修复重复请求问题，不要动 UI 样式，并在最后告诉我改了哪些文件”。

### 第二步：控制范围

明确哪些文件能动、哪些不能动、是否允许跑测试、是否允许装依赖、是否允许联网。

### 第三步：要求它解释

让它说明原理、影响范围、风险等级和验证方法。这样你不至于只收下一堆黑盒改动。

### 第四步：自己验证

至少要看改动文件、运行结果、报错信息和关键路径是否正常。

> 🌐 **In English:**
> The safest workflow is: define the task clearly, constrain the scope, require explanations, and verify the output yourself.

## ⚠️ 常见误区

### 误区一：AI coding 工具会自己理解业务上下文

除非你明确告诉它需求、边界、环境和目标，否则它只能猜。

### 误区二：工具越强，我越不用懂技术

工具越强，你越需要懂最小必要技术，否则你连它错在哪都判断不了。

### 误区三：只要代码能跑就算成功

真实工程里还要看风格一致性、边界条件、回归风险、测试覆盖、部署影响和安全问题。

## ⚖️ 怎么选更实际

最实际的做法通常不是“二选一”，而是理解它们都属于同一类生产工具，然后根据任务场景使用：

- 想快速理解仓库和执行局部修复时，用你当前手上最顺手的那个
- 想做多工具联动、MCP、文档、浏览器、设计协作时，看当前环境配置
- 想降低风险时，优先用小范围、可验证、可回滚的任务来驱动

## 🎯 PM 的最佳姿势

- 把它们当“带工具的高级实习工程师”，不是“自动交付机器”
- 让它们先分析、再改、再验证，而不是一步到位盲改
- 用 Git 和测试保护改动边界
- 用 JSON、日志、接口、命令输出做客观验证

## 🧩 它们和普通聊天机器人最大的区别

你可以把普通聊天机器人理解成“只会基于你贴进去的文本回答”。而 `Codex CLI`、`Claude Code` 这种终端型编码 agent，差别在于它们通常还能：

- 直接读取仓库文件
- 搜索目录结构
- 运行命令
- 调用工具
- 根据结果继续下一步

这意味着它们不只是“回答问题”，而是在参与一个真实工作回路。

## 🔄 一个完整的 AI Coding 回路长什么样

真正高质量的使用方式通常长这样：

1. 你定义目标和边界
2. 工具先读代码和相关文件
3. 工具给出分析和修改计划
4. 工具执行局部改动
5. 工具运行验证步骤
6. 你审查改动和结果
7. 根据问题继续下一轮

如果少了第 5、6、7 步，这就更像“碰运气生成代码”，而不是可靠协作。

> 🌐 **In English:**
> The real value of AI coding tools comes from iterative loops: inspect, plan, edit, verify, review, and repeat.

## 🎯 什么样的任务最适合交给它们

### 适合

- 解释一个旧模块在做什么
- 定位报错来源
- 小范围修复
- 生成测试思路或脚本
- 生成初版页面或自动化脚本
- 把 PRD 转成工程任务清单

### 不适合直接放手

- 完全没有边界的大重构
- 涉及高风险数据或密钥的随意操作
- 你自己完全无法验证结果的任务
- 需求本身还没想清楚的任务

## 🔧 MCP、工具调用、上下文控制为什么重要

当 agent 不只是“写文本”，而是要访问浏览器、设计稿、文档、数据库或其他服务时，工具能力就很重要。这也是你之前允许我自己接 MCP 的原因：它可以让知识和动作更贴近真实工作环境。

但工具越多，风险也越高，所以你更要关注：

- 它能访问什么
- 它改了什么
- 哪些动作需要额外审查
- 哪些结果必须人工验证

## 💡 你该怎么给这类工具下任务

一个更靠谱的任务描述通常应该包含：

- 目标：要解决什么问题
- 范围：允许改哪些文件
- 禁止项：哪些内容不能动
- 验证：执行后要跑什么检查
- 输出：最后要汇报哪些结果

例如：

“请分析登录页面重复请求问题，只修改登录相关文件，不要动样式和文案，修复后运行测试，并告诉我修改原因、风险和验证结果。”

## ⚠️ AI Coding 最容易翻车的四种情况

### 需求边界不清

工具会自己脑补，结果改动越做越大。

### 上下文给得不够

它不知道历史约束、项目规范或真实业务规则，只能按常识猜。

### 不做验证

代码可能能跑，但逻辑、边界条件或回归问题仍然存在。

### 不看 diff

你只看“它说修好了”，却没看它到底改了什么。

## 📝 记住这些

- 工具只是放大器，真正稀缺的是你的上下文组织能力和判断力。
- `Codex CLI` 和 `Claude Code` 都很适合进入你的 PM 技术工作流。
- 最稳的方式永远是“小步快跑、局部修改、持续验证”。
