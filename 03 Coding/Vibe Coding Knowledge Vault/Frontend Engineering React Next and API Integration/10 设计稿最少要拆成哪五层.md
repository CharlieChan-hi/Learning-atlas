> 📌 **一句话版本：** 设计稿不是“长得像就行”的图片，它应该被拆成组件、布局、状态、交互和接口五层；这也是 PM、设计和前端协作最容易出偏差的地方。

---

## 📖 这一节在讲什么

很多需求从设计稿走向实现时，会出现这种情况：

- 页面长得差不多
- 但状态不对
- 交互不完整
- 错误反馈缺失
- 接口字段根本没对齐

这不是设计稿没价值，而是设计稿没有被拆成工程可执行结构。

## 🧠 设计稿最少要拆成哪五层

### 第一层：布局层

页面整体区块怎么分，哪些区域固定，哪些区域滚动，移动端和桌面端差异在哪。

### 第二层：组件层

设计稿里的按钮、卡片、列表、表单，是不是能映射成已有组件，还是需要新组件。

### 第三层：状态层

这页在：

- 加载中
- 成功
- 失败
- 空态
- 权限受限

这些状态下分别长什么样。

### 第四层：交互层

点击、输入、悬停、切换、复制、跳转这些动作分别会触发什么。

### 第五层：数据层

页面显示的哪些值来自接口，哪些来自前端本地状态，哪些来自 URL 或上下文。

> 🌐 **In English:**
> A design should be translated into layout, components, states, interactions, and data dependencies before it becomes an implementation task.

## 🎯 为什么“设计还原度高”不等于“实现质量高”

因为设计稿只能告诉你静态画面，不会自动告诉你：

- 错误怎么处理
- 数据没回来时怎么办
- 接口失败时怎么反馈
- 某个模块是否需要权限判断

所以实现真正难的部分，经常不是“画出来”，而是“把系统状态补完整”。

## 💡 PM 在设计转实现阶段最该补哪一刀

就是把设计稿里没有明说的动态逻辑补出来：

- 什么时候 loading
- 什么时候禁用按钮
- 什么情况下要提示错误
- 页面哪些部分依赖真实接口

这一步如果没人补，前端就会开始自己猜，后面返工概率很高。

## ⚠️ 最常见的错位

### 错位一：设计稿只覆盖主路径

### 错位二：设计稿里的组件和项目组件库不一致

### 错位三：设计稿里显示的是静态内容，但实现需要动态数据

### 错位四：设计稿没有体现权限和异常态

## 🧪 一个很好用的拆解动作

以后你拿到设计稿，可以先用笔记列出：

1. 页面区块
2. 组件清单
3. 状态清单
4. 交互动作
5. 数据依赖

只要你把这五行写出来，设计到实现的落差就会小很多。

## 📝 记住这些

- 设计稿不是终点，而是实现拆解的起点。
- PM 的关键价值，是把静态设计转成动态执行结构。
- 组件、状态、交互和数据层不拆清，页面再好看也容易落地失真。
