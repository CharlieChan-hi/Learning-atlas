> 📌 **一句话版本：** 一个页面是否“像个成熟产品”，很多时候不取决于主流程，而取决于权限页、空态、错误态和骨架屏这些看似边角、实际决定体验质量的状态设计。

---

## 📖 这一节在讲什么

很多团队在做页面时，只重点关注“正常情况下内容怎么展示”，但真实用户经常遇到的却是：

- 没权限
- 没数据
- 请求失败
- 数据还在加载

这就是为什么空态、错误态、权限态和骨架屏不是“最后再补”的小优化，而是页面设计的一部分。

## 🧠 什么是权限态

权限态表示：用户进入了某个页面或某个模块，但系统判断他不能进行某些操作或不能看到某些内容。

常见表现包括：

- 页面可见，但按钮禁用
- 页面可见，但内容被遮罩或替换
- 页面直接给出无权限提示

> 🌐 **In English:**
> Permission states, empty states, error states, and loading skeletons are part of the real product experience, not optional polish.

## 🎯 什么是空态

空态不是“什么都没有”，而是系统要明确告诉用户：

- 当前真的没有数据
- 为什么没有
- 下一步可以做什么

一个好的空态通常会包含：

- 清楚解释
- 建议动作
- 可能的入口按钮

例如：

- “你还没有生成过任何周报，点击下方开始第一次生成。”

## 🎯 什么是错误态

错误态的目标不是证明系统失败了，而是帮助用户理解当前状况并决定下一步。

一个好的错误态至少要回答：

- 大概发生了什么
- 用户现在能做什么
- 是否值得重试

如果只有一句“系统异常”，用户和 PM 都很难继续推进。

## 🎯 什么是骨架屏

骨架屏可以理解成“加载中的结构预览”，用来告诉用户：

- 页面还在加载
- 大概会出现什么布局
- 系统没有卡死

它和转圈的差别在于：骨架屏更强调结构感和心理预期。

> 🌐 **In English:**
> A skeleton screen is not just decoration. It helps users understand that content is loading and what kind of structure to expect.

## 💡 这四类状态为什么特别容易被 PM 忽略

因为它们通常不在需求 happy path 的中心位置。但恰恰是这些状态，最容易暴露团队有没有真正把体验想完整。

例如：

- 没权限时到底是拦路，还是给解释
- 没数据时是冷冰冰空白，还是给出下一步
- 报错后是只能重来，还是能局部恢复
- 加载时是全屏阻塞，还是局部占位

这些差别，会直接决定产品看起来是“粗糙 Demo”还是“成熟系统”。

## ⚠️ 最常见的体验坑

### 坑一：空态和错误态混在一起

用户分不清是“本来就没数据”，还是“系统挂了”。

### 坑二：权限态只在前端做隐藏

表面体验看似正常，但实际风险仍然存在。

### 坑三：骨架屏和真实布局不一致

用户等到内容出来时，会觉得页面跳动和混乱。

### 坑四：错误态没有恢复动作

用户知道出错了，但完全不知道下一步能做什么。

## 🧩 PM 在验收这类状态时最该看什么

你可以按下面这条线去看：

1. 用户能不能分清当前到底是什么状态
2. 用户是否知道下一步动作
3. 页面不同状态之间切换是否自然
4. 权限、空态、错误和加载是否都被单独设计过

## 🧪 一份很实用的页面状态检查单

以后做页面验收时，除了主路径，你可以强制再看四类状态：

- 无权限
- 无数据
- 请求失败
- 数据加载中

只要这四类状态有两类没设计清楚，这个页面通常都还不算成熟。

## 📝 记住这些

- 真正成熟的页面，不只是主流程通，而是异常和边角状态也通。
- 权限态、空态、错误态和骨架屏，都是产品体验的一部分。
- PM 如果能稳定关注这四类状态，页面质量会显著提升。
