## 📖 这一节在讲什么

当页面接入真实后端以后，问题通常不会像教程里那样简单。你会遇到：

- 页面能打开但没数据
- 请求发了但页面没更新
- 接口返回了但字段不对
- 用户点击后状态卡死
- 某些账号能用，某些账号不能用

这些问题如果不按层排，很容易陷入“前端说后端有问题，后端说前端有问题”的循环。

## 🧠 一个联调故障最常见的五层定位法

### 第一层：路由和入口

先看用户是不是从正确路径进入了页面。

你要确认：

- 当前 URL 对不对
- 路由参数有没有丢
- 页面初始化逻辑是否触发

### 第二层：请求参数

再看前端发出去的请求到底长什么样。

重点确认：

- 请求有没有真的发出
- 参数名对不对
- 参数值是不是旧的
- 是否缺少 Token 或上下文信息

### 第三层：接口返回

接口返回时要看：

- 状态码
- 业务 code
- data 字段结构
- 错误信息

很多联调问题看起来像页面问题，其实只是字段名、字段类型或业务 code 约定不一致。

### 第四层：状态映射

有时接口和参数都没问题，真正问题在于前端没有把返回结果正确写进状态。

例如：

- loading 没关闭
- resultData 没更新
- 错误状态覆盖了成功状态

### 第五层：渲染与组件协作

最后再看组件有没有拿到正确 props，页面有没有在正确时机重新渲染。

> 🌐 **In English:**
> Frontend integration debugging becomes much easier when you check route entry, request params, response shape, state mapping, and rendering in order.

## 🎯 四类高频故障场景

### 场景一：页面空白但接口正常

这种情况往往不是后端挂了，而是：

- 前端状态没接好
- 空态和错误态混淆
- 页面依赖的关键字段没拿对

### 场景二：请求不断重复发

常见原因有：

- 状态依赖写错
- 页面初始化逻辑反复触发
- 某个筛选条件不断变化

### 场景三：切换筛选后还是旧数据

常见原因有：

- 请求参数没更新
- URL 和状态不同步
- 旧请求后返回覆盖了新结果

### 场景四：部分用户正常，部分用户异常

这类问题第一反应就该想到：

- 权限
- 角色
- Token
- 环境或数据差异

## 💡 PM 在联调会里最有价值的提问方式

不要问“到底是谁的问题”，更好的问法是：

- 用户是怎么进这个页面的
- 当前发出的请求长什么样
- 接口回来的结构和页面预期一致吗
- 前端状态是怎么接这个返回结果的
- 为什么用户最后看到的是这个状态

这种问法能让讨论更快回到链路本身。

## ⚠️ 最容易拖慢联调的坏习惯

### 坏习惯一：只贴一句错误现象

例如“这里没数据了”，但不说明动作和上下文。

### 坏习惯二：只盯接口成功

接口成功不等于页面状态和组件渲染都正确。

### 坏习惯三：不区分“请求失败”和“展示失败”

两者的处理路径完全不同。

## 🧪 一个很好用的联调记录模板

以后你可以这样记：

1. 用户动作
2. 页面入口
3. 请求参数
4. 接口返回
5. 页面状态
6. 最终现象

这六项一列出来，问题通常就会清楚很多。

## 📝 记住这些

- 联调问题本质上是链路问题，不是单点问题。
- 先按层排，再谈解决方案，效率会高很多。
- PM 最有价值的地方，是把“混乱现象”整理成“可定位链路”。
