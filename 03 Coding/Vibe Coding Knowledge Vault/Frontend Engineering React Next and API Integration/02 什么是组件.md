> 📌 **一句话版本：** 组件是前端页面的可复用模块，`props` 是输入，`state` 是组件内部当前状态；很多前端问题，本质上都和这三者关系没理清有关。

---

## 📖 这一节在讲什么

当你看 React 项目时，最容易反复碰到的就是组件、props、state。你不理解它们，页面逻辑就很容易看成一团。理解以后，很多“这个值到底从哪来”“为什么页面没更新”会清楚很多。

## 🧠 什么是组件

组件可以理解成页面上的一个功能单元，比如：

- 按钮
- 搜索框
- 卡片
- 列表
- 弹窗

组件化的好处是：可以复用、方便维护、容易拆职责。

## 🎯 什么是 props

`props` 更像父组件传给子组件的输入参数。它告诉子组件“你要显示什么”“你该怎么表现”。

例如，一个按钮组件可能收到：

- 按钮文字
- 是否禁用
- 点击回调

## 🎯 什么是 state

`state` 是组件自己当前掌握的内部状态。比如：

- 输入框当前内容
- 弹窗是否打开
- 列表是否正在加载
- 当前选中的 tab

> 🌐 **In English:**
> Components are reusable UI units. Props are external inputs. State represents the component’s current internal condition.

## 🧩 为什么这三个概念会决定你看页面时是不是清楚

如果你不分清组件、props、state，页面逻辑很容易看成：

- “这块为什么会变”
- “这个值到底从哪来”
- “为什么父组件改了，子组件没跟着变”

一旦你把这三层分开，很多问题就能更明确地落位。

## 🎯 一个很常见的页面例子

假设有一个“生成周报”页面：

- 页面里有输入框组件
- 有提交按钮组件
- 有结果卡片组件

这时候：

- `props` 可能来自上层页面传入的标题、按钮是否禁用、结果数据
- `state` 可能是输入框当前值、是否在加载、是否显示错误

这样一想，组件就不再抽象了。

## 💡 为什么很多“页面没更新”其实是状态问题

因为现代前端很多展示都不是手工刷新的，而是状态一变，界面跟着变。

只要状态流没设计好，就会出现：

- 值改了但界面没变
- 请求回来了但页面还在 loading
- 错误发生了但按钮状态没恢复

> 🌐 **In English:**
> Many UI bugs are really state-flow bugs. The interface depends on state changes being modeled and propagated correctly.

## ⚠️ 一个很值得 PM 形成的直觉

以后看页面问题时，你可以先问：

- 这是组件拆分问题
- 是 props 没传对
- 还是 state 没更新对

这会比笼统说“页面有 bug”更有质量。

## ⚠️ 为什么很多 Bug 都和状态有关

因为页面不是静态海报，而是会不断变化。只要变化处理不清楚，就会出现：

- 用户操作了，界面没更新
- 界面更新了，但数据是旧的
- 加载和错误状态没覆盖全
- 子组件和父组件状态不同步

## 💡 PM 看组件最值得问的三个问题

- 这个组件负责什么，不负责什么
- 它需要哪些输入
- 它的状态由自己管理，还是由上层统一管理

## 📝 记住这些

- 组件负责拆页面，props 负责传输入，state 负责存当前状态。
- 很多前端问题不是“页面坏了”，而是状态流没设计清楚。
- 你越会从组件和状态角度观察页面，越能提出有质量的问题。
