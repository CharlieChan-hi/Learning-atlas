## 📖 这一节在讲什么

前面案例模块一直在拆结构和流程，这一页则直接给你一个更接近真实工作的 PRD 样例框架。

## 🧠 功能背景

用户需要把本周零散工作内容快速整理成一份更清晰、更适合汇报的周报。当前问题是：

- 手工整理费时
- 表达质量不稳定
- 不同人格式不一致

## 🎯 目标

提供一个 AI 周报生成功能，让用户输入原始素材后，可以快速得到结构化周报草稿，并支持查看历史和再次编辑。

## 👤 目标用户

- 一线产品经理
- 项目经理
- 需要频繁做周报、日报的知识工作者

## 🧭 核心流程

1. 用户进入周报生成页
2. 输入原始工作内容
3. 选择时间范围和输出风格
4. 点击生成
5. 页面显示处理中
6. 成功后展示结果
7. 用户可复制、编辑、再次生成或查看历史

## 🧱 页面结构

- 输入区
- 配置区
- 提交区
- 处理中状态区
- 结果展示区
- 历史记录区

## 🔌 接口需求

- 创建生成任务接口
- 查询生成结果接口
- 获取历史记录接口

## 📦 数据结构要求

输出结果至少包含：

- `summary`
- `highlights`
- `risks`
- `next_week_focus`

## ⚠️ 异常与边界

- 输入为空
- 输入过长
- 模型生成失败
- 超时
- 重复提交

## ✅ 验收标准

- 用户可以完成一次完整生成
- 失败时有明确反馈
- 历史记录可查
- 输出结构稳定
- 重复提交不会产生异常副作用

> 🌐 **In English:**
> A useful PRD sample should make the feature executable by defining users, goals, flows, structures, APIs, edge cases, and acceptance criteria.

## 📝 记住这些

- PRD 的价值不在于写得多漂亮，而在于是否足够可执行。
- 页面、接口、数据、状态和验收标准，至少要一起出现。
- 一个“像样”的 PRD，会直接降低后续返工成本。
