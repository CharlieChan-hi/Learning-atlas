> 📌 **一句话版本：** 一个 Web 应用的本质，是浏览器向服务器请求资源和数据，再用 HTML、CSS、JavaScript 把结果展示和更新出来。

---

## 📖 这一节在讲什么

很多 PM 会看到页面、按钮、接口、报错、数据库这些词，但不知道它们之间是怎么连起来的。这一节的任务，就是用最少的术语把一个 Web 应用的完整运行过程讲清楚。

## 🧠 最简单的 Web 闭环

你可以把一个网页请求想成下面这个过程：

1. 你在浏览器打开一个网址
2. 浏览器向服务器请求页面资源
3. 服务器返回 HTML、CSS、JavaScript 或数据
4. 浏览器把页面渲染出来
5. 你点击按钮时，前端代码再去请求接口
6. 服务器处理逻辑并返回 JSON
7. 页面根据新数据更新显示

> 🌐 **In English:**
> A web app works through repeated request-response loops. The browser loads resources, runs frontend code, sends API requests, receives JSON data, and updates the UI.

## 🎯 浏览器到底做了什么

浏览器不是“一个看网页的壳”，它实际上做了很多重要工作：

- 发起网络请求
- 解析 HTML
- 应用 CSS 样式
- 执行 JavaScript
- 管理页面状态
- 存储 Cookie、本地缓存、表单数据
- 提供调试工具

## 🧱 前端和后端分别在干什么

### 前端 Frontend

前端负责用户直接看到和操作的部分，比如页面布局、交互、按钮、输入框、状态展示。

### 后端 Backend

后端负责数据处理、业务规则、鉴权、数据库读写、接口返回、任务调度等。

你可以把它们想成：

- 前端负责“用户体验层”
- 后端负责“业务和数据层”

## 💡 请求、响应和接口

当页面需要数据时，前端通常会发起一个 HTTP 请求。后端收到后会处理逻辑，再返回一个响应。

这个“前端和后端约定好的交互边界”，通常就叫接口 `API`。

一个常见响应可能长这样：

```json
{
  "code": 0,
  "data": {
    "name": "Alice",
    "role": "PM"
  }
}
```

所以你今天提到的 JSON，就直接处在 Web 工作流的核心位置上。

> 🌐 **In English:**
> APIs define how the frontend and backend talk to each other. JSON is commonly used as the response format because it is structured and easy to parse.

## ⚖️ 静态页面和动态页面的区别

### 静态页面

页面内容基本固定，请求回来是什么就展示什么。

### 动态页面

页面会根据用户身份、输入、接口结果、实时状态不断变化。现在大多数产品页面都是动态的。

## 🧪 一个按钮点击后发生了什么

假设你点击“保存”按钮，背后可能发生这些事：

1. 前端收集表单数据
2. 前端做基础校验
3. 前端调用保存接口
4. 后端校验权限和参数
5. 后端写入数据库
6. 后端返回成功或失败结果
7. 前端根据结果提示“保存成功”或展示报错

当你理解这一串动作以后，你在写需求、查问题、看 AI 输出时就不会再只盯着页面表象。

## ⚠️ 最常见的认知误区

### 误区一：页面能打开，就说明后端没问题

页面能打开只说明静态资源大概率加载成功，不代表接口、鉴权、数据库都正常。

### 误区二：看到接口报错就是后端锅

很多接口报错其实是前端传参错误、用户权限不对、环境配置不一致导致的。

### 误区三：前端只是“美化页面”

现代前端已经负责很复杂的状态管理、交互逻辑、性能处理和组件组织，不只是画页面。

## 🎯 PM 看 Web 应用时最该问的五个问题

- 这个页面的数据来自哪里
- 这个交互会触发哪些接口
- 失败时前端和后端分别怎么反馈
- 状态变化由谁控制
- 这个功能上线后如何验证

## 🧩 从输入网址到看到页面，中间还发生了什么

如果再展开一点，一个 Web 页面从“输入网址”到“页面显示”往往会经历：

1. 浏览器找到目标站点
2. 发起请求
3. 收到页面资源或应用入口
4. 下载 CSS、JavaScript、图片等资源
5. 执行前端代码
6. 再请求接口拿业务数据
7. 渲染页面并响应用户操作

这说明页面不是一次性“下载完就结束”，而是一个持续交互的系统。

## 💡 为什么有时页面打开了，但功能还是坏的

这是非常典型的场景。常见原因有：

- 静态资源加载成功，但接口报错
- 页面骨架渲染了，但数据没回来
- 用户没有权限，接口被拒绝
- 后端成功了，但前端状态没更新
- 字段结构变了，页面解析失败

所以你以后判断问题时，不能只看“页面有没有显示出来”，还要看它是不是“真的可用”。

> 🌐 **In English:**
> A page can render successfully while the actual feature is still broken. Visual loading and business success are not the same thing.

## 🎯 PM 最容易忽略的一层：状态

状态可以理解成“页面此刻知道什么、显示什么、等待什么”。比如：

- 正在加载
- 已经拿到数据
- 提交中
- 提交失败
- 权限不足

很多产品体验问题，表面上看像样式或接口问题，实际上是状态设计不完整。

## ⚠️ 为什么你要知道前端、后端、数据库不是一条线，而是一个环

一个按钮点击后，可能经过：

- 前端收集参数
- 接口发给后端
- 后端执行业务逻辑
- 数据库保存结果
- 后端返回响应
- 前端更新状态
- 页面重新渲染

这个环里任何一层出问题，用户看到的现象都可能很像。所以你越理解全链路，越不会只盯着某一个点。

## 📝 记住这些

- Web 应用本质上是浏览器、前端、后端、数据之间的协作系统。
- 页面只是表面，真正重要的是背后的请求、状态、接口和数据流。
- JSON 是前后端和 AI 工具之间高频出现的结构化数据形态。
