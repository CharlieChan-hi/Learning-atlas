# Core Flows and Information Architecture | 核心路径与信息架构

> 零基础先记住这一句：用户打开一款消费级产品时，最想知道的不是你有多少页面，而是我现在先做哪一步。
> English takeaway: If the next step is unclear, the structure is already too heavy.

## 核心路径和信息架构，不是同一个概念

核心路径讲的是，用户为了完成主任务，最短要经过哪几步。信息架构讲的是，这些内容、功能和入口怎样被组织、命名和导航。前者更像主骨架，后者更像骨架外面的秩序。两者互相依赖，但不能混为一谈。很多团队的问题就在这里，它们一边在谈导航，一边其实是在掩盖主任务不清；一边在谈路径，一边其实是信息层级已经太乱。

对零基础读者来说，最稳的理解方式是先把核心路径放前面。因为消费级产品最怕的，从来不是页面少，而是用户在第一次进入时不知道先做什么。主任务不清，后面的架构再细也只是把复杂性重新排列。

## 先识别主任务，再决定首页应该承担什么

识别主任务时，可以先问一个简单问题，用户打开产品最希望立刻推进的动作是什么。对聊天产品来说，是开始提问；对存储工具来说，是上传、同步或分享；对学习产品来说，是开始一小段练习；对创意产品来说，是导入素材或直接编辑。只要这一步抓错，首页就很容易变成一张“什么都有一点”的信息墙。

首页真正该承担的，不是展示团队做了多少功能，而是让主任务足够显眼，让第一次进入的人知道自己现在就能开始。很多消费级产品最成功的地方，恰恰在于它们把这个决定做得足够狠，别的内容可以后置，主任务不能躲。

## 一条好路径，通常都有三个特征

第一，开始动作够轻，用户不会在价值发生前就先被流程耗掉。第二，下一步够清楚，每个节点都知道接下来要做什么。第三，回到产品时也够顺，用户不是每次都重新找路。真正好的路径，不会要求用户先读说明书，也不会用一大堆中间动作证明自己很完整。它更像一条被压过的轨道，用户沿着它走，能不断感到自己在往结果推进。

信息架构的任务，就是帮助这条轨道更好被理解。命名要贴近用户语言，层级不要太深，主入口要稳定，搜索和导航要互相补位。它不是为了让结构看起来更专业，而是为了让用户在短时间里少做几次判断。

## 信息架构会反过来影响引导、增长和付费

路径一旦不清，引导就会被迫变重，因为它要替产品补很多本应在结构里解决的解释。增长入口一旦和主路径不连，用户被拉进来以后也只会迷路。付费边界一旦和信息架构脱节，用户会在还没理解基础使用时就遇到高级层，于是付费请求看起来像硬拦截。也就是说，信息架构不是单独的设计问题，它会一路影响后面的激活、留存、增长和收费。

这也是为什么很多支付墙优化最后没有真正解决问题。团队只在价格页上用力，却没有回头看用户是否顺利经过了前面的主路径。如果前面的动作已经太碎，支付页很难凭空变成强转化。

## 真正常见的反模式，是把复杂性推给用户

最常见的反模式有三种。第一种是首页过度拥挤，重要和次要内容并列，用户进来先开始认导航。第二种是命名来自团队内部语言，用户读得懂每个字，却不知道分别是什么。第三种是进入路径和返回路径都没被设计，第一次勉强走过去了，第二次回来还得重新摸索。看起来每一个问题都不大，叠在一起以后，用户就会觉得这款产品“说不清楚”。

一条非常实用的检查方法是，让自己只带着一个具体任务进入产品，看能不能在几步之内完成。如果你总是忍不住想解释界面、解释按钮、解释分组，那通常说明主路径还没有被压清。

## 推荐体验 App 与体验任务

### Dropbox
- 平台：iOS / Web / Android
- 为什么推荐它：它适合观察工具产品怎样把主任务藏得很浅。
- 先体验哪一步：尝试上传一个文件，再尝试分享。
- 重点观察什么：它怎样把“存、取、同步、分享”串成同一条主路径。
- 体验任务：
  - [ ] 记录你从首页到完成分享用了几步。
  - [ ] 看它有没有让次要功能抢走首页主任务的位置。

### ChatGPT
- 平台：iOS / Web
- 为什么推荐它：它适合观察当主任务极清楚时，信息架构可以有多克制。
- 先体验哪一步：直接开始一次提问，再看历史记录和推荐入口。
- 重点观察什么：它怎样让新用户和回访用户都能迅速找到下一步。
- 体验任务：
  - [ ] 判断它真正稳定的主入口是什么。
  - [ ] 记录你第二次回来时，第一眼会去哪里继续动作。

## X 与论坛原话

### 1. Nikita Bier：路径设计的本质，是别浪费每一次点击

**为什么这条帖值得放进本章**

这条帖很适合放在核心路径章节，因为消费级产品里每一次点击都带着流失风险。

**原帖原话**

> "Every single tap is a miracle."

**中文解释**

点击不是理所当然的。用户愿意继续往下走，本身就是你已经争取到的一次信任，所以路径越要克制。

## 本章复盘

- 本章最重要的判断是，核心路径先于信息架构，主任务先于页面完整度。
- 最容易犯的错，是把复杂结构重新整理一遍，却没有把主任务真正压短。
